This page summarizes `docs/cli-desktop-parity.md`, which tracks `apps/cli` (the Rust ratatui TUI,
binary `ryu`) against `apps/desktop` (Tauri plus React).

## The principle

The CLI reaches the **same Core** (`:7980`) the desktop uses, so parity is overwhelmingly a matter
of surfacing existing Core endpoints, not new backend work. The goal is **capability parity, not
modality parity**: a terminal cannot render a React Flow canvas or a screen-capture timeline, so some
desktop surfaces are deliberately out of scope rather than missing. Those are marked N/A in the TUI.

Two design notes shape the layout. First, extend, do not rewrite: the CLI is one ratatui event loop
with a single `App` struct, per-tab keys in `main.rs`, rendering in `ui.rs`, and Core fetchers in
`api.rs`, so each new surface follows the same seam. Second, the command palette (`Ctrl+P`) is the
information architecture: a narrow sidebar cannot hold every tab, so a fuzzy palette is the primary
jump surface (the Cmd+K analog) and the sidebar stays a short grouped list.

<Callout type="info">
Verification ceiling: without a live Core plus a TTY, `cargo build -p ryu-cli` is the gate. Each
slice is compile-verified; live round-trips (judge verdicts, installs, transcripts) are
runtime-unverified.
</Callout>

## Matrix

Legend: ✅ built, 🟡 in progress, ⬜ planned, 🚫 N/A in TUI (with reason).

### Chat

| Feature | Status | Notes |
|---|---|---|
| Multi-turn streaming chat | ✅ | `POST /api/chat/stream`, `conversation_id` minted per chat |
| Agent picker (`Ctrl+A`) | ✅ | `GET /api/agents` |

| `/btw` side question | ✅ | `POST /api/btw`, ephemeral overlay |
| `/goal` set/clear plus judge loop | ✅ | judge endpoint plus client continuation loop (capped at 25), goal status bar |

| Council team (`/team <id>`) | ✅ | `team_id` body field |

| File attach / Plate editor | 🚫 | rich editor, `ryu open` hands media to desktop |

## The deferred set

The pieces consciously not yet in the TUI, in roughly priority order:

- Permission-mode and reasoning-effort pickers (need an `acp-config` probe).
- Conversation fork (`/fork`) and a client-side message queue.
- Memory and settings tabs (blocked on Core list endpoints that do not exist yet).
- Marketplace browse (lives on the control-plane `:3000`, behind auth, not Core).

Modality-bound surfaces (voice input, the workflow canvas, recipe recording, marketplace buy, the
Shadow timeline, split view and the island) stay N/A in the TUI by design: a terminal hands those off
to the desktop with `ryu open` rather than reimplementing them.
