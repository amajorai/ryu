`apps/tui` is a Bun plus [OpenTUI](https://github.com/sst/opentui) React terminal client for Ryu Core. It is the successor to the Rust ratatui CLI (`apps/cli`, binary `ryu`): a rewrite in TypeScript that reaches the **same Core** node over plain HTTP and SSE, holding no model and running no inference of its own. It resolves its target the same way the MCP bridge does - `RYU_CORE_URL` (default `http://127.0.0.1:7980`) and an optional `RYU_CORE_TOKEN` (`apps/tui/src/core/target.ts`).

<Callout type="info">
  The OpenTUI client ships **alongside** the ratatui CLI, not as a replacement yet. The TUI is at
  full interactive feature parity with the CLI (Teams surface, guided Setup wizard, launch update-notice
  all added) and is ahead on chat, split-pane workspace, and store-wide search. It is the recommended
  terminal client. See [Desktop parity](/docs/cli/parity) for the parity matrix.
</Callout>

## Launch it

The CLI owns the launcher. `ryu tui` shells out to the OpenTUI client and hands it the active node so both talk to the same Core (`run_tui_command`, `apps/cli/src/main.rs`):

```sh
ryu tui
```

Under the hood it resolves what to run in this order:

1. `RYU_TUI_CMD` - a full command line, the escape hatch for packaged installs.
2. Otherwise it walks up for the monorepo root (the first ancestor with `apps/tui/package.json`) and runs `bun run --filter tui dev` there.

Either way it exports `RYU_CORE_URL` from the active node (and `RYU_CORE_TOKEN` when the node has one), so the OpenTUI client inherits the node you selected with the CLI's `node` subcommands. Run it directly during development from the repo root:

```sh
bun run --filter tui dev
```

If `ryu tui` cannot find `apps/tui`, run it from inside the Ryu repository or set `RYU_TUI_CMD` to the launch command, for example `RYU_TUI_CMD="bun run /path/to/ryu/apps/tui/src/index.tsx" ryu tui`.

## The shell

The client is a tabbed workspace shell (`apps/tui/src/App.tsx`). Its Ctrl-modified bindings own the workspace:

| Key | Action |
|---|---|
| `Ctrl+K` | Open the fuzzy command palette (the Cmd+K analog, `palette/CommandPalette.tsx`) |
| `Ctrl+T` | Open a new chat tab (`Ctrl+Shift+T` restores the last-closed tab) |
| `Ctrl+W` | Close the focused pane's active tab |
| `Ctrl+Alt+S` | Toggle a two-pane split |
| `Ctrl+Tab` | Cycle tabs in the focused pane (Shift reverses) |
| `Alt+←` / `Alt+→` | Move between panes |
| `Ctrl+C` | Quit |

The command palette is the primary navigation surface - a narrow sidebar cannot hold every tab, so the palette fuzzy-jumps to any tab plus global actions. Switching nodes is one of those actions: the node picker (`workspace/NodePicker.tsx`) is opened from the palette's "Switch node" entry and from the sidebar's node selector, then driven with the arrow keys and Enter.

## The tabs

The shell registers **17 tabs** in one intentional barrel (`apps/tui/src/tabs/registry.ts`), in an order that mirrors the ratatui CLI's sidebar so the two surfaces line up:

Chat, Services, Agents, Models, Skills, Tools, Apps, Gateway, Workflows, Recipes, Teams, Spaces, Engines, Monitors, Meetings, Schedules, Account.

Each tab is a thin view over Core endpoints, matching the CLI's list-tab model. Because this is a pure HTTP/SSE client to the same Core, parity is a matter of surfacing endpoints that already exist rather than new backend work - the same principle the CLI is built on (see [CLI](/docs/cli/index)).

## Chat

The Chat tab streams over Core's `POST /api/chat/stream` via SSE (`apps/tui/src/core/chatStream.ts`) and supports the same chat-side flows as the CLI (`apps/tui/src/surfaces/chat/index.tsx`):

- **`Ctrl+A`** opens the agent picker. The selected agent and team show as chips; the model is set via the `/model` command and shown as a read-only chip.
- **`Ctrl+L`** starts a new chat.

The typed slash commands are:

| Command | What it does |
|---|---|
| `/goal <text>` | Pass-through to Core's server-side goal turn-hook, which owns the completion condition and judge loop. |
| `/proof` | Pass-through to Core's server-side proof turn-hook. |
| `/check` | Arm the per-turn double-check hook via `plugin_flags`; Core reviews the next turn and the note lands in the stream. |
| `/btw <question>` | Ask a quick side question (`POST /api/btw`); ephemeral, dismissible, never enters history. |
| `/model <id>` | Override the model for the turn (cleared with a bare `/model`). |
| `/team <id>` | Route the turn to an agent team. |
| `/sessions` | Open the run history for the current conversation. |
| `/new` (`/newchat`) | Reset the chat. |

`/goal`, `/proof`, and `/check` are Core server-side plugin turn-hooks: `/goal` and `/proof` pass through as normal messages, and `/check` arms the double-check hook, so their behavior lives in Core, not the client. The tool-call loop is surfaced inline as the agent works.
