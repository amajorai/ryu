title: "Double-Check"
description: "A second model reviews each assistant reply, fired once per message as a turn-hook plugin"
---

Double-check is a built-in feature that runs a second, separately configured model over the latest
assistant answer and reports whether it spots a likely mistake. It fires **once per message** with
no loop and no re-kick (unlike [Goals](/docs/core/goals), which re-judge every turn).

<Callout type="info">
  Double-check runs as a `post_assistant_turn` plugin turn-hook (`io.ryu.double-check`), not as a
  Core HTTP endpoint. The reviewer logic lives in `apps/core/src/server/mod.rs`; the hook is
  dispatched by `apps/core/src/plugin_host/mod.rs` after each assistant reply via
  `dispatch_turn_hooks`.
</Callout>

## How it runs

After each assistant turn the plugin host calls `dispatch_turn_hooks`. The `io.ryu.double-check`
hook checks the `io.ryu.double-check` flag on the conversation; if it is `true`, it:

1. Reads the recent transcript (`DOUBLE_CHECK_TRANSCRIPT_LIMIT = 6` messages).
2. Takes the latest `assistant` message as the answer under review.
3. Resolves the reviewer model and effort (see below).
4. Calls `call_side_model` - one non-streaming `POST /v1/chat/completions` to the local gateway
   with a fixed reviewer system prompt asking for a `VERDICT: ok` or `VERDICT: issues` first line.
5. Runs `parse_double_check_verdict` to get `(ok, critique)`.
6. Returns a `note` directive with the verdict so the client can show a badge and critique popover.

The reviewer judges **only from the text given** - it cannot run code or browse.

## Reviewer model and effort

Nothing is hardcoded. The reviewer model resolves through this precedence:

| Order | Source |
|---|---|
| 1 | `double-check-model` preference |
| 2 | `RYU_DOUBLE_CHECK_MODEL` env var |
| 3 | `RYU_DEFAULT_LLM_MODEL` env var |
| 4 | `registry::DEFAULT_LLM_MODEL` literal |

Effort resolves the same way: the `double-check-effort` preference, else the
`RYU_DOUBLE_CHECK_EFFORT` env var, else empty. When non-empty, effort is forwarded as
`reasoning_effort` on the gateway request.

<Callout type="info">
  The model id is a free, gateway-routable id (any local stem, `openrouter/...` slug, or provider
  id). The desktop config surfaces provider/model/effort as suggestions, not a constraint.
</Callout>

## Verdict parsing (fail-loud)

`parse_double_check_verdict` is deliberately fail-loud, the opposite of the goal judge's
fail-safe-stop. Flagging a false positive is cheaper than hiding a real mistake.

| Reviewer reply | Result |
|---|---|
| First line starts with `ok` / `pass` / `correct` | `ok = true` |
| First line starts with `issue` / `fail` / `wrong` / `no` | `ok = false`, detail as critique |
| Unreadable or empty | `ok = false`, whole reply surfaced as the critique |

Six unit tests (`double_check_tests`) cover these branches.

<Callout type="warn">
  `reasoning_effort` reaches OpenAI-compatible, local, and OpenRouter providers verbatim. The
  Anthropic-direct transform (`to_anthropic_body`) rebuilds the request and currently drops
  `reasoning_effort`. This is a known caveat, not a failure.
</Callout>

## Turning it on

Double-check is driven by a composer toggle in the desktop chat. When on, the client sets the
`io.ryu.double-check` flag on the conversation; the hook fires automatically after each assistant
turn and the client shows a verdict badge plus a critique popover.
