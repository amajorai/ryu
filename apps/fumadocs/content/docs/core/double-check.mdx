---
title: "Double-Check"
description: "A second model reviews each assistant reply, fired once per message"
---

Double-check is a built-in Core primitive that runs a second, separately
configured model over the latest assistant answer and reports whether it spots a
likely mistake. It is stateless and fires **once per message**: there is no loop,
no cap, and no re-kick (unlike [Goals](/docs/core/goals), which re-judge every
turn). The handler reuses the same `call_side_model` side-model call as the goal
judge and [`/btw` side questions](/docs/core/side-questions), so the gateway
request shape lives in exactly one place.

It is implemented in `apps/core/src/server/mod.rs`.

## Endpoint

```
POST /api/conversations/{id}/double-check
```

The handler (`double_check_handler`) is stateless. It reads the conversation
transcript, runs the reviewer, and returns the verdict. It persists nothing.

| Field | Value |
|---|---|
| Path param | `id` - the conversation id |
| Request body | none |
| Transcript window | last `DOUBLE_CHECK_TRANSCRIPT_LIMIT` (6) messages, via `get_recent_messages` |
| Reviewed text | the latest `assistant` message |
| Context given | the most recent preceding `user` turn |
| Success | `200` with `{ ok, critique, model }` |
| No answer yet | `400` if there is no assistant message to review |
| Reviewer down | `502` if the gateway side-model call fails |

### Response

| Field | Type | Meaning |
|---|---|---|
| `ok` | bool | `true` when the reviewer found no issues |
| `critique` | string | the reviewer's detail, or a default phrase when none was given |
| `model` | string | the resolved reviewer model id |

## How it runs

1. `get_recent_messages(id, 6)` loads the recent transcript.
2. The latest `assistant` message is the answer under review; the most recent
   `user` turn before it gives the reviewer the question to judge against.
3. The reviewer model and effort are resolved (see below).
4. `call_side_model` makes one non-streaming `POST /v1/chat/completions` to the
   local gateway with a fixed reviewer system prompt instructing a `VERDICT: ok`
   or `VERDICT: issues` first line.
5. `parse_double_check_verdict` turns the reply into `(ok, critique)`.

The reviewer judges **only from the text given** - it cannot run code or browse.

## Reviewer model and effort

Nothing is hardcoded. The reviewer model resolves through this precedence
(`resolve_double_check_model`):

| Order | Source |
|---|---|
| 1 | `double-check-model` preference |
| 2 | `RYU_DOUBLE_CHECK_MODEL` env var |
| 3 | `RYU_DEFAULT_LLM_MODEL` env var |
| 4 | `registry::DEFAULT_LLM_MODEL` literal |

Effort resolves the same way (`resolve_side_effort`): the `double-check-effort`
preference, else the `RYU_DOUBLE_CHECK_EFFORT` env var, else empty. When non-empty,
effort is forwarded as `reasoning_effort` on the gateway request.

<Callout type="info">
The model id is a free, gateway-routable id (any local stem, `openrouter/...`
slug, or provider id). The desktop config surfaces provider/model/effort as
**suggestions, not a constraint**.
</Callout>

<Callout type="warn">
`reasoning_effort` reaches OpenAI-compatible, local, and OpenRouter providers
verbatim because they clone the request body. The Anthropic-direct transform
(`to_anthropic_body`) rebuilds the request and currently **drops** `reasoning_effort`.
This is a known caveat, not a hard failure.
</Callout>

## Fail-loud parsing

`parse_double_check_verdict` is deliberately **fail-loud**, the opposite of the
goal judge's fail-safe-stop: here, flagging a false positive is cheaper than
silently hiding a real mistake.

| Reviewer reply | Result |
|---|---|
| First line starts with `ok` / `pass` / `correct` | `ok = true` |
| First line starts with `issue` / `fail` / `wrong` / `no` | `ok = false`, detail as critique |
| Unreadable or empty | `ok = false`, the whole reply surfaced as the critique |

When a clean verdict carries no detail the critique defaults to `No issues found.`;
a flagged verdict with no detail reads `Issues found (no detail given).`. Six unit
tests (`double_check_tests`) cover these branches.

<Callout type="warn">
Built-in feature, not a plugin. Double-check is modeled on the Goal feature and
wired directly into Core and the desktop composer; the plugin runtime cannot yet
add composer toggles or settings. The wiring is compiled and unit-tested, but an
end-to-end critique was **not live-verified** - it needs a routable model reachable
through the gateway.
</Callout>

## Turning it on

Double-check is driven by a composer toggle in the desktop chat. When on, the
client fires this endpoint once per assistant message as soon as the turn
finishes and shows a verdict badge plus a critique popover. For the how-to, see
[Chat](/docs/using-ryu/user-guide/chat).
