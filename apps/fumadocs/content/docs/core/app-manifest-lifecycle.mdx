title: "App Manifest & Lifecycle"
description: "How manifest.json manifests are validated, installed, enabled, disabled, and updated"
---

A Ryu **plugin** is a bundle described by a `manifest.json` manifest: it names the
[Runnables](/docs/start-here/architecture/runnable-model) it ships, the permission grants it needs,
and an optional Companion surface. Core owns the lifecycle (what runs - load, validate, install,
enable, disable, update). The Gateway owns enforcement (what is allowed - grant validation, signing).

This page is the reference for the manifest validation rules, the lifecycle operations and their
routes, the two-tier built-in registry, and activation events. For the full manifest field schema,
see [/docs/develop/sdk/plugin-api](/docs/develop/sdk/plugin-api). For the grant and signing model,
see [/docs/gateway/governance](/docs/gateway/governance).

Source: `apps/core/src/plugin_manifest/` (manifest type, loader, per-kind schema) and
`apps/core/src/plugins/` (lifecycle operations, built-in registry).

## The manifest file

The canonical file name is `manifest.json`. The legacy `ryu.json` is still read so plugins installed
before the apps -> plugins rename keep loading (`MANIFEST_FILE_NAMES` in
`apps/core/src/plugin_manifest/mod.rs`).

The loader (`PluginManifestLoader`) merges two sources: the built-in manifests compiled into the
binary, then user-installed manifests scanned from a directory resolved in this order.

| Order | Source | Notes |
|---|---|---|
| 1 | `RYU_PLUGINS_DIR` | Explicit override, if set |
| 2 | `RYU_APPS_DIR` | Legacy env var, still honoured |
| 3 | `~/.ryu/plugins` | Default when present |
| 4 | `~/.ryu/apps` | Legacy fallback only when the new dir is absent but the old one exists |

Each plugin lives in its own subdirectory holding a `manifest.json` (preferred) or `ryu.json`
(legacy). A manifest that fails to parse is skipped with a warning; the rest keep loading.

A manifest carries the fields below. The serde shapes are owned by the SDK schema reference
([/docs/develop/sdk/plugin-api](/docs/develop/sdk/plugin-api)); this table is orientation only.

| Field | Required | Purpose |
|---|---|---|
| `id` | yes | Reverse-domain unique id (e.g. `com.example.my-app`), also the on-disk directory name |
| `name` | yes | Display name in the store and launcher |
| `version` | yes | Semver string |
| `runnables` | yes | The bundled Runnables, each with a `kind` and per-kind `config` |
| `permission_grants` | no | Grants the plugin declares it needs (declarations only here; the Gateway enforces) |
| `companion` | no | An in-desktop overlay or sidebar panel surface |
| `contributes` | no | VS Code-style contribution points, referenced by runnable id |
| `activation_events` | no | Lazy-wake events (empty list means eager) |
| `engines` | no | `engines.ryu` semver requirement the running Core must satisfy |
| `runtime` | no | A declarative external runtime (e.g. a Python venv + pip deps + assets) |
| `sidecars` | no | Managed child processes Core downloads, spawns, and health-checks (see [Managed sidecars](#manifest-declared-managed-sidecars)) |

## Per-kind Runnable validation

Every `RunnableEntry` carries a `kind` discriminant and an optional typed `config`. The loader runs
`validate_runnable` (`apps/core/src/plugin_manifest/schema.rs`) on each entry and rejects the whole
manifest if any entry fails its per-kind contract. There is no `_ =>` fallback arm, so every one of
the eight [`RunnableKind`](/docs/start-here/architecture/runnable-model) variants is checked
explicitly.

| Kind | Config required | Required fields |
|---|---|---|
| `agent` | no | All fields default (`system_prompt`, `model`, `tools`) |
| `workflow` | yes | `entry` (non-empty) |
| `tool` | yes | `slug` (non-empty) |
| `skill` | yes | `skill_id` (non-empty) |
| `companion` | yes | `label` (non-empty) |
| `channel` | yes | `platform` (non-empty) |
| `engine` | yes | `engine_type` (non-empty) |
| `policy` | yes | `policy_type` (non-empty) and `definition` |

A missing required config, an empty required string, or a config blob that does not parse as the
expected shape all produce a descriptive error that names the runnable id and kind.

## Loader validation rules

Beyond per-kind config, the loader (`parse_and_validate`) enforces these manifest-level rules. A
violation rejects that manifest only; the others continue loading.

| Rule | Behavior |
|---|---|
| Id format | Strict allowlist `[a-zA-Z0-9.-_]`, must contain a `.`, no `..`, no leading/trailing `.`, no leading `-`, max 128 bytes. The id is a filesystem path component, so this is path-traversal hardening (`validate_plugin_id`) |
| Semver | `version` must parse as semver; otherwise rejected |
| Duplicate id | Across built-ins and user manifests, the first occurrence with an id wins; later duplicates are rejected |
| `engines.ryu` pin | When present, it must parse as a semver requirement and the running Core version (`CARGO_PKG_VERSION`) must satisfy it; otherwise rejected |
| `contributes` cross-validation | Every runnable id referenced in `contributes` must resolve to an entry in `runnables` (declare-by-id); a typo is caught at load |

## Lifecycle operations

The four state transitions live in `apps/core/src/plugins/lifecycle.rs` and are backed by the
SQLite `PluginStore`. Each maps to an HTTP handler in `apps/core/src/server/mod.rs`.

| Operation | Behavior |
|---|---|
| `install_app` | Creates a `PluginRecord` with `enabled = false`. Re-validates semver. Fails if already installed |
| `enable_app` | Validates the manifest's declared grants via the Gateway, then flips `enabled = true` and records the approved grants. Fails closed |
| `disable_app` | Flips `enabled = false` and clears approved grants |
| `update_app` | Compares semver. Refuses a downgrade (new < installed) unless `force = true`. A same-version update is a no-op. Does not re-enable a disabled plugin |

Install always lands disabled. Enabling is the gated step: Core calls the Gateway's
`/v1/grants/validate` for every grant the manifest declares and stores the result, applying no
inline policy of its own. If the Gateway denies a grant the plugin stays disabled; if the Gateway is
unreachable the enable fails closed rather than silently allowing. Enabling with an empty grant list
always succeeds (nothing to validate). See [/docs/gateway/governance](/docs/gateway/governance) for
how grants are authorised and signed.

<Callout type="warn">
`RYU_STUB_GRANT_VALIDATION=1` makes grant validation an allow-all stub for environments where the
Gateway endpoint is not yet available. It is logged at WARN so it is visible in operator logs. The
stub keeps the seam explicit; it is not a production setting.
</Callout>

### Routes

`/api/plugins*` is canonical. `/api/apps*` is a deprecated one-release alias pointing at the same
handlers, kept only so older clients keep working.

```
GET  /api/plugins                 list loaded manifests (built-in + user-installed)
GET  /api/plugins/catalog         browse installable plugins from the remote registry
POST /api/plugins/install         install by fetching manifest.json from a URL
POST /api/plugins/reload          re-scan built-ins + the plugins dir (no restart)
POST /api/plugins/:id/install     record as installed (disabled)
POST /api/plugins/:id/enable      validate grants via Gateway, then enable
POST /api/plugins/:id/disable     disable + clear approved grants
POST /api/plugins/:id/update      update to a new manifest version
```

## Two-tier registry

A plugin's trust tier is **derived from membership** in a constant, never asserted by a manifest
field, so a plugin cannot promote itself. The lists live in `apps/core/src/plugins/builtins.rs`.

| Tier | Meaning |
|---|---|
| `core` | First-party, shipped with Ryu. Listed in `CORE_PLUGINS`. Renders in the "Core" store section |
| `community` | Everything else, including user-installed plugins and built-in fixtures not in `CORE_PLUGINS`. Always install-then-enable opt-in |

A subset of Core plugins is seeded enabled on a fresh install (`CORE_DEFAULT_ON` in
`apps/core/src/plugins/builtins.rs`). It spans the local engine (`engines`) and durable workflow
engine (`durable`), the four default tool apps (`ghost`, `shadow`, `spider`, `agentbrowser` - see
[Default tool apps](#default-tool-apps-auto-seed)), the chat turn-hook plugins (`goal`, `proof`,
`double-check`), the pre-turn prompt improver (`com.ryuhq.auto-expand`), and the bundled Ryu Apps
(`checklist`, `smart-intake-form`, `data-grid-explorer`, `chart-studio`, `decision-wizard`,
`quest-board`, `worktree-diff-review`, `gateway-budget-dial`). Ids are the bare manifest `id`
(e.g. `engines`, not `io.ryu.engines`). The seed runs once at startup and respects user choice: it
acts only when there is no prior record for that plugin, so a later disable survives a restart
(`main.rs` default-on seeding, #444). The opt-in Core plugins (`firewall`, `routing`, `sandbox`,
`predict`) are Core-tier but deliberately excluded from default-on because they change gateway,
sandbox, or capture behaviour.

## Default tool apps auto-seed

Ghost, Shadow, Spider, and Agent Browser are the batteries-included tool apps. Their manifest ids
are in both `CORE_PLUGINS` and `CORE_DEFAULT_ON` (`apps/core/src/plugins/builtins.rs`), so on a
fresh install the plugin seeder auto-installs and enables each one - they appear installed exactly
like the auto-downloaded default models, rather than waiting for a manual install-then-enable.

| App | Tool | Cross-platform | Reaches the web |
|---|---|---|---|
| `ghost` | Desktop automation (30 tools) | Windows-first | no |
| `shadow` | Screen and audio capture + semantic memory | Windows-first | no |
| `spider` | Web crawler (Rust `spider-rs/spider`) | yes | yes |
| `agentbrowser` | Web browsing (npx-launched MCP server) | yes | yes |

The seeded record is a **governance shell**: each fixture declares `runnables: []`. The actual tools
are served by their own dedicated provider - Ghost and Shadow through Core's MCP registry, Agent
Browser as an npx MCP server, Spider as a Rust sidecar - so declaring zero runnables means the seed
never double-lists a tool that already comes from the MCP provider. Enabling the record just makes
each one a first-class, governed, disable-able App; the tool process still runs through its own
sidecar/MCP lifecycle. This is why they are also **system plugins** (`SYSTEM_PLUGINS`,
`is_builtin`): the record is the governed App surface, the sidecar/MCP provider is the run path, and
the two facts coexist.

<Callout type="info">
Because the run path is the sidecar/MCP provider, not the plugin record, the tool process still has
to be installed and started on its own (`POST /api/setup/:name/install`,
`POST /api/sidecar/:name/start`). The auto-seeded record makes the App show as installed; it does not
by itself download a Windows-only binary onto a non-Windows host. See the Ghost/Shadow note under
[Built-in manifests](#built-in-manifests).
</Callout>

## Built-in manifests

Built-in manifests are compiled into the binary (`BUILTIN_MANIFESTS`) and always load regardless of
whether a user plugins directory exists.

| Manifest | What it declares |
|---|---|
| `sample.manifest.json` | Research Assistant demo (first-run + integration tests) |
| `spider.manifest.json` | Spider web-crawler tool |
| `exa.manifest.json` | Exa neural search tool (BYOK) |
| `ghost.manifest.json` | Ghost desktop-automation MCP tool |
| `shadow.manifest.json` | Shadow screen and audio capture + semantic memory |
| `headroom.manifest.json` | Gateway egress compression as a `compression` Policy runnable |
| `firewall.manifest.json` | Gateway firewall on/off Policy (Core-tier, opt-in) |
| `routing.manifest.json` | Smart classifier routing on/off Policy (Core-tier, opt-in) |
| `sandbox.manifest.json` | Wasmtime ephemeral sandbox on/off Policy (Core-tier, opt-in) |
| `engines.manifest.json` | Local engine bindings (llama.cpp + embeddings), default-on |
| `durable.manifest.json` | Durable workflow execution engine, default-on |

<Callout type="warn">
Ghost and Shadow are **Windows-first** and **sidecar-managed**. Their lifecycle is sidecar-based
(`POST /api/setup/:name/install` to install, `POST /api/sidecar/:name/start|stop` to run), not the
`PluginStore` install/enable path, so they never appear in the SQLite plugins table. On other
platforms the desktop renders a graceful unavailable state.
</Callout>

## Manifest-declared managed sidecars

A manifest can declare `sidecars[]` - long-running child processes Core owns end-to-end (download or
provision, spawn, health-check, stop). This is the **app to sidecar bridge**: a capability sidecar
(Ghost, Shadow, a TTS engine) can be a fully manifest-defined app instead of hardcoded Rust, and a
third-party app can ship its own process. Infra sidecars (llama.cpp, the Gateway, embeddings) stay
Core substrate and are deliberately not expressible here. The type is `SidecarSpec` in
`apps/core/src/plugin_manifest/schema.rs`; the generic driver is `ManifestSidecar`
(`apps/core/src/sidecar/manifest_sidecar.rs`). The manifest field itself is documented in
[Bundle Runnables with manifest.json](/docs/develop/extensions/plugin-json-manifest#declare-a-managed-sidecar).

Each `SidecarSpec` carries a `name` (unique within the plugin), a `process`, a `port`, and a
`health_path` (default `/health`). The process is obtained one of two ways.

| `process.kind` | How Core runs it |
|---|---|
| `binary` | Fetches the executable (or a `tar.gz`/`tar.bz2`/`zip` archive) via the shared `DownloadCenter` - streaming `.part` + resume + SHA-256 verify - into a version-namespaced `bin/<version>/` dir, extracts if archived (whole tree so sibling libraries stay co-located), makes it executable, then spawns it with the declared `args` + `env` |
| `python` | Reuses the external-runtime provisioner (venv + pip + assets), then spawns `python -m <entry>` |

The URL must be `https` and is screened by the SSRF egress check at download time. Without a
`sha256` an already-present artifact is trusted (idempotent skip); with one, a mismatch re-fetches
(fail closed).

**The `sidecar:process` grant gates it.** Downloading and spawning an arbitrary process is a broader
surface than the manifest metadata path, so `may_run_sidecar` (`manifest_sidecar.rs`) fails closed - a
Core-tier plugin is auto-allowed, and a Community-tier plugin must hold the Gateway-approved
`sidecar:process` grant, read from the plugin's *approved* grants (post-Gateway validation), never its
declared, unvalidated `permission_grants`. For the Python flavor this one grant also stands in for
`runtime:external`, since binary execution is the broader surface.

Once permitted, the sidecar is registered under the namespaced key `<plugin_id>/<name>` into the live
`SidecarManager`, so it rides the same managed lifecycle - health monitor, resource sampler,
`/api/sidecar/status`, graceful stop - as any built-in. Before spawning, the manager claims the
declared `port` with a `TcpListener` bind-probe, which rejects a collision with a built-in (e.g.
llama.cpp on 8080) or another plugin. There is no port registry in v1, so the plugin author is
responsible for choosing a free port.

<Callout type="warn">
Manifest sidecars are **not** in the `SidecarManager` startup order, so nothing restarts them after a
Core restart on its own. `reconcile_plugin_sidecars` (spawned from `main.rs` next to the `onStartup`
fire) is the load-bearing boot step: it re-registers and starts every enabled plugin's declared
sidecar. A manifest sidecar is always optional (`is_required` is false), so a plugin process failing
to start never aborts Core boot.
</Callout>

## Activation events

A plugin can declare `activation_events` to be woken lazily instead of eagerly. An **empty** list
means eager activation: every existing manifest keeps activating on enable, so this is back-compatible.

| Token | Meaning |
|---|---|
| `*` | Always active (eager) |
| `onStartup` | Fired once at boot, after `ServerState` exists |
| `onChat` | Declared and gated; firing wiring is a follow-on |
| `onCommand:<id>` | Declared and gated; firing wiring is a follow-on |
| `onRunnable:<id>` | Declared and gated; firing wiring is a follow-on |

When an event fires, `fire_activation_event` (`apps/core/src/server/mod.rs`) records it in a
process-global monotonic set (`FIRED_ACTIVATION_EVENTS` in `apps/core/src/runnable/mod.rs`) and runs
every installed-and-enabled plugin through `RunnableRegistry::register_active` against the new
snapshot. Eager plugins activate unconditionally; a gated plugin activates only once one of its
declared events has fired. The driver is idempotent and best-effort: a per-runnable failure is
logged, never fatal.

<Callout type="warn">
Only `onStartup` actually fires today (from `main.rs` at boot). `onChat`, `onCommand:<id>`, and
`onRunnable:<id>` are declared, validated, and gated, but the wiring that fires them from the chat
and command-palette paths is a follow-on. Do not rely on them activating a plugin yet.
</Callout>
