---
title: "Fine-Tuning"
description: "Train open models locally with LoRA/QLoRA via the Unsloth sidecar, merge to GGUF, and serve as the active chat model."
---

Fine-tuning lets you customize an open model's behavior on your own data — no API key, no cloud
dependency, fully local. Ryu wraps the Apache-2.0 [Unsloth](https://github.com/unslothai/unsloth)
library as a Core-managed sidecar, so you can train from the desktop, watch live progress via SSE,
merge the result to a GGUF, and serve it as the active chat model — all from one UI.

<Callout type="warn">
  Fine-tuning requires an NVIDIA CUDA GPU. When the local machine can't train, the job can be
  proxied to a remote Ryu Cloud GPU node.
</Callout>

## How it works

```
Desktop (FinetuningPage)
  └─ Core /api/finetune/*  (server/finetune.rs)
        ├─ finetune/store.rs          ~/.ryu/finetune.db
        ├─ finetune/adapters.rs       ~/.ryu/installed-adapters.json
        ├─ model_catalog/device.rs    GPU gate
        └─ sidecar/providers/unsloth  python -m ryu_unsloth
              FastLanguageModel → SFTTrainer → GGUF
```

Core decides **what runs** (a job on this node's GPU, or a remote node) and owns the durable job
record. The sidecar is a pure training runtime. The desktop is the UI.

## Lifecycle

1. **Check capability** — `GET /api/finetune/capability` reports whether local training is
   possible (GPU present, VRAM sufficient, sidecar running).
2. **Start a job** — `POST /api/finetune/start` with a base model, dataset, and optional LoRA /
   training config. Returns a `job_id` and initial `state`.
3. **Stream progress** — `GET /api/finetune/{id}/stream` is an SSE endpoint that proxies the
   sidecar's live training metrics (loss, epochs, ETA).
4. **Merge** — `POST /api/finetune/merge` converts the trained LoRA adapter into a standalone GGUF
   file ready for local inference.
5. **Serve** — The merged GGUF appears in the [Model Catalog](/docs/core/model-catalog) and can be
   set as the active chat model.

## Routes

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/api/finetune/capability` | `{ can_train_local, gpu, vram_bytes, reason, sidecar }` |
| `POST` | `/api/finetune/start` | Start a fine-tune job. Body: `{ base_model_id, dataset, output_name?, lora?, training? }` |
| `GET` | `/api/finetune/list` | List all jobs with current state |
| `GET` | `/api/finetune/{id}` | One job's details (prefers sidecar's live snapshot) |
| `DELETE` | `/api/finetune/{id}` | Cooperative cancel |
| `GET` | `/api/finetune/{id}/stream` | SSE progress stream |
| `POST` | `/api/finetune/merge` | Merge adapter → GGUF |
| `GET` | `/api/finetune/adapters` | List installed trained adapters |

## Durable job store

Jobs persist in `~/.ryu/finetune.db` (SQLite via rusqlite). The store is the system of record and
survives restarts. Each job tracks its state machine (`pending → training → completed | failed |
cancelled`), the base model ID, dataset path, LoRA config, output path, and training metrics.

## Adapter catalog

Merged adapters are tracked in `~/.ryu/installed-adapters.json` with provenance metadata (base
model, dataset, training config, creation date). The catalog filters by whether the GGUF file is
still present on disk, so deleting a file automatically removes the adapter from the list.
