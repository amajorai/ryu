---
title: "Side Questions (/btw)"
description: "An ephemeral question about the current conversation, with no tools and no persistence"
---

A side question lets you ask something quick *about the conversation you are already in* without
starting a new turn, spawning a tool, or leaving a trace. It is modeled on Claude Code's
interactive `/btw` and is the inverse of a sub-agent: full conversation context, zero tools. The
model sees the transcript, answers once, and Core persists nothing.

Core owns one stateless primitive, `POST /api/btw`, handled by `btw_handler` in
`apps/core/src/server/mod.rs`. It reuses the same single non-streaming gateway call as
[double-check](/docs/core/double-check) (`call_side_model`), so the model call still routes
through the Gateway like every other model call in Ryu.

For how to fire a side question from the composer, see
[Chat](/docs/using-ryu/user-guide/chat). This page documents the primitive.

## What it is

| Property | Value |
|---|---|
| Route | `POST /api/btw` (top-level, not under `/conversations/:id`) |
| Handler | `btw_handler`, `apps/core/src/server/mod.rs` |
| Backing call | `call_side_model` - one non-streaming gateway call |
| Tools | None. The system prompt tells the model it has no tools and cannot run commands, read files, or browse |
| Persistence | None. The question and answer never enter chat history |
| Returns | `{ answer, model }` |

The route is top-level rather than nested under a conversation because some clients (the CLI and
mobile) hold their own transcript and have no Core `conversation_id` to address.

## Request

The body is a `question` plus a transcript, supplied one of two ways.

| Field | Type | Required | Notes |
|---|---|---|---|
| `question` | string | yes | The side question. An empty/whitespace-only value returns `400` |
| `messages` | array of `{ role, content }` | no | A client-held transcript (CLI, mobile) |
| `conversation_id` | string | no | A stored Core conversation. The last `BTW_TRANSCRIPT_LIMIT` (30) messages are loaded |

If `messages` is present and non-empty it wins; otherwise Core loads the transcript from
`conversation_id`. If neither is supplied, the side question still answers, it just has no prior
context to draw on.

```json
POST /api/btw
{
  "question": "wait, which provider did we decide on earlier?",
  "conversation_id": "conv_01abc..."
}
```

```json
{
  "answer": "You chose OpenRouter routing the Gemini slug, since there is no direct Google provider.",
  "model": "..."
}
```

## Model and effort resolution

Nothing is hardcoded. The answering model and reasoning effort resolve through a preference chain,
matching the rest of Core's side-model surfaces.

| Setting | Preference | Env fallback | Final fallback |
|---|---|---|---|
| Model | `btw-model` | `RYU_BTW_MODEL`, then `RYU_DEFAULT_LLM_MODEL` | built-in default (`registry::DEFAULT_LLM_MODEL`) |
| Effort | `btw-effort` | `RYU_BTW_EFFORT` | resolver default |

## Behavior and guarantees

- **Stateless.** Core writes nothing. There is no row, no message, no run record for a side
  question.
- **No tools.** The system prompt instructs the model to answer only from the provided context and
  to say so briefly rather than guess when the answer is not there.
- **Single answer.** One gateway call in, one `{ answer, model }` out. There is no streaming and no
  follow-up turn.
- **Context-bounded.** When loading from `conversation_id`, only the most recent 30 messages
  (`BTW_TRANSCRIPT_LIMIT`) are included.
- **Fails loud on the model.** If the gateway model is unreachable the route returns `502` with
  `side-question model unavailable: ...`.

## Clients

Side questions are wired on all three clients that reach Core, each intercepting `/btw <question>`
before the message is sent.

| Client | Intercept | Transport | Surface |
|---|---|---|---|
| Desktop | `ChatPage` | `lib/api/btw.ts` (passes `conversation_id`) | `components/chat/BtwOverlay.tsx` dialog |
| CLI | `apps/cli` before send | `chat::ask_btw` (posts the held transcript) | ratatui overlay in `ui.rs`, dismiss Space/Enter/Esc |
| Mobile | `apps/native/app/(drawer)/ai.tsx` `onSubmit` | `lib/btw.ts` (posts the `useChat` transcript) | `components/btw-modal.tsx` |

<Callout type="warn">
The route, its `cargo check`, the CLI `cargo build`, and the desktop/mobile `tsc` + biome checks
pass, but the live answer round-trip needs a routable gateway model. It has not been
end-to-end verified against a running model.
</Callout>
