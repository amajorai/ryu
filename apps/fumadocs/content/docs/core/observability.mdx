Four small Core subsystems round out the backend: per-run traces, cross-device conversation sync, a
preferences key-value store, and per-engine resource sampling. They share a placement rule (Core vs
Gateway, see [Core vs Gateway](/docs/start-here/architecture/core-vs-gateway)): all four describe
*what runs* and *what state exists*, so they live in Core. Measurement of *what is paid for* (token
counts, cost, provider latency) stays in the [Gateway audit log](/docs/gateway/evals-audit).

## Traces

Core records an ordered span for each tool call and model call in a run
(`apps/core/src/server/trace.rs`), stored in SQLite at `~/.ryu/traces.db`. The run id is the
conversation id from the chat request.

A `Span` carries:

| Field | Type | Meaning |
|---|---|---|
| `id` | string | Stable row id (UUID) |
| `conversation_id` | string | The run this span belongs to |
| `kind` | string | `tool-call` or `model-call` |
| `name` | string | Tool name (tool-call) or model id (model-call) |
| `args_hash` | string, nullable | SHA-256 hex of the raw tool input - never the raw payload |
| `started_at` | i64 | Unix milliseconds, write time |
| `ended_at` | i64, nullable | Unix milliseconds, finish time; `None` while in-flight |
| `error` | string, nullable | Set when the span ended with a failure |
| `session_id` | string, nullable | Reserved link to the Gateway audit row |
| `seq` | i64 | Autoincrement ordering key, ascending within the run |

Tool arguments are stored only as a SHA-256 fingerprint (`args_hash`, produced by `hash_args`), never
the raw JSON, so sensitive input is not persisted in Core.

Fetch a run's spans in order:

```
GET /api/runs/:id/trace
```

Background runs that carry a `run_status` are listed by:

```
GET /api/runs
```

Both routes are registered in `apps/core/src/server/mod.rs` (`list_runs_handler`,
`get_run_trace_handler`).

Core deliberately records sequencing, fingerprints, latency, and errors but stores no token counts and
no cost. Those belong to the [Gateway audit log](/docs/gateway/evals-audit), where measurement lives.

<Callout type="warn">
  The `session_id` field on a span is reserved to link a span to its matching Gateway audit row, but
  it is not yet populated - it stays `None` until issue #176 threads the id through. Treat the
  cross-link as scaffolded, not live.
</Callout>

## Sync

Core ships a client for cross-device conversation sync (`apps/core/src/server/sync.rs`). When enabled,
it pushes every local conversation to the `:3000` server store and pulls everything changed since the
last successful pull, reusing the device-auth token from `~/.ryu/auth.json`.

| Property | Value |
|---|---|
| Push route | `POST /api/conversations-sync/push` |
| Pull route | `GET /api/conversations-sync/pull?since=<ms>` |
| Conflict rule | Last-writer-wins on conversation metadata by `updated_at` |
| Message merge | Union-merge by stable UUID (append-only, additive insert) |
| Interval | 60s (`SYNC_INTERVAL`) |
| Server base URL | `RYU_SERVER_URL`, default `http://localhost:3000` |

The loop is spawned unconditionally at startup (`spawn_sync_loop`, called from
`apps/core/src/main.rs`), but every tick is a no-op until you opt in. Errors are best-effort: logged
and swallowed, so a flaky network or a signed-out user never panics or stalls Core. With sync on but no
token present, the client returns `SyncError::Unauthenticated` and logs a one-time hint to complete
device login.

<Callout type="warn">
  Sync is **off by default** (local-first: Core never phones home unless asked). Enable it with the
  `RYU_SYNC_ENABLED` environment variable or the `cloud-sync-enabled` preference. The pull watermark
  is kept in memory only, so a restart does one extra full pull.
</Callout>

## Preferences

Core exposes a generic key-value preferences store with live change notifications
(`apps/core/src/server/preferences.rs`), backed by SQLite at `~/.ryu/preferences.db`. It is the seam
many features read their config from (sync, goals, auto-recall, the voice engine, and more), and the
mechanism that keeps separate processes in sync.

| Action | Endpoint |
|---|---|
| Get a preference | `GET /api/preferences/:key` |
| Set a preference | `PUT /api/preferences/:key` |
| Subscribe to changes | `GET /api/preferences/stream` (SSE) |

This is how the desktop and the Island companion stay aligned without shared browser storage: the
desktop publishes a value (for example a feature toggle), and the Island subscribes to the SSE stream
and updates live. The `stream` route is registered before the `:key` route so the static segment
matches first.

## Engine resource sampling

For the node selector, Core samples per-engine memory and CPU for the sidecars it owns a resident child
for (`apps/core/src/sidecar/resources.rs`). A `ResourceSample` is three numbers:

| Field | Type | Notes |
|---|---|---|
| `pid` | u32 | The sampled OS process id |
| `memory_bytes` | u64 | Resident-set size (RSS) |
| `cpu_percent` | f32 | Per single core; can exceed 100 across cores |

CPU is a *delta*: `sysinfo` reports usage since the previous refresh of a PID, so a one-shot read
always returns `0.0`. The `SidecarManager` background sampler therefore holds one long-lived `System`
and refreshes only the known sidecar PIDs every two seconds (`RESOURCE_SAMPLE_INTERVAL`), so the second
tick onward yields real CPU numbers. Only known PIDs are refreshed, never a full process scan.

The latest sample rides the existing status poll as optional fields (`pid`, `memory_bytes`,
`cpu_percent`) on:

```
GET /api/sidecar/status
```

They are serde-skipped when absent, so adopt-mode, serverless, and in-process engines that own no child
report no usage rather than zeros. The desktop joins this with the engine catalog to render the
node-selector breakdown.

## Crash reporting

Core reports Rust **panics** (only panics) to Sentry (`apps/core/src/crash.rs`). It deliberately does
not forward `tracing` or log events, because those can carry prompt or agent content and `~/.ryu`
paths - the only integrations are panic, backtrace, and contexts. A `before_send` hook scrubs the
user's home-dir prefix out of the panic message and every stack frame's path, `send_default_pii` is
forced off, and `server_name` is `None`, so the machine hostname never leaves the box.

This is a consent tier separate from product analytics. Two gates must both hold or the module is a
true no-op: the `crash-reports-enabled` preference (resolved via `crate::privacy::crash_reports_enabled`),
and a configured DSN (`SENTRY_DSN` or `RYU_SENTRY_DSN`). The resolved consent is mirrored into a
process-global that the gateway sidecar spawn-env reads, so Core forwards the same consent to the
gateway tier under `RYU_SENTRY_DSN`.

<Callout type="warn">
  The `crash-reports-enabled` pref is default-on (opt-out), but the whole module is a hard no-op until
  a DSN is configured, so a stock local install reports nothing. The Rust side reads the pref once at
  boot, so flipping the desktop toggle off takes effect on the next Core restart.
</Callout>

## Community-savings telemetry

Core runs an anonymous savings beacon (`apps/core/src/stats_beacon.rs`). Every five minutes
(`BEACON_INTERVAL`) an enabled tick reads the Gateway's public savings counters at
`GET /v1/savings` (the aggregate compression and cache metrics the Gateway measures, for example
`provider_cached_input_tokens` and `compression_tokens_saved`) and POSTs a snapshot to the control
plane at `/api/community/ingest`. Per the Core vs Gateway rule, Core owns consent and the phone-home
but never computes the measurement inline - it forwards what the Gateway already measured.

The row is keyed by a locally-minted random install id (`community-stats-install-id`, a uuid v4) plus
a per-process session id. No hostname, gateway key, org id, or any identity value is ever sent. Every
step is fail-open: a gateway-down, network, or parse error logs and skips the tick, and the beacon
never panics or blocks chat.

<Callout type="warn">
  The beacon is default-on (opt-out) and, unlike crash reporting, has no DSN gate, so a stock install
  phones anonymous aggregates home once the gateway is healthy. Turn it off with the
  `community-stats-enabled` preference (or `RYU_COMMUNITY_STATS_ENABLED`); the pref is re-read every
  tick, so an in-session opt-out stops egress at once. The public community page that surfaces these
  aggregates is out of scope here.
</Callout>
