---
title: "Programmatic tool calling (PTC)"
description: "Run model-authored JavaScript that fans out across many tools and returns only the final value, with Gateway-governed exec budgets and HITL pause/resume."
---

Programmatic tool calling lets a model author one JavaScript program that calls many tools, and
returns only its final value plus console logs. Intermediate tool results never re-enter the model.
The implementation lives in `apps/core/src/tool_exec/`.

## Why PTC

The usual chat tool loop sends one tool call per turn, and every intermediate result re-enters the
model's context. PTC inverts that: the model emits a single program that fans out across tools
through an injected `tools` proxy, and only its `return` value and `console.log` output come back.
That is the context-saving win documented in `apps/core/src/tool_exec/mod.rs` - intermediate
results stay inside the sandbox.

The program reaches tools through a proxy, for example:

```js
const issues = await tools.composio.GITHUB_LIST_ISSUES({ owner: "ryu", repo: "core" });
const open = issues.filter((i) => i.state === "open");
return open.length;
```

The `execute` meta-tool's own description (`apps/core/src/tool_exec/schema.rs`) steers the model to
call `tool_search` first to discover tool paths, then write a program. A tool path like
`tools.spider.crawl` maps to the registry id `spider__crawl`, and `tools.composio.<SLUG>` maps to
`composio__<SLUG>` (`apps/core/src/tool_exec/invoker.rs`, `tool_path_to_id`).

## Endpoints

Core exposes two routes (`apps/core/src/server/mod.rs`):

| Action | Endpoint |
|---|---|
| Run a program | `POST /api/tools/exec` |
| Resume a paused execution | `POST /api/tools/exec/resume` |

`POST /api/tools/exec` takes a required `agent_id`, the `code` to run, and an optional
`conversation_id` (used only to select the Composio entity for per-user connected accounts). It
returns a flattened outcome: `completed` with the final value plus logs, or `paused` awaiting a
connect step.

`POST /api/tools/exec/resume` takes `agent_id`, the `execution_id` returned by the pause, an `action`
of `accept`, `decline`, or `cancel`, and optional `content` (form values when the pause requested a
schema). An unknown id returns `404 execution_not_found`.

## Per-agent allowlist, no escalation

The agent's resolved tool allowlist travels unchanged into the execution. A program cannot reach a
tool the agent could not call in chat. `resolve_agent_allowlist` (`apps/core/src/tool_exec/mod.rs`)
is fail-closed: an absent or unknown `agent_id` is rejected before any code runs. The flagship `ryu`
agent defaults to no restriction; other agents are restricted to their allowlist. Tool calls route
through the same `McpRegistry::call_tool` path the chat tool loop uses
(`apps/core/src/tool_exec/invoker.rs`).

## Execution backend

The v1 default backend is a Deno subprocess (`apps/core/src/tool_exec/deno_backend.rs`). Core spawns
`deno run` with zero `--allow-*` flags plus `--no-prompt`, so the program has no network, no
filesystem, and no environment access, and any attempt fails rather than prompting. The `tools`
proxy travels over stdio, never the network: each call writes a tagged JSON request line to stdout,
and Core writes the tool result back to stdin. The privileged registry call happens in Core; the
sandbox itself can reach nothing.

The in-process `rquickjs` backend is gated behind the off-by-default `tool-exec-quickjs` cargo
feature and is not built for the default surface (`apps/core/src/tool_exec/mod.rs`). The
`CodeExecutor` enum keeps the choice reversible.

### Bounds

Each run carries hard limits (`apps/core/src/tool_exec/mod.rs`, `deno_backend.rs`):

| Bound | Default | Override |
|---|---|---|
| Wall-clock deadline | 30 seconds | - |
| V8 old-space memory | 256 MiB | `RYU_TOOL_EXEC_MEMORY_MB` |
| Max output (logs plus final value) | 30,000 bytes | - |
| Concurrently parked executions | 64 | - |
| Parked-execution TTL | 30 minutes | - |

A runaway is killed at the deadline. The deadline bounds active compute only, so a human wait during
a Composio pause does not count against it (that wait is bounded by the parked-store TTL). Tool
output, logs, and error text are control-stripped and length-capped before they re-enter the model.

## Human-in-the-loop pause and resume

When a tool call needs a connected account or consent, the registry returns a `__ryu_elicitation__`
envelope (`apps/core/src/tool_exec/invoker.rs`, `detect_elicitation`). The invoker suspends the whole
program: the host keeps the subprocess alive and blocked on stdin, parks it in the bounded global
store, and returns `paused` with an `execution_id` and the elicitation details.

`POST /api/tools/exec/resume` continues it. `accept` writes the form content so the pending call
resolves, `decline` makes that call reject (the program may catch it), and `cancel` kills the
subprocess. The resume is ownership-checked: only the agent that created the execution may resume it,
and a mismatch returns `404` exactly like an unknown id, so a foreign agent cannot use resume as an
existence oracle (`apps/core/src/tool_exec/deno_backend.rs`, `resume_parked`).

## Gateway governance

Per the Core vs Gateway rule, what runs is Core and what is measured is the Gateway. Every execution
and every resume brackets the work with a fail-closed gateway exec-budget check before it runs and an
exec-audit report after (`apps/core/src/tool_exec/mod.rs`, `execute_code` and
`resume_execution_opt`). Without the budget bracket a program could pause then resume to run an
unmetered, unaudited second segment.

The gateway also exposes a direct governance front, `POST /v1/exec/tool`
(`apps/gateway/src/tools/exec.rs`). It has three kinds:

| `kind` | Forwards to |
|---|---|
| `tool` (default) | Core `POST /api/mcp/tools/call` |
| `execute` | Core `POST /api/tools/exec` |
| `resume` | Core `POST /api/tools/exec/resume` |

This front is gated on a trusted-forwarder or master key, and is neutralized when mesh is enabled
(`(trusted_forwarder || master_key) && !mesh_enabled()`). It is inert unless the gateway has a Core
URL configured: without `CORE_URL` the gateway leaves its tool catalog unset and returns a clear
error (`apps/gateway/src/tools/catalog_client.rs`).

<Callout type="warn">
  The exec surface only runs when a code-execution backend is actually available. `is_available()`
  probes for the `deno` binary on `PATH`; with no backend feature it is always false. The default
  Deno backend therefore needs the `deno` binary installed, and the gateway exec front needs
  `CORE_URL` plus a trusted-forwarder or master key and a running Core to exercise end to end. The
  in-process `rquickjs` backend stays feature-gated off.
</Callout>

The desktop Tools page (`apps/desktop/src/pages/ToolsPage.tsx`) lists and test-calls tools over the
same MCP registry that PTC programs fan out across, filtered by the per-agent allowlist.
