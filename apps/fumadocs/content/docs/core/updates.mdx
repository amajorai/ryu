---
title: "Auto-update"
description: "One Core-computed update verdict, a shared toggle, and a native install mechanism per surface."
---

Ryu ships one consistent update experience across every surface (desktop, CLI, gateway, island, extension,
mobile). The pieces that are *shared* are the verdict, the toggle, and the toast UX. The install *mechanism*
is each platform's own native updater. Per the [Core vs Gateway](/docs/start-here/architecture/core-vs-gateway)
rule, deciding *what runs* (including which build of the binaries runs) is Core's job, so the update verdict
and the shared toggle live in Core (`apps/core/src/update/`). The Gateway governs what is allowed, measured,
and paid, never the install lifecycle.

## One verdict, native install

Ryu uses a **single release train**: one git tag (`vX.Y.Z`) builds and publishes every binary together, so
`core`, `gateway`, `cli`, and `desktop` always share a version. Core's own crate version
(`CARGO_PKG_VERSION`) is therefore the canonical Ryu version, and `current_version()` returns it.

Every client reads Core's one verdict instead of each re-implementing GitHub-release polling. What differs is
only how the bytes land:

| Surface | Mechanism | On launch |
|---|---|---|
| Desktop (Tauri) | `tauri-plugin-updater` (signed `latest.json` feed) | Checks Core; if auto-updates on (or forced) it downloads, installs, and relaunches, else a persistent "update available" toast with an install action (`apps/desktop/src/components/updater/AutoUpdater.tsx`). |
| CLI (Rust) | Manual download (prints the release URL) | Prints a one-line notice when an update is available; `ryu version` / `ryu update` show details. |
| Extension (Chrome) | Chrome Web Store (auto) | Notify-only toast, deduped per release. |
| Island (Electron) | `electron-updater` (GitHub feed) | Reads the shared `auto-updates` pref; auto-downloads when enabled; "Restart to update" in Settings. |
| Mobile (Expo) | `expo-updates` (EAS OTA) | Checks on launch when enabled; a Settings tab toggles it. |
| Gateway (Rust) | Updated with Core (managed sidecar) | Ships in the same release, replaced when Core is. |

The headless binaries (core, gateway, cli) have no native updater of their own, so Core also owns the binary
self-update for them (see [Applying an update](#applying-an-update)).

## Checking for an update

Three Core routes back the whole system. They are registered in `apps/core/src/server/mod.rs` and each has an
API reference page.

| Endpoint | Purpose |
|---|---|
| [`GET /api/version`](/docs/develop/api-reference/core/health/get_version) | Installed version plus the per-component build list and the running `os-arch` platform. |
| [`GET /api/update/check`](/docs/develop/api-reference/core/health/update_check) | Compares the installed version against the latest GitHub release and returns the verdict. |
| [`POST /api/update/apply`](/docs/develop/api-reference/core/health/update_apply) | Downloads and installs an update for the headless binaries. |

`GET /api/update/check` queries the latest release of `amajorai/ryu`, compares semver (a malformed tag parses
to `(0,0,0)`, so it never claims to be newer than a real version), picks the best release asset for the
running platform, and returns an `UpdateCheck`:

| Field | Meaning |
|---|---|
| `current` | Currently installed Ryu version. |
| `latest` | Latest published release tag, with a leading `v` stripped. |
| `update_available` | `true` only when `latest` is strictly newer than `current`. |
| `notes` | The GitHub release body, if any. |
| `html_url` | Link to the human-readable release page. |
| `asset` | The release asset matching this platform (`name`, `url`, `kind`, `size`), when one resolved. |

The asset matcher gates by OS first (filename and extension heuristics), then scores an arch bonus for
`x86_64`/`aarch64` and common aliases (`amd64`, `x64`, `arm64`), keeping the highest-scoring asset. The `kind`
is inferred from the extension (`msi`, `exe`, `dmg`, `appimage`, `deb`, `archive`, or `unknown`).

<Callout type="info">
  The check **fails open**: a network or API error is treated by callers as "no update known" rather than
  blocking launch. The desktop client mirrors this, returning a "no update" verdict on any error so a flaky
  check never stalls the UI.
</Callout>

## Applying an update

For the headless binaries `POST /api/update/apply` stages the asset and either self-replaces or hands the path
back (`apps/core/src/update/apply.rs`):

- A bare executable on Windows is swapped in place. A running `.exe` cannot be overwritten, so Core renames the
  live binary aside to `<exe>.old`, copies the new binary into the original slot, and reports
  `restart_required`. The `.old` crumb is removed on the next launch by `cleanup_stale_backup()`.
- An OS installer (`msi`, `dmg`, `deb`, `appimage`) is staged to disk and its path returned. The client runs
  the platform installer; Core does not launch GUI installers itself.

The download stages atomically: write to a `.part` file in `~/.ryu/updates/`, then rename into place.

## The shared auto-update toggle

The on/off switch is a single key in the cross-surface [preferences KV](/docs/core/observability#preferences),
`auto-updates`, stored as `{"enabled": bool}` and defaulting to enabled when unset. Every surface reads and
writes the *same* key (Core's `AUTO_UPDATE_PREF_KEY`), so flipping it on the desktop changes it everywhere.
This is the same publish/subscribe seam the rest of Ryu uses to keep separate processes aligned.

<Callout type="warn">
  The desktop currently sets a build-time `FORCE_AUTO_UPDATE = true` policy that installs an available update
  on every launch, **ignoring** the user toggle. This is deliberate for the free beta so nobody can sit on an
  old build; the toggle remains the source of truth for non-forced behaviour, and flipping the constant back to
  `false` restores user control with no other change. Forcing never hard-blocks the shell: if the signed feed is
  unreachable (an unsigned dev build), the installer degrades to a manual-download toast.
</Callout>

## Desktop signing

The desktop updater verifies a minisign signature on every downloaded artifact. The **public** key is committed
in `apps/desktop/src-tauri/tauri.conf.json` under `plugins.updater.pubkey`, and the feed endpoint is the
release's `latest.json`. The matching **private** key is generated locally
(`apps/desktop/.tauri/ryu-updater.key`), is **gitignored**, and is never committed. It is provided to the
release workflow only as a CI secret (`TAURI_SIGNING_PRIVATE_KEY`, with its password), so signed artifacts are
produced in CI and nowhere else.

<Callout type="warn">
  If the private signing key is lost, signed desktop updates can no longer be produced and installed apps stop
  auto-updating until reinstalled with a new key. It must be backed up in the team secret store.
</Callout>

## Developer note: forcing a verdict

To exercise the toast, toggle, and notice paths without a published release, set
`RYU_UPDATE_FAKE_LATEST=9.9.9` when running Core. The check short-circuits and returns a simulated
"update available" verdict (with no asset), so the verdict, toggle, and toast layers can be driven end to end.
Never set it in production.

<Callout type="warn">
  Status: code-complete across all surfaces, but the actual **install** steps (signed desktop updates, electron
  auto-update, EAS OTA, and the headless self-replace) cannot be exercised until a real signed release exists
  and the CI signing secrets are configured. The system is verified today only at the verdict, toggle, and toast
  layer via the fake-latest hook, not yet at the binary-install layer.
</Callout>
