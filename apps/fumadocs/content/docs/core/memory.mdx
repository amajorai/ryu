Ryu gives agents two kinds of recall. **Short-term memory** is the recent turns of the current
conversation, assembled from the conversation store. **Long-term memory** is durable facts carried
across conversations for a given user and agent. This page covers long-term memory
(`apps/core/src/server/memory.rs`); for short-term context see
[Conversations and sessions](/docs/core/conversations-sessions).

## How it works

Long-term memory is stored encrypted in the same SQLite database as conversations
(`~/.ryu/conversations.db`), in a `memory_entries` table.

- **Encryption** - each entry is sealed at rest with Core's unified `FieldCipher`
  (ChaCha20-Poly1305 AEAD), the same cipher used across the backend. The master key is resolved
  from the environment, then the OS keychain, then a local file, and a legacy `~/.ryu/memory.key`
  is adopted as that master key on first run. Each entry gets its own random nonce, and entries
  that fail to decrypt are skipped rather than crashing recall. See
  [Encryption at rest](/docs/core/encryption-at-rest) for the full model.
- **Per-agent** - one agent's memory does not bleed into another's. Memory is keyed by the user and
  agent pair.
- **Opt-in on the chat path** - long-term memory is recorded and recalled when the chat request
  enables it. By default Core recalls a small number of the most recent entries to prefix the model
  context.

The defaults are conservative: a short-term limit of the last 10 turns and a long-term recall limit
of 5 entries, both configurable on the request.

## What is and is not exposed

Memory is an **internal subsystem**. There is no public `/api/memory` route. Memory is written and
read on the chat path, and stored knowledge is searchable through the unified retrieval endpoint.

```
POST /api/retrieval/search
```

This searches both encrypted memory and your [Spaces](/docs/core/spaces-rag), and it is the same
retrieval path the desktop uses to surface relevant context. See the
[HTTP API reference](/docs/develop/api-reference) for the request shape.

<Callout type="info">
  Memory encryption uses a single local master key today. There is no key rotation or remote
  escrow. Treat that key as a secret: lose it and the encrypted entries become unreadable.
</Callout>

## Searchable chat history and auto-recall

Long-term memory is one source of recall. The other is your **past chat messages**, made
semantically searchable by a dedicated message index (`apps/core/src/server/message_index.rs`).

### The message index

Chat message bodies stay encrypted at rest in `conversations.db`. To avoid copying decrypted text
into a plaintext store, the index lives in its own database (`~/.ryu/message-embeddings.db`) and
holds **only vectors and metadata**, never the message text. Each row carries the `message_id`,
`conversation_id`, `role`, `embed_model`, `embed_dims`, and `created_at`. A search returns
`message_id`s, and the snippet text is re-read and decrypted from `conversations.db` at that point.

- **Indexed on write** - `ConversationStore::append_message` spawns a best-effort task that embeds
  the plaintext before it is sealed and inserts a vector row. The conversation DB mutex is never
  held across the embed.
- **Lazy backfill** - the first search embeds any not-yet-indexed messages, decrypting their stored
  content first, so chats already on disk become searchable.
- **Embedder-scoped** - rows are tagged with the embedding model. A query embedded by a different
  model never matches an incomparable vector space. If you swap to a different-dimension embedder
  and restart, the index recreates itself at the new width and re-backfills on the next search.

Search is exposed to agents as the built-in tool `search_conversations__search`
(`apps/core/src/sidecar/mcp/search_conversations.rs`). It is a reserved registry server, so it rides
the unified `call_tool` path: allowlist-gated and audited on both the ACP and openai-compat planes,
and it appears in `GET /api/tools/search`. When no message index is wired (CLI, tests, headless),
the tool returns an `ok:false` envelope rather than failing.

### Auto-recall

Before each chat turn, `route_chat_stream` (`apps/core/src/sidecar/adapters/mod.rs`) can
automatically retrieve relevant context and fold it into the `long_term_system` block, the same seam
skills use, so both the ACP and openai-compat planes inherit it. Two sources are pulled:

- relevant **past chat messages**, with the current conversation excluded
- relevant **long-term memory facts**, lazy-backfilled into the retrieval index and then deduped
  **by fact id** against what the recency window already injected, so a fact never appears twice

Auto-recall is **on by default**, gated by the `auto-recall-enabled` preference (env
`RYU_AUTO_RECALL_ENABLED`). The number of recalled snippets injected per turn, across both sources
combined, is the `auto-recall-top-k` preference, defaulting to 5. The desktop toggle lives in
Settings, Memory (`apps/desktop/src/components/settings/MemoryTab.tsx`). Auto-recall is orthogonal to
the memory record toggle: it controls what is recalled, not what is written.

<Callout type="warn">
  The whole auto-recall step runs inside a 4 second wall-clock budget and is fully fail-open: any
  embed, retrieve, or search error logs and skips recall for that turn, never blocking the reply.
  Live semantic quality needs a running Core plus the `llamacpp-embed` sidecar on `:8081`. With the
  embedder down, indexing on write and recall both no-op, and chats still send normally.
</Callout>
