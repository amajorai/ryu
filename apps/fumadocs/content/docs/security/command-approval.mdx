title: "Command approval and HITL"
description: "How Ryu scans shell commands before execution, when it stops for human approval, and the hardline blocklist that never yields."
---

Before Core runs any shell command a tool produced, it sends the command to the Gateway's exec-governance scanner. The scanner classifies the command and returns a verdict - allow, deny, or approval required. This is the human-in-the-loop gate that sits between untrusted model output and your shell.

## Modes

The mode comes from `RYU_EXEC_APPROVAL_MODE` and defaults to `manual`.

| Mode | Wire value | Behavior |
|---|---|---|
| Manual | `manual` (default) | Every matched pattern requires human approval |
| Smart | `smart` | A cheap classifier auto-allows low-risk commands, auto-denies clearly dangerous ones, and escalates the uncertain to human approval |
| Off | `off` | Pattern approval is disabled; the hardline blocklist below still always denies |

A session or global YOLO bypass, where one exists, can skip pattern approval - but the hardline blocklist ignores mode and YOLO entirely.

## The hardline blocklist

These always return `deny`, regardless of mode, YOLO, or any override. They are catastrophic and have no legitimate agent use.

- `rm -rf /` and its variants
- `rm -rf --no-preserve-root`
- the fork bomb `:(){ :|:& };:`
- `mkfs.*` targeting a root or mounted device
- `dd if=/dev/zero of=/dev/sd*` (disk zeroing)
- piping an untrusted URL straight to a shell at the filesystem root

## Pattern categories

Beyond the hardline set, roughly forty compiled regex patterns flag commands for approval (in `manual`) or classification (in `smart`). Patterns are compiled once at the top level, never inside the request path.

| Category | Examples |
|---|---|
| Recursive destructive | `rm -r`, `chmod 777 -R`, `chown -R root` |
| System modification | `mkfs`, `dd if=`, writes to `/dev/sd*`, `/etc/`, `~/.ssh/` |
| Service control | `systemctl stop/restart/disable/mask` |
| Process kill | `kill -9 -1`, `pkill -9` |
| Code execution | `bash -c`, `python -c`, `curl \| sh`, `bash <(curl ...)` |
| Sensitive overwrites | `tee` or `>` / `>>` to sensitive paths |
| Dangerous combos | `find -exec rm`, `xargs rm`, `sed -i` on `/etc/` |

## The scan endpoint

Core posts the command to the Gateway's exec-governance plane before execution.

```
POST /v1/exec/scan
```

Request:

```json
{
  "backend": "local",
  "command": "rm -rf ./build",
  "session_id": "sess_123",
  "agent": "coder"
}
```

Response:

```json
{
  "decision": "approval_required",
  "reason": "matched recursive destructive pattern",
  "findings": [
    { "pattern": "rm -r", "category": "recursive_destructive", "severity": "high" }
  ]
}
```

`decision` is one of `allow`, `deny`, or `approval_required`. Each finding carries the matched `pattern`, its `category`, and a `severity` of `low`, `medium`, `high`, or `critical`.

## How Core maps the verdict

| Verdict | Core action |
|---|---|
| `allow` | Proceed and run the command |
| `deny` | Block, and post an exec audit event |
| `approval_required` | Depends on the exec path (see below) |

How `approval_required` resolves depends on where the command originated:

- Core PTC and tool-exec path: this path is synchronous and deadline-bounded, and there is no in-process machinery to await an approval decision, so `approval_required` currently fails closed - deny, audit, and log. Wiring a true blocking approvals-inbox gate here is follow-on work.
- Interactive ACP seam: the request surfaces to the user as a permission prompt and execution blocks on their decision (allow or reject). In headless ACP sessions there is no human to ask, so `approval_required` also fails closed to reject.

## Fail-closed

If the scanner is unreachable, Core default-denies. The only escape hatch is `RYU_ALLOW_GATEWAY_FALLBACK=1`, which mirrors the exec-budget fallback semantics exactly. Do not set it in a shared or server deployment.

## ACP seam

ACP agents (Claude Code, Codex, and the flagship Pi engine) run their own shell, so Ryu gates them at the ACP permission-request seam. When an agent asks to run a tool and `RYU_EXEC_APPROVAL_MODE` is not `off`, Core extracts the command from the request and runs it through the same scanner before granting: a `deny` verdict rejects the permission outright - even in a headless session that would otherwise auto-approve - and `approval_required` rejects in headless mode or surfaces to the user in interactive mode.

Command extraction is best-effort (the scanner is accident prevention, not containment): a tool call with no recoverable command string is not scanned. Note that ACP agents are first-party binaries that legitimately need their own credentials, so Ryu gates their commands rather than scrubbing their environment - see [Credential and environment scrubbing](/docs/security/credential-scrubbing). Command approval also composes with each agent's permission mode; see [ACP session controls](/docs/develop).

<Callout type="warn">
Smart mode's classifier may be heuristic or a stub at ship - if so, prefer `manual` anywhere the blast radius matters. The hardline blocklist is the only layer guaranteed to deny unconditionally; everything else depends on the configured mode. In shared or server deployments keep `RYU_EXEC_APPROVAL_MODE=manual` and never set `RYU_ALLOW_GATEWAY_FALLBACK=1`.
</Callout>
