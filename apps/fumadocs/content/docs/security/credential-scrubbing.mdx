---
title: "Credential and environment scrubbing"
description: "How Ryu strips secrets out of the environment before spawning any subprocess."
---

When Core spawns an untrusted subprocess - an MCP stdio server or the Deno programmatic-tool-calling sidecar - it does not hand the child your full environment. A shared helper scrubs secret-like variables out of the inherited environment and passes only an explicit allowlist plus whatever that call site genuinely needs.

ACP agents (Claude Code, Codex, Pi) are deliberately excluded from env-scrubbing: they are first-party binaries you chose to run, and they legitimately need their own credentials (OAuth tokens, `GH_TOKEN`, provider keys) to work. Stripping those would break them. ACP agents are governed by [command approval](/docs/security/command-approval) at the permission seam instead, not by environment scrubbing.

## The helper

The scrubbing lives in one place - `scrub_child_env` in `apps/core/src/sidecar/env_scrub.rs`. The default rule drops any variable whose key matches `(?i)(KEY|TOKEN|SECRET|PASSWORD|PASSWD|CREDENTIAL|AUTH)` unless that key is in the explicit passthrough allowlist for the call site.

```
scrub_child_env(base, extra_allow) -> Vec<(String, String)>
```

`base` is the candidate environment, `extra_allow` is the call site's additional passthrough keys. Anything secret-like and not allowlisted is stripped.

## Per-site allowlists

| Spawn site | Passthrough |
|---|---|
| MCP subprocess | `PATH`, `HOME`, `USER`, `LANG`, `LC_ALL`, `TERM`, `SHELL`, `TMPDIR`, any `XDG_*`, the Windows essentials (`SYSTEMROOT`, `APPDATA`, `LOCALAPPDATA`, `PATHEXT`, `COMSPEC`, `USERPROFILE`, `TEMP`, ...), plus the server config's own `env` block |
| Deno PTC sidecar | the normal env minus secret-like keys (deny-list scrub), keeping only what Deno needs to launch |
| ACP agent | not scrubbed - gated by [command approval](/docs/security/command-approval) instead (see above) |

The MCP allowlist is deliberately tiny and exact. An MCP server that needs a specific credential declares it in its own config `env` block, which is passed through explicitly - it is never inherited silently from your shell.

## Defense in depth with the vault

Environment scrubbing pairs with the identity vault - the LLM never sees raw credentials in the prompt, and child processes never inherit them from the environment. A secret reaches a subprocess only when a call site explicitly allowlists the exact key, so a compromised or misbehaving tool cannot read, say, your `OPENAI_API_KEY` just because it happened to be exported in your shell. See [Identity vault](/docs/core/identity-vault).
