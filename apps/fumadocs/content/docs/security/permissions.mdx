title: "Permissions and roles"
description: "The org/team RBAC permission vocabulary, built-in roles, and how Core enforces them on a shared node."
---

Ryu is local-first and single-tenant by default, but a node can be bound to an organization (a shared "company brain"). On such a node, Core enforces org/team role-based access control on its REST surface, keyed to a per-user identity verified from a signed JWT. The permission vocabulary and the built-in role sets are a contract shared byte-for-byte with the TypeScript control plane (`apps/core/src/identity_verify/permissions.rs`), so an effective-permission check agrees on both sides.

<Callout type="warn">
This is Core-side enforcement of a caller's role. Scoped API tokens and MCP OAuth key exchange (a narrower per-key capability grant) are a separate track and are not built yet. See [Scoped tokens](/docs/security/trust-model) for the current trust boundary.
</Callout>

## The permission vocabulary

17 `resource.action` keys, defined in `PERMISSIONS` (`apps/core/src/identity_verify/permissions.rs`). The Rust constants and the TypeScript control-plane array must match exactly.

| Key | Grants |
|---|---|
| `gateway.view` | Read gateway config and status |
| `gateway.configure` | Edit gateway config (firewall, budgets, auth, routing) |
| `workflow.view` | List and read workflows |
| `workflow.run` | Execute a workflow |
| `workflow.edit` | Create or edit a workflow |
| `workflow.delete` | Delete a workflow |
| `space.read` | Read a space's documents |
| `space.write` | Write a space's documents |
| `space.delete` | Delete a space or its documents |
| `agent.view` | List and read agents |
| `agent.run` | Run an agent |
| `agent.edit` | Create or edit an agent |
| `tool.exec` | Execute a tool |
| `members.manage` | Add, remove, and re-role members |
| `roles.manage` | Create, edit, and assign custom roles |
| `billing.manage` | Manage billing and subscriptions |
| `audit.view` | Read the audit log |

## Built-in roles

Four built-in roles map from a caller's Better Auth org role (`permissions_for_role`). An unknown role fails closed to `viewer` (the least-privileged floor).

| Role | Permission set |
|---|---|
| `owner` | Every permission |
| `admin` | Every permission except `billing.manage` |
| `member` | `gateway.view`, `workflow.view`, `workflow.run`, `space.read`, `space.write`, `agent.view`, `agent.run`, `tool.exec` |
| `viewer` | `gateway.view`, `workflow.view`, `space.read`, `agent.view` |

A member's **effective** permissions are the built-in set of their role UNION every permission granted by a custom role assigned to them in the org. The role tier is the fail-closed fallback when the custom-role lookup is unavailable.

## How Core gates a request

Every request may carry a signed user identity in the `x-ryu-user-jwt` header (`USER_JWT_HEADER`, `apps/core/src/server/mod.rs`). Middleware verifies it offline (EdDSA against the control plane's JWKS) and attaches an optional `VerifiedCaller`. A handler then calls `enforce_permission(state, caller, perm)`.

Enforcement branches on whether **this node** is bound to an org (`registered_org()`), because per-user RBAC only makes sense on a shared node and the single-user local flow must never be degraded.

| Node state | Caller | Result |
|---|---|---|
| Unbound (local/dev) | any (or none) | Allow - the shared `RYU_TOKEN` already implies one trusted operator |
| Org-bound | No / invalid JWT | Deny (403) - a tokenless caller must not inherit full trust |
| Org-bound | Valid JWT for a different org | Deny all (403) - a non-member holds nothing here |
| Org-bound | Member of the node's org | Allow iff `perm` is in their effective set |

For a member, the built-in role tier is checked first (no network on the common path). On a miss, Core resolves the custom-role slice from the control plane and unions it in.

### Handlers gated today

Core wires `enforce_permission` into these handler families (call sites in `apps/core/src/server/mod.rs`):

| Permission | Surface |
|---|---|
| `gateway.configure` | `PUT /api/gateway/config` (the "employees can't edit the gateway" gate) |
| `workflow.view` / `run` / `edit` / `delete` | Workflow REST handlers |
| `space.read` / `write` / `delete` | Space document REST handlers |
| `agent.view` / `run` / `edit` | Agent REST handlers |

The remaining keys (`tool.exec`, `members.manage`, `roles.manage`, `billing.manage`, `audit.view`) are part of the shared vocabulary; member, role, and billing management are enforced on the control-plane side (the epic's `roleSatisfies` gates), and audit reads are governed by the [Gateway audit log's](/docs/gateway/evals-audit) own master-key gate.

## Custom-role resolution and caching

The custom-role slice Core cannot compute locally is fetched from the control plane (`resolve_permissions`, `apps/core/src/sidecar/control_plane.rs`). Core sends only its `x-gateway-key` credential and the `userId`; the org is derived server-side from the key.

- The lookup is **fail-closed** - any failure (no gateway key, network error, non-2xx, decode error) returns an empty set, never an error and never a wide set. The caller falls back to the built-in role tier alone (never full access).
- Only **successful** lookups are cached, for `PERMISSIONS_TTL` (30 seconds), keyed by user id. A transient failure is not cached, so it never denies a user for the whole window. A role or grant change propagates within 30 seconds.
