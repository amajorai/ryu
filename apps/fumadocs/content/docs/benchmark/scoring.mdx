## The scorecard

Every run emits a JSON record (the machine artifact) and a Markdown scorecard (the human one),
both led by the **pinned config** so a reader can reproduce the number. A score is a tuple, never
a bare scalar:

```
model · driver · docs condition · core version · gateway version · task-set version · k
```

## Two reliability numbers

- **Pass rate** — fraction of attempts that passed. The headline.
- **pass^k** — fraction of *tasks* whose every one of `k` repetitions passed. Stricter; it
  catches tasks that pass once by luck and fail on repeat. Run with `--k 3` or higher for a
  reliability read.

Both come with a **Wilson 95% confidence interval**, so small task counts show honest error bars
instead of a falsely precise percentage.

<Callout type="info">
  The headline is **Tier A only**. Tier B (computer-use) is reported on its own line and never
  folded into the main number, because it is not deterministically reproducible.
</Callout>

## Cost normalization

Ryu's Gateway already meters spend, so RyuBench reports **`$/task` and `tokens/task`** next to
pass rate. Cost-normalized scoring is exactly what the 2026 benchmark pushback demands, and it
is nearly free here because measurement is the Gateway's job. Prices are a swappable default
(override with the `RYUBENCH_PRICES` env var, JSON of `{ "<model>": { "in": n, "out": n } }`);
local models are priced at zero tokens — their real cost is compute, tracked elsewhere.

## LLM-judge is an auditor, not a scorer

RyuBench never uses an LLM to grade a task. If you want to audit verifier quality, run an LLM
judge *against the execution verifiers* and measure disagreement — the DeepSWE lesson was that
purpose-written verifiers disagreed with a judge on 1.4% of rollouts versus 32.4% for inherited
tests. A judge that disagrees with your verifier means your verifier is wrong, not the model.

## Anti-gaming rules (open-core means the routes are public)

Ryu's Core and Gateway are open source, so the task routes are known. The benchmark is designed
for that:

- **Inverted split.** The public task set is demonstration-only and **never scored** — you can
  even ship a scripted solver that aces it, to prove it is not the benchmark. Only a **private,
  out-of-distribution set** counts, and that set is hosted, not shipped. (v1 ships the public set;
  the sealed private tier is staged next — see the spec.)
- **Fresh, OOD private tasks.** New tasks are authored from scratch and preferentially allocated
  to the private set; the public and private tiers are difficulty-calibrated.
- **Rotate hidden tasks.** Reusing the same hidden set across many releases leaks it through the
  leaderboard; rotate to avoid the feedback loop.
- **Open-source-to-score.** Official standing requires publishing the harness config so a number
  can be reproduced.
- **Pin everything.** Core/Gateway version and config, agent card, model version and sampling
  params, task-set version — all captured in the scorecard.

## Reproducibility &amp; the community harness

RyuBench is meant to be run by anyone, not just internally. The community-runnable target is a
sandboxed `docker run` against a fresh pinned node: it seeds the reference agent card, runs the
public task set, and prints a scorecard (pass rate, `$/task`, `tokens/task`, pass^k). The scored
run is offline and pinned so results are reproducible and not silently phoning a bigger model.

## Drivers (the model-vs-harness split, operationally)

| Driver | Track | What it does |
|---|---|---|
| `direct-tool` | **model** | fixed minimal scaffold: one `ryu_request` HTTP proxy tool, OpenAI tool-calling loop against the node's governed `/v1/chat/completions`. Swap `--model` to compare models on equal footing. |
| `ryu-agent` | **native / harness** | drives the task through Ryu's real agent runtime (`POST /api/chat/stream`) on a frozen reference card with only the `chat_model` slot swapped. |

The scorecard records which driver produced a score, so a model-track number and a harness-track
number can never be confused for each other.
