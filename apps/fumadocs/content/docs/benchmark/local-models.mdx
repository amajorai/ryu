RyuBench is model-agnostic. It POSTs a model id to the node's governed
`/v1/chat/completions`, and the Gateway routes that id to whichever provider serves it —
including the **local llama.cpp engine**. So yes, you can benchmark local models. There is one
real caveat about tool-calling; read it before you trust a low local score.

## How a local model is selected

Routing is prefix-based in the Gateway's `ModelRouter`. Any model id starting with one of these
prefixes maps to the **Local** provider (the llama.cpp `llama-server`):

```
llama · mistral · mixtral · gemma · phi · qwen · deepseek · apple-foundationmodel
```

The default local chat model is `gemma-4-E2B-it-Q4_K_M`. `llama-server` ignores the id string
and answers with whichever GGUF is currently loaded (the registry default or your active-model
override), so the id only decides routing.

```bash
# routes to the local engine
bun run apps/benchmark run --model gemma-4-E2B-it-Q4_K_M --k 3
bun run apps/benchmark run --model qwen3-32b-instruct-q4   # also local (qwen prefix)
```

The local engine is upstream llama.cpp's OpenAI-compatible `llama-server`, run as a
Core-managed sidecar on `127.0.0.1:8080` with continuous batching on. Its response `usage`
carries real `prompt_tokens` / `completion_tokens`, so **`tokens/task` is reported for local
models** just like cloud ones. Token **cost** is priced at zero — a local model's real cost is
compute, tracked outside the token bill.

<Callout type="warn">
  **Tool-calling requires `--jinja`.** The `direct-tool` driver works by giving the model one
  tool (`ryu_request`) and reading `tool_calls` back. llama.cpp only parses generated tool
  syntax into `tool_calls` when `llama-server` is launched with `--jinja` — and Ryu does **not**
  pass `--jinja` by default. So a default Gemma run will score ~0 on the `direct-tool` driver
  not because the model can't operate Ryu, but because it never emits a tool call.
</Callout>

## Making local tool-calling work

Two things must both be true:

1. **Launch `llama-server` with `--jinja`.** Enable the per-model `jinja` launch flag in the
   inference launch config (`LaunchConfig.jinja = true`). Without it, no `tool_calls`.
2. **Use a tool-capable GGUF.** The loaded model's chat template must support tools. Small
   instruct Gemma quants often do not; a tool-tuned model (e.g. a Qwen or Llama instruct build
   with a tools template) does. This is an upstream llama.cpp / model concern, not Ryu code.

RyuBench already sends the `x-ryu-raw-tools` header on its completion calls, which forces the
Gateway's **Plain** passthrough branch — so the model's `tool_calls` are returned to the harness
verbatim instead of being intercepted and executed server-side by the unified tool catalog. You
do not need to configure anything on the Gateway side for this.

## If your local model can't tool-call

That is a legitimate benchmark signal, but distinguish it from a harness limitation:

- A **near-zero** `direct-tool` score with a non-tool model means "this model + this scaffold
  can't operate Ryu," which is true but uninteresting.
- To measure a non-tool-calling local model fairly, use the **`ryu-agent`** driver, which drives
  the task through Ryu's native agent runtime (`/api/chat/stream`) where tool orchestration is
  handled by Ryu's own loop, not by the model emitting OpenAI `tool_calls`:

```bash
bun run apps/benchmark run --driver ryu-agent --model gemma-4-E2B-it-Q4_K_M
```

<Callout type="info">
  Keep model-track and harness-track numbers separate. A local model measured under `ryu-agent`
  (native orchestration) is not comparable to a cloud model measured under `direct-tool` (the
  model does its own tool-calling). The scorecard pins the driver so the two never merge.
</Callout>

## Practical recipe

```bash
# 1. make sure a local engine is active and a tool-capable GGUF is loaded
#    (enable the per-model jinja flag for direct-tool tool-calling)

# 2. run a small, cheap slice first to confirm tool_calls flow
bun run apps/benchmark run --model <local-id> --category agent --k 1

# 3. if agents create correctly, run the full Tier-A suite
bun run apps/benchmark run --model <local-id> --k 3 --out ./.ryubench/local
```

If step 2 shows every task failing with "agent never created …", the model is not emitting tool
calls — revisit `--jinja` and the GGUF's tools template, or switch to the `ryu-agent` driver.
