title: Deploying Ryu

When you deploy Ryu for a team, two open-source Rust services do the work: the Core and the Gateway. The Core orchestrates what runs (agents, sessions, tools, workflows). The Gateway governs every model call (routing, firewall, budgets, audit). The Core is Apache-2.0 and the Gateway is AGPL v3; both are open-source and self-hostable. The desktop, web, and backend stay closed, which is the open core plus managed cloud model in practice.

## What gets deployed

A Ryu deployment is the Core plus the Gateway. A few properties matter for a server:

- Headless-first: the Core runs with no UI, no cloud requirement, and no streaming required.
- Encrypted by default, with no telemetry by default.
- The Gateway sits on the default chat path, so a hosted node is already governed the moment it serves traffic.

You point your clients (desktop, CLI, bots) at the node, and they use it like any other Ryu node.

## Self-hosting is about trust

Self-hosting Ryu is not only about cost or control. The parts that orchestrate open-source engines are themselves open. You can read the Core and the Gateway, audit how model calls are routed and governed, and run them on your own hardware. Nothing about the control layer is a black box.

<Callout type="info">
  The Gateway being on the default path means self-hosting does not trade away governance. The firewall, routing, and audit travel with the deployment.
</Callout>

## Managed Cloud and provisioning

There is one-command cloud provisioning, an hcloud-based provision flow with a backups add-on and deploy docs. That gets a node stood up quickly.

Be honest about the rest. Much of managed Ryu Cloud is a service offering, not a single button: A Major audits your needs, builds the agents, deploys them, and hosts them for you. The provisioning command is real, but "we run it for you" is the part that is people plus process.

<Callout type="warn">
  Do not expect the full managed Cloud experience from a provisioning script alone. The script stands up infrastructure; the audit, build, and ongoing operation are the service layer on top.
</Callout>
