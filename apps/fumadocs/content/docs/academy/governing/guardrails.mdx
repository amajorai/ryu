title: Guardrails

Guardrails are the firewall and data-protection surfaces of the Gateway. They are how you decide what is safe to send and what should be screened before it ever reaches a provider.

## The firewall

The firewall screens requests, including prompt-injection style protections. It draws on security tooling wired into the Gateway, so the protection is part of the control layer rather than something each agent has to add on its own.

## PII and DLP

PII and DLP controls govern what data is safe to send on egress. This is about keeping sensitive data from leaving on a model call you did not intend. Because these run at the Gateway, they apply across every governed call rather than being re-implemented per agent.

These are Ryu's own user-level guardrails. They are distinct from anything an upstream provider offers, so you keep control even when the request is destined for a third-party model.

<Callout type="warn">
  Be honest about maturity: some of these config UIs are still being surfaced in the desktop. The data-plane wiring exists and the Gateway is on the path, so governed calls already flow through the control layer, but expect the configuration surfaces to keep filling in.
</Callout>

## Egress compression

There is also an optional egress compression transform that can shrink request messages before the provider call. Like the rest of the guardrail surfaces, it fails open: if compression cannot run, the original request goes through unchanged rather than failing.

<Callout type="info">
  Failing open is a recurring theme across the Gateway. Guardrails and transforms improve safety and efficiency, but they are designed not to become a new way for requests to break.
</Callout>
