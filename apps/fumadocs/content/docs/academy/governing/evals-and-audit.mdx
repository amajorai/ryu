Measurement and audit are Gateway responsibilities. Evals tell you whether an agent is good. Audit lets you prove what it did. Together they close the loop on governance.

## Evals

Evals measure output quality. The Gateway integrates an evals capability so you can replay prompts (with per-case variables) through the pipeline and score the outputs against a judge rubric. This is how you move from "the agent feels fine" to evidence that an agent meets a bar.

## Audit

Audit is the trail of what happened. Every governed call and every tool execution emits an audit record. When you need to answer what an agent did, when, and at what cost, the audit trail is the source of truth.

<Callout type="info">
  Evals plus audit answer two different questions. Evals answer "is this agent good?" Audit answers "what exactly did it do?" You want both.
</Callout>

## Observability on the way

There is read-only observability in the form of per-run traces heading to the desktop. Be honest that this surface is still arriving: the measurement and audit responsibilities live in the Gateway today, and the per-run trace viewing experience is being surfaced.

## A safety note on admin endpoints

Admin endpoints like config and audit are loopback-restricted for safety. They are not meant to be reachable from across a network by default, which keeps the most sensitive control surfaces close to the machine that owns them.

<Callout type="warn">
  Do not assume admin surfaces are remotely reachable. The loopback restriction is a deliberate safety boundary, not a limitation to work around.
</Callout>
