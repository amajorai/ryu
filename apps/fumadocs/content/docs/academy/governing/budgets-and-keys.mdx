What is paid for is a Gateway concern. That is why budgets, keys, and the way you connect agents all live here rather than in Core.

## Budgets

Budgets cap spend. The Gateway is the place model spend is measured, and it tracks and enforces an exec-budget for tool and programmatic execution. So when an agent runs tools, the cost of those executions is counted and bounded at the control layer, not left to each agent to police.

## BYOK: bring your own key

BYOK means you store provider keys in a key vault rather than hardcoding them anywhere. The Gateway injects the key when it makes the provider call. Keeping keys in the vault means they live in one governed place, and an agent never needs to hold a raw provider key itself.

## BYOA: bring your own agent

BYOA means pointing any OpenAI-compatible agent at the Gateway by swapping its base URL. You govern an agent you already run, without replacing it. The agent keeps working as before, except now its calls flow through routing, guardrails, budgets, and audit.

<Callout type="info">
  BYOA builds directly on the fact that the Gateway is OpenAI-compatible. If an agent can point at OpenAI, it can point at the Gateway with a base-URL change.
</Callout>

## Why all of this is a Gateway concern

Keys and budgets are about what is paid for, which is squarely the Gateway's job per the Core-vs-Gateway rule. Core decides what runs; the Gateway decides what it costs and which credentials it may use.

<Callout type="warn">
  BYOA governs an agent's calls when they route through the Gateway. Remember the honest limit from earlier: an ACP subprocess agent that makes its own internal provider calls cannot have those chat calls intercepted, though its tool execution is still budgeted and audited.
</Callout>
