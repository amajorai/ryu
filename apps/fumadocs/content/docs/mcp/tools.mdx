---

The Ryu Core MCP server publishes a focused set of `ryu_*` tools that map onto Core capabilities. Each tool translates an MCP call into a typed Core HTTP request and returns the result as JSON. Read tools are safe to call freely; write tools change state on the node and are called out below.

<Callout type="info">
  Run the host's tool inspector after connecting to see the live list and schemas for the node you are pointed at. `tools/list` works even when Core is down, since the definitions are static.
</Callout>

## System

- `ryu_health` - probe whether the Core node is alive.
- `ryu_system_info` - live hardware snapshot: CPU, RAM, disk, GPU and VRAM, OS.
- `ryu_system_status` - service status: active engine, engine running state, sidecars, gateway reachability, mesh.

## Agents and teams

- `ryu_list_agents` - list the agents configured on the node.
- `ryu_list_teams` - list the multi-agent teams configured on the node.

## Models and engines

- `ryu_search_models` - search the model catalog (Hugging Face GGUF by default).
- `ryu_get_active_model` - read which installed model the local chat engine serves.
- `ryu_set_active_model` - switch the served model to an installed one. Write.
- `ryu_list_engines` - list the inference engines and their installed models.

## Skills

- `ryu_list_skills` - list the skills installed on the node and their active state.
- `ryu_search_skills` - search the skills directory.
- `ryu_install_skill` - install a skill by catalog id and hot-reload Core's registry. Write.

## Workflows

- `ryu_list_workflows` - list the workflows defined on the node.
- `ryu_run_workflow` - run a workflow by id with an optional string input map. Write.

## MCP bridge

- `ryu_list_mcp_servers` - list the MCP servers Ryu has registered.
- `ryu_call_mcp_tool` - invoke a tool on any MCP server the node has registered, scoped to a registered agent's allowlist. This is what makes the node a hub onto other systems. Write.

## Spaces and retrieval

- `ryu_list_spaces` - list the knowledge Spaces (document collections) on the node.
- `ryu_search_space` - semantic search within a single Space.
- `ryu_search_retrieval` - unified RAG search across memory and all Spaces.

## Ask

- `ryu_ask` - ask Ryu a single synchronous question, routed to the node's side-question model. Pass a `conversationId` to ground it in an existing conversation, or omit it for an ephemeral context. Consumes model budget. Write.

## Identity

- `ryu_whoami` - report the Ryu user this server is signed in as, or prompt to run `ryu-mcp login`. See [security](/docs/mcp/security) for the device-authorization flow.

## Read versus write

The write tools are `ryu_set_active_model`, `ryu_install_skill`, `ryu_run_workflow`, `ryu_call_mcp_tool`, and `ryu_ask`. They change state on the node, consume model budget, or both. Everything else is read-only. The Gateway governs every model call a tool makes, so an `ryu_ask` from an MCP host is subject to the same routing, budget, and audit as a call from any other surface. See [security](/docs/mcp/security) for the write-tool caution.
