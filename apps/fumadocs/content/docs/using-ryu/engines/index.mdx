Ryu drives agents through their real runtimes. Each runtime, provider, and tool process is a
**sidecar**: a separate process that Core downloads, checksum-verifies, installs, starts, and
health-checks for you (`apps/core/src/sidecar/`). You never touch a terminal or a package manager.

Nothing is hardcoded. Every engine, model, and strategy is a swappable default resolved through one
registry (`~/.ryu/registry.json`) and environment variables, never a lock.

## The lifecycle

Every sidecar implements one trait with `start`, `stop`, `health_check`, and `uninstall`. The
`SidecarManager` orchestrates them.

- **Download and verify** - the download manager fetches the right asset for your platform, verifies
  its SHA-256, retries with backoff, extracts archives, and installs atomically.
- **Storage** - binaries go to `~/.ryu/bin/`, model weights to `~/.ryu/models/`, and per-sidecar data
  to `~/.ryu/data/<name>/`. Installed versions are tracked in `~/.ryu/versions.json` so an up-to-date
  binary is not re-downloaded.
- **Startup order** - a defined boot sequence starts the installed sidecars. Voice engines are
  deliberately not in the startup order; they start on demand.
- **Health** - each sidecar is health-checked on an interval. Health is observable but Core does not
  auto-restart on degradation.

## Managing sidecars

| Action | Endpoint |
|---|---|
| Install progress for all sidecars | `GET /api/catalog` |
| Full install status | `GET /api/setup/status` |
| Install a sidecar | `POST /api/setup/:name/install` |
| Uninstall a sidecar | `POST /api/setup/:name/uninstall` |
| Running status of all sidecars | `GET /api/sidecar/status` |
| Start, stop, restart one | `POST /api/sidecar/:name/{start,stop,restart}` |
| Start or stop all | `POST /api/sidecar/{start-all,stop-all}` |

## Per-engine resource usage

Core samples how much each engine is using so the desktop can answer "what is this engine costing me
right now". A background sampler on the `SidecarManager` refreshes per-process resident memory (RSS)
and CPU usage every two seconds for the sidecars Core owns a child process for
(`apps/core/src/sidecar/resources.rs`, `apps/core/src/sidecar/manager.rs`).

The sampler holds one long-lived `sysinfo::System` for its whole lifetime and refreshes only the
known sidecar PIDs, never a full process scan. This matters for CPU: `sysinfo` reports CPU usage as a
delta since the previous refresh of that PID, so a one-shot refresh always reads `0`. Reusing the same
`System` across ticks is why the second sample onward yields real CPU numbers.

The numbers ride the existing status poll. `GET /api/sidecar/status` gains optional `pid`,
`memory_bytes`, and `cpu_percent` fields per sidecar (`apps/core/src/sidecar/mod.rs`), each omitted
when absent, so no new endpoint is needed.

| Field | Meaning |
|---|---|
| `pid` | OS process id, present only when Core owns a resident child for the sidecar |
| `memory_bytes` | Resident-set memory of that process, in bytes |
| `cpu_percent` | CPU usage as a percentage of one core |

The desktop surfaces these in the node selector's Engines block, grouped by modality
(`apps/desktop/src/components/shell/NodeSelector.tsx`). Engines that adopt an external server, run
serverless, or run in-process report no `pid` and render running-but-usage-omitted. Chat-engine rows
are status-only there: swapping the chat engine is a Store action, not a sidecar start/stop, so the
panel shows only the active chat engine, not the installed-but-idle alternatives.

<Callout type="warn">
`cpu_percent` uses `sysinfo`'s per-core convention, so a busy multi-threaded engine can read above
100. The fields appear only for sidecars Core owns a PID for, and only after a Core rebuild and
restart, since the sampler ships with the new binary.
</Callout>

## Image and video generation

The `sdcpp` media engine wraps stable-diffusion.cpp for text-to-image and text/image-to-video
(`apps/core/src/sidecar/providers/sdcpp/mod.rs`). It is a generative-media engine, so like the voice
engines it runs **alongside** the resident chat engine rather than instead of it. It is opt-in: not in
the startup order, installed and started from the Store.

| Property | Value |
|---|---|
| Sidecar name | `sdcpp` |
| Default port | `8083` |
| Model | A diffusion model resolved from `RYU_SD_MODEL`, or the bundled default |
| Lifecycle | Adopt an already-running `sd-server` on the port, otherwise spawn one from `~/.ryu/bin` |

The lifecycle mirrors whisper.cpp: if an `sd-server` is already answering on the port, Core adopts it
and leaves it running on stop; otherwise it spawns `sd-server` with the resolved model. Diffusion
weights take a while to load, so startup allows a generous timeout.

Two data paths consume the engine (`apps/core/src/server/media.rs`):

```
POST /api/images/generate
POST /api/video/generate
```

`POST /api/images/generate` proxies an OpenAI-style body to the engine's OpenAI-compatible
`/v1/images/generations` endpoint and returns the upstream JSON, with image bytes as base64 in
`data[].b64_json`. `POST /api/video/generate` proxies to the same engine's native `/sdcpp/v1/vid_gen`
endpoint. Both forward the request body as-is, with only a sensible default merged in when absent, so
the full sd-server parameter surface stays reachable without Core hardcoding a schema.

The desktop Store surfaces `sdcpp` under the Engines tab's Image group with install, start, and stop
(`apps/desktop/src/components/store/EnginesCatalogSection.tsx`).

<Callout type="warn">
The engine must be installed and running before these endpoints work. Image generation has been
exercised end-to-end producing a real PNG, but video generation expects user-supplied models that are
large and GPU-preferred, set via `RYU_SD_MODEL`.
</Callout>

## First-run install

On first launch Core installs a local stack so chat works immediately
(`apps/core/src/sidecar/onboarding.rs`). Each step is non-fatal: a failure warns and never blocks
chat.

1. The llama.cpp binary.
2. The bundled Gemma 4 chat GGUF (only if the binary installed).
3. The whisper.cpp voice engine and its `ggml-base.en` model (download only; voice is opt-in to run).

The nomic embedding model is owned by the `llamacpp-embed` sidecar, which auto-starts on first boot
and downloads its own model. This keeps it out of the first-run stack to avoid a concurrent
double-download.
