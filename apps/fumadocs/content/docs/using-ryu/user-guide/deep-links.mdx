---
title: "Deep Links (ryu://)"
description: "Install or switch a model or skill from any web link, and open the desktop app to a surface"
---

A `ryu://` link is a one-click handoff from any web page (or another Ryu surface) into the
desktop app. Click one and Ryu opens, then either jumps straight to a screen or asks you to
confirm an install or switch. The grammar is shared verbatim by desktop, web, and mobile from a
single source of truth (`packages/protocol/src/deep-link.ts`), so a link built on the website
parses identically in the app.

## The two kinds of link

The host segment (the word right after `ryu://`) decides whether a link just navigates or
performs an action with a side effect.

| Link | What it does | Confirm? |
|---|---|---|
| `ryu://open/<page>` | Opens a desktop surface (agents, settings, models, ...) | No |
| `ryu://chat/new?prompt=...` | Opens a new chat with the composer pre-filled | No |
| `ryu://chat/<id>` | Opens an existing conversation | No |
| `ryu://models/<source>/<id>` | Installs or switches the local model | Yes |
| `ryu://skills/<source>/<id>` | Installs a skill | Yes |
| `ryu://nodes/connect?url=...` | Connects the app to a Core node | Yes |

Navigation links have no side effect, so they open immediately. Action links install, switch, or
connect, so they always pass through a confirm dialog first.

<Callout type="warn">
Live click-from-browser handoff needs a packaged, OS-registered build with a display. In a
`dev:vite` session the scheme is not registered, so the parser and confirm dialog are exercised
by unit tests rather than a real browser click. The grammar, parser, builder, and confirm flow
all compile and pass tests; the end-to-end click is not yet live-verified.
</Callout>

## Install or switch a model

A model link points at a catalog entry. For a Hugging Face GGUF the shape is:

```
ryu://models/huggingface/unsloth/gemma-4-12B-it-qat-GGUF
```

Everything after `<source>` is the verbatim catalog id, joined by `/`, so an `author/repo` id
(including a trailing `-GGUF`) survives intact.

What happens when you click it:

1. Ryu opens and shows a confirm dialog
   (`apps/desktop/src/components/deeplink/DeepLinkConfirmDialog.tsx`).
2. If the model is **not installed**, Ryu picks the best-fitting quantization for your device
   (`pickRecommendedQuant` in `apps/desktop/src/lib/deep-link.ts`) and the dialog reads
   `Install & use`. Confirming downloads the GGUF through Core's verified path
   (`POST /api/models/catalog/install`) and then makes it the served model.
3. If the model **is installed**, the dialog reads `Switch` and confirming just switches the
   local engine to serve it via `POST /api/models/active`.

The "switch" half is the active-served-model override owned by Core. See
[Model Catalog](/docs/core/model-catalog) for the `local-chat-model` preference and the
`/api/models/active` route that backs it.

<Callout type="info">
The switch only affects the llama.cpp engine. The success toast is honest about this: it reads
"Now serving ..." only when llama.cpp actually reloaded, otherwise "selected (takes effect on
next engine start)" or "applies when llama.cpp is the active engine".
</Callout>

## Install a skill

A skill link works the same way, against the skills catalog:

```
ryu://skills/skills.sh/<owner>/<repo>/<slug>
```

Clicking it opens the confirm dialog. If the skill is already installed the dialog just says so;
otherwise confirming installs it via `POST /api/skills/catalog/install`. See
[Skills Catalog](/docs/core/skills-catalog) for what a skill is and where it lands on disk.

## Connect to a node

A node-connect link carries its payload in the query string rather than the path:

```
ryu://nodes/connect?url=<base-url>&token=<bearer>&name=<label>
```

`url` is required; `token` and `name` are optional. Confirming adds the node and makes it active.
The desktop generates these links for you from the node selector's Share button, so you rarely
hand-write one. See [Nodes and Presence](/docs/core/node-and-presence) for how nodes work.

<Callout type="warn">
Only connect to nodes you trust. The confirm dialog spells this out, because a node connection
points the app at a backend that runs your agents and sees your chats.
</Callout>

## Open a surface

A navigation link jumps straight to a screen without any confirm step:

```
ryu://open/agents
ryu://open/settings
ryu://open/models
```

The canonical page keys are the `DEEP_LINK_PAGES` list in `packages/protocol/src/deep-link.ts`
(chat, agents, models, skills, tools, spaces, workflows, monitors, marketplace, settings,
channels, engines, store, and more). Each surface maps a key to its own route; an unknown key is
ignored rather than erroring.

A chat link can pre-seed the composer:

```
ryu://chat/new?prompt=Summarize%20this%20page&agent=ryu&project=/repo/path
```

<Callout type="warn">
A `prompt` only PRE-FILLS the composer; it is never auto-sent. The prompt is attacker-controllable
(any page can fire a `ryu://` link), so the message waits for you to read and press send. There is
deliberately no `send=true` parameter.
</Callout>

## The confirm dialog is the security boundary

A `ryu://` link is untrusted input: a malicious page can fire one. The protocol module only
parses; it never installs, connects, or sends a message on its own
(`packages/protocol/src/deep-link.ts`). Two guarantees follow:

- **Nothing installs without your confirmation.** Every action link routes through the confirm
  dialog, and the actual download runs through Core's verified, checksum-pinned install path, not
  the link.
- **Source is advisory, not an instruction.** The `<source>` segment names a catalog
  (`huggingface`, `skills.sh`) for display, but installs stay pinned to your configured catalog
  source. A link cannot point Ryu at an arbitrary registry. See
  [Marketplace Federation](/docs/using-ryu/marketplace) for how the active source is chosen.
