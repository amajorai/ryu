title: "Data Folder & Storage"
description: "Move your data folder to another disk, back it up, export to a zip, and restore"

Everything Ryu keeps lives in one data folder - the DBs (conversations, spaces, memory),
generated media, downloaded models, and the `bin/` of engine binaries. By default it is `~/.ryu`.
This page covers moving that folder to another disk, backing it up, and restoring it.

All of the path logic lives in Core (`apps/core/src/paths.rs` resolves the folder,
`apps/core/src/data_path.rs` owns the copy/export/import operations). The desktop **Settings ->
System -> Storage** tab (`apps/desktop/src/components/settings/StorageSettings.tsx`) is the UI.

<TryInRyu page="settings" />

## Where the data folder is resolved from

Core resolves the active folder once at startup (cached for the process lifetime, so any change
needs a Core restart) in this order:

| Order | Source | Notes |
|---|---|---|
| 1 | Env `RYU_DIR` | Power users, headless, tests. Overrides everything. |
| 2 | Pointer file | Written by the Storage UI. Lives **outside** the data folder. |
| 3 | Default `~/.ryu` | Used when neither is set. |

The pointer file is `data-path.json` in the OS config dir (`%APPDATA%\ryu` on Windows,
`~/.config/ryu` on Linux, `~/Library/Application Support/ryu` on macOS). It lives outside the data
folder on purpose: the preferences DB is *inside* the data folder, so it cannot record its own
location.

## See the current location and free space

`GET /api/data-path` returns the active folder, the default, whether it is custom, the folder size,
and free space on its disk. The Storage tab renders this so you can see how much room you have
before moving.

## Move to another disk

Open **Settings -> System -> Storage**, pick a target folder, and choose one of two paths. They are
deliberately different operations:

| Choice | What happens | API / mechanism |
|---|---|---|
| **Copy data & restart** | Copies the whole folder to the new disk, then points Core at it | Offline `ryu-core data-path migrate` subcommand, orchestrated by the Tauri command `migrate_data_folder` |
| **Start fresh here** | Points Core at the new (empty) folder; the old data stays intact | `POST /api/data-path/switch` (point-only, no copy) |

Before either, the desktop validates the target with `POST /api/data-path/validate`: the folder must
be writable, empty, not nested inside the current data folder, and have enough free space for a copy.

A switch or copy takes effect on the next Core restart. The point-only `switch` returns
`{ "ok": true, "restart_required": true }`.

<Callout type="warn">
The **Copy data & restart** path runs offline because copying a live `.db` (with its `-wal`/`-shm`
files) while Core has it open corrupts it. The Tauri command stops the managed Core, **waits until
`/api/health` is actually down**, runs the `ryu-core data-path migrate` subcommand (forwarding
progress to the `data-folder-progress` event), then restarts. In dev, Core is owned by `turbo`, so
the desktop **refuses** the copy rather than risk a live DB. Verified by `cargo test path` and
desktop `tsc`; a live copy + restart needs a packaged app.
</Callout>

To go back to the default `~/.ryu`, use **Reset** (`POST /api/data-path/reset`, also point-only).

## Back up by exporting a zip

`POST /api/data-path/export` zips the current data folder to an output path you choose. Export is
read-only on the data folder, so it runs **online** with no restart. Use it for a portable backup
you can move to another machine.

## Restore from a zip

Restoring overwrites the live DB files, so it is the offline twin of export: the
`ryu-core data-path import` subcommand, orchestrated by the Tauri command `import_data_folder`. The
desktop runs the same stop-Core -> wait-for-health-down -> subcommand -> restart sequence the copy
migration uses. The zip importer is zip-slip guarded.

<Callout type="info">
Export and import are one zip. Export the folder on the old machine, then import that zip on the new
one to clone your full setup (conversations, spaces, memory, models, engine binaries).
</Callout>

## Cross-device sync

Sync is the live-mirror alternative to copying a zip, but it is not wired up yet.

<Callout type="warn">
The cross-device `SyncClient` is declared (`apps/core/src/server/sync.rs`) but **inert** - it is
never invoked, and node selection is manual-only with no reachability probe. For now, move data
between machines with the **export / import** zip above. See
[Ryu Cloud roadmap context](/docs/start-here/architecture/why-ryu) for where managed sync is headed.
</Callout>
