In Ryu, an agent is a **"Pokemon card"**: a bundle of independently swappable slots. Each agent
has its own chat model, its own tools, its own memory, and its own persona (system prompt). No
two cards are alike, and you can change any slot without rebuilding the agent. This is Ryu's
core idea made literal - you are locked to nothing.

## Browse agents

Open **Agents** from the sidebar to see every agent you have. Pi is always present as the
local, no-setup default. Selecting an agent in chat routes your messages through that card's
configuration.

## Create a new agent

1. On the Agents page, click **+** to create a new agent.
2. Give it a **name**.
3. Write a **system prompt** that defines its persona and behavior.
4. Pick the **model** it should use.

Save, and the new agent appears in the picker, ready to chat.

## Configure tools

Each agent can be granted a set of **tools** through MCP (the Model Context Protocol)
connections. Tools let an agent do things beyond chatting - browse the web, read files, or
call a service. Configure which tools an agent can reach from its settings; only allowlisted
tools are injected into the agent's sessions.

<Callout type="info">
  Tool calls made through Ryu Core are governed by the Gateway. Built-in tool servers such as
  Ghost (desktop automation) and Shadow (screen and audio capture) are available through the
  MCP registry.
</Callout>

## Configure memory

Attach a **Space** to an agent so it can ground its answers in your documents (see
[Spaces and Memory](/docs/using-ryu/user-guide/spaces-memory)). Separately, each agent keeps its own
encrypted long-term memory automatically.

## Agent builder studio

The agent edit page is a two-pane **studio**. The left pane is a builder chat that edits *this*
agent's configuration, and the right pane is the organized config panel (model, tools, skills,
identities, rules, instructions) that updates live as you make changes.

Tell the builder what you want in plain language, for example "make it formal" or "add the web
search tool", and it applies the change for you. Under the hood the builder chat is offered Core's
built-in `agent_builder` tools (`get_agent`, `configure_agent`, `create_agent`), wired into the
MCP registry in `apps/core/src/sidecar/mcp/mod.rs` and dispatched from
`apps/core/src/runnable/agent_builder.rs`. When you ask for a change, the model calls
`configure_agent` to patch the persisted agent record, and the right-hand panel re-hydrates once the
turn settles (`apps/desktop/src/components/agents/AgentBuilderChat.tsx`).

A few details worth knowing:

- The builder conversation runs as the flagship **Ryu** agent runtime (its tool loop is what
  applies the edits) with `persist: false`, so these builder turns never enter your chat history.
- For a brand-new agent, a **draft record** is created on your first builder message so the chat has
  something real to edit (`apps/desktop/src/pages/AgentEditPage.tsx`).
- `configure_agent` refuses to edit a `built_in` or `locked` record, so the flagship `ryu` agent and
  the builder meta-agent itself cannot be rewritten through the builder chat
  (`apps/core/src/runnable/agent_builder.rs`).

<Callout type="warn">
  The builder chat is a tool-loop round-trip, so it needs a running Core and a model that can drive
  tool calls.
</Callout>

## Per-agent evals

The edit page has an **Evals** view scoped to the agent you are editing
(`apps/desktop/src/components/agents/AgentEvalsView.tsx`). Running an eval scores cases on latency,
token efficiency, policy pass, and an optional substring match. With no custom dataset, the gateway
falls back to its built-in case set, so the panel is useful immediately.

Evals are a measurement concern, so they reuse the Gateway's eval and audit API rather than
re-implementing policy: the run posts to `POST /api/gateway/evals/run` via `runGatewayEvals`, and the
run history reuses `fetchGatewayAudit`. A **Prompt Studio** view on the same page lets you edit the
agent's system prompt.

<Callout type="warn">
  This is the agent-scoped eval surface. Full promptfoo-parity evals (custom assertions, judges,
  multi-model comparison, prompt versioning) are a multi-pass effort and not all of it is here yet.
</Callout>

## Council chat

Council chat lets several agents share one conversation. **@mention** an agent by name to
direct a turn to it, and bring multiple agents into the same thread to compare answers or have
them build on each other. It is the multi-agent counterpart to a normal one-on-one chat.
