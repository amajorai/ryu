Tools are how an agent does things beyond talking: read a page, run a command on your machine,
file a GitHub issue, search the web. In Ryu every tool comes from one of three sources, all
surfaced on the desktop **Tools** page (`apps/desktop/src/pages/ToolsPage.tsx`):

- **MCP servers** you register, plus the built-in ones (Ghost, Shadow, spider, exa, and reserved
  servers like `web_fetch` and `search_conversations`).
- The **unified tool catalog**, which makes every tool searchable by a query rather than dumped as
  one giant list.
- **Composio**, a hosted catalog of 800+ third-party actions (Gmail, GitHub, Slack, and more) that
  is searchable but never listed in full.

Whatever the source, an agent only sees the tools you put on its **allowlist**. This page is the
how-to for that. For how the registry and catalog work under the hood, see
[MCP registry](/docs/core/mcp-registry) and the
[unified tool catalog](/docs/core/unified-tool-catalog).

## Browse and allowlist tools

Open **Tools** in the desktop sidebar. The page lists every registered server and the tools each
exposes, backed by `GET /api/mcp/servers` and `GET /api/mcp/tools`.

To scope tools to one agent, use the **agent filter** at the top of the page. Picking an agent
re-fetches the tool list narrowed to that agent's allowlist server-side - Core resolves
`GET /api/mcp/tools?agent=<id>` against the agent's allowlist (`useMcp`,
`apps/desktop/src/hooks/useMcp.ts`). An empty allowlist means the agent sees every enabled tool; a
non-empty allowlist is the intersection with what is enabled.

You edit the allowlist itself in the **agent editor** (open an agent, then the Tools section), not
on the Tools page. The Tools page is for browsing, testing, and connecting servers.

The allowlist key is the **fully-qualified tool id** end to end - `<server>__<tool>`, for example
`spider__crawl` or `ghost__ghost_click`. The same id is what you grant, what you test, and what the
agent calls.

<Callout type="info">
Tool execution is governed on both planes. Whether a tool is invoked by an ACP agent or through the
OpenAI-compatible chat path, it is allowlist-gated and emits a Gateway exec-budget plus exec-audit
entry. See [unified tool catalog](/docs/core/unified-tool-catalog) for the governance path.
</Callout>

### Test a tool

On the Tools page, pick an agent and a tool, supply JSON arguments, and run it. The call goes to
`POST /api/mcp/tools/call` with the chosen agent as the invoker, so it exercises the same
allowlist and governance an agent would hit. A test that the agent could not make (the tool is not
on its allowlist) fails the same way for you.

## Connect an MCP server

To add your own MCP server, use **Add server** on the Tools page. It registers the server via
`POST /api/mcp/servers` with a command and arguments:

| Field | Meaning |
|---|---|
| `name` | The server name; becomes the `<server>__` prefix on its tool ids |
| `command` | The executable Core spawns to run the server |
| `args` | Arguments passed to the command |

Once registered the server and its tools appear immediately in the lists above, with no manual
refresh. Each server row shows an availability indicator (whether the command is present on disk)
so a missing binary is visible before an agent tries to call it.

For installing servers from a catalog (the official MCP registry, Smithery, or the Ryu-hosted
marketplace) rather than hand-registering a command, see the [Store](/docs/using-ryu/store) and
[MCP registry](/docs/core/mcp-registry). Catalog installs land as a **disabled** `~/.ryu/mcp.json`
entry that you explicitly enable - Core never auto-launches an installed registry command.

## Search the catalog

The unified tool catalog turns "which tool do I need" into a query instead of a scroll. It backs
two routes:

```
GET /api/tools/search?query=<text>
GET /api/tools/describe?id=<tool-id>
```

`search` returns ranked descriptors across MCP servers, built-ins, plugin tools, and Composio in
one `{ object, data }` envelope; `describe` returns the full schema for one tool. Both accept
`?agent=` or `?kind=` to narrow the results. Agents use this same search-then-describe-then-call
loop so they can reach a large tool surface without every tool being loaded up front. The
mechanics live in [unified tool catalog](/docs/core/unified-tool-catalog).

## Connect Composio

Composio gives agents real-world actions across 800+ apps. It is **searchable, not listed** - it is
never registered as an MCP server and never dumped into the full tool list. Instead a capped slice
of Composio actions is pulled live into `search()` when a key is configured, and an action is
called by its fully-qualified id `composio__<slug>` (for example `composio__GITHUB_CREATE_ISSUE`),
dispatched behind the same `call_tool` path (`apps/core/src/sidecar/mcp/composio.rs`).

To enable it:

1. Open **Settings → Integrations** and paste your Composio API key
   (`apps/desktop/src/components/settings/IntegrationsTab.tsx`). Saving the key reloads the gateway
   so Composio actions become available.
2. In the **agent editor**, add the Composio actions you want to that agent's allowlist. The
   gateway runs the actions; agents only pick which ones.

### The connection-required prompt

Composio actions run against a per-user **entity** (`COMPOSIO_DEFAULT_ENTITY` or a per-request
`user_id`). The first time an agent calls an action for an app you have not yet authorized, Composio
returns a connection-required response and Core surfaces it as a `__ryu_elicitation__` envelope - a
pause that asks you to log in to that app before the action proceeds. This is the same elicitation
pattern the [Identity Vault](/docs/core/identity-vault) uses, so a tool never silently fails on a
missing connection; it asks.

<Callout type="warn">
The exact Composio wire shape for the unconnected case is unverified against a live account
(`apps/core/src/sidecar/mcp/composio.rs`), and the connection-required detection is defensive across
response shapes. Composio actions also require a configured API key - with no key, Composio simply
does not appear in search.
</Callout>

## Built-in tool servers

Some tools ship with Core and need no registration. They appear in the lists like any server:

| Server | What it does | Notes |
|---|---|---|
| `ghost` | Desktop automation (click, type, screenshot, recipes) | Windows-first; needs the Ghost sidecar installed |
| `shadow` | Screen and audio capture, search | Windows-first; device-bound |
| `spider` | Web scraping | Default tool app |
| `exa` | Web search | Default tool app |
| `web_fetch` | Fetch a page, optionally as the logged-in user | Uses [Identity Vault](/docs/core/identity-vault) credentials when bound |
| `search_conversations` | Semantic search over past chats | See [search and recall](/docs/using-ryu/user-guide/search-and-recall) |
| `threads` | Spawn and manage worker conversations | Opt-in per agent; see [coordinator threads](/docs/core/coordinator-threads) |

<Callout type="warn">
Ghost and Shadow are Windows-first and are not surfaced in a dedicated desktop UI; they are
callable through the MCP registry. The `threads` tools are agent-facing and allowlist-opt-in -
only an agent granted `threads__*` can spawn workers.
</Callout>
