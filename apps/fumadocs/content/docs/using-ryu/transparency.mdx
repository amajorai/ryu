title: "Privacy and Telemetry"
description: "Exactly what Ryu collects, the default for each stream, what it never collects, and how to turn each one off."
---

Ryu is local-first and encrypted by default. This page is the full, honest accounting of every
telemetry and support stream: its default, what it does and does not collect, and how to disable it.
Nothing here is hidden, and every stream has an off switch.

The single rule that decides each default is the layer a stream lives in:

- The **data plane** is anything that can touch a prompt, a completion, agent content, a file path,
  a tool argument, memory, or a Space: the Rust **Core** and **Gateway**. Telemetry here is **off by
  default and opt-in**. This is what "no telemetry by default" means.
- The **closed UI plane** is the desktop app shell, the website, and the extension: which screen you
  opened, whether onboarding finished, whether an install succeeded. Anonymous, content-free product
  analytics here is **on by default and opt-out**, disclosed at first run.
- **Support access** is a deliberate, momentary, user-granted exception. It is **off by default** and
  auto-expires.

<Callout type="info">
  Your prompts, conversations, files, and any agent content are never part of product analytics or
  crash reports. The data plane never has a code path that ships that content off your device except
  the model call itself (governed by the Gateway) and a support bundle you explicitly grant.
</Callout>

## Closed UI plane (on by default, opt-out)

### Product analytics

- **Default:** on (opt-out), and disclosed once at first run.
- **What it collects:** a fixed, typed set of content-free events such as onboarding completed, an
  agent installed or removed, the active engine swapped, a model install started or completed, a chat
  turn started (count only), a feature enabled or disabled, and an error surfaced (by stable code). The
  desktop event schema is a discriminated union of scalars only, so there is no free-text field to
  misuse.
- **What it never collects:** prompt or completion text, conversation content, file paths, tool
  arguments, or any agent content. Identity is a random install ID that is not linked to your account
  (no `identify` call is ever made), and the EU Cloud host is used so IP capture is off project-side.
- **See what is sent:** the desktop ships a local egress audit log (a ring buffer of the exact typed
  events that were sent) and an in-app inspector in **Settings, Privacy**.
- **How to disable:**
  - Desktop and Core: toggle off in **Settings, Privacy**, which writes the `product-analytics-enabled`
    preference (default `true`). Env mirror `RYU_PRODUCT_ANALYTICS_ENABLED=false`.
  - Web: the site reads a self-contained local gate (`localStorage` key `ryu:product-analytics-enabled`
    set to `false`), because the website has no Core connection. With no PostHog key configured the web
    provider is a graceful no-op and collects nothing.

### Crash reports

- **Default:** on (opt-out), as a separate consent tier from product analytics.
- **What it collects:** scrubbed crash and error stacks (desktop renderer and Rust panics) to fix what
  breaks.
- **What it never collects:** prompt or agent content. The install ID is the same random,
  account-unlinked identifier used by analytics.
- **How to disable:** toggle off in **Settings, Privacy**, which writes the `crash-reports-enabled`
  preference (default `true`). Env mirror `RYU_CRASH_REPORTS_ENABLED=false`. This is independent of the
  product-analytics toggle, so you can keep one and drop the other.

## Data plane (off by default, opt-in)

### Local run traces and Gateway audit

- **Default:** on, but **local-only** and never exported on their own. Run traces live in
  `~/.ryu/traces.db` and the Gateway audit lives in its own SQLite store.
- **What it records:** per-run spans (tool and model calls) where tool arguments are stored as a
  SHA-256 hash, never the raw arguments, plus the Gateway audit (tokens, cost, latency, provider,
  model, errors) with API keys redacted to a short prefix.
- **How to disable:** these are local stores you own; there is no off switch because nothing leaves
  your device. They only egress if you turn on diagnostics export below.

### Diagnostics export (OTLP)

- **Default:** **off (opt-in)**.
- **What it does:** when, and only when, you opt in **and** configure a destination, Ryu attaches an
  OpenTelemetry (OTLP) exporter that drains the same local trace and audit spans to a backend you
  choose. The destination is fully swappable: Axiom, Grafana, a self-hosted Collector, or PostHog LLM
  analytics. Nothing is hardcoded to a vendor.
- **What it never collects:** the exported records carry the same redacted, content-free shape as the
  local stores (hashed args, redacted keys). The experimental GenAI semantic conventions (`gen_ai.*`
  attributes: model, provider, input and output tokens, latency) are emitted only when you separately
  opt into them; the conventions define no cost field, so cost is derived downstream from tokens.
- **How to enable or disable:**
  - Set the destination in **Settings, Privacy**, which writes `diagnostics-otlp-endpoint`
    (empty by default) and flips `diagnostics-export-enabled` (default `false`). Env mirrors
    `OTEL_EXPORTER_OTLP_ENDPOINT` and `RYU_DIAGNOSTICS_EXPORT_ENABLED`.
  - With the toggle off or no endpoint set, **nothing egresses**: the exporter layer is simply not
    installed. Your one opt-in also drives the Gateway sidecar, so both Core and Gateway spans drain to
    the same destination only when you consent.
  - Destinations that need auth use the standard `OTEL_EXPORTER_OTLP_HEADERS` env. For PostHog there is
    a convenience env (`RYU_POSTHOG_KEY`) that derives the bearer header for you.
  - Enable the experimental LLM spans with `OTEL_SEMCONV_STABILITY_OPT_IN` containing `gen_ai`.

## Control-plane server logs

- **Default:** on, on Ryu's own machines (so there is no consent question for your device).
- **What it collects:** one canonical wide event per request hop (method, route, status, duration,
  trace and span IDs) shipped to Axiom via OTLP. No user content. With no OTLP endpoint configured the
  logging is a no-op.
- **How to disable:** this runs server-side on Ryu Cloud infrastructure, not on your machine, so there
  is no client toggle. Self-hosters control it with the server's `OTEL_EXPORTER_OTLP_ENDPOINT` and
  `AXIOM_TOKEN` env.

## Support access (off by default, user-granted)

Support access is a deliberate exception, not passive telemetry. It is off by default, you initiate it,
it is scoped and auto-expiring, and the actor is always recorded (delegation, not silent impersonation).

### Control-plane support access

- **Default:** off. You grant it from your account settings.
- **What it covers:** control-plane data only (billing, marketplace licenses, sync mirrors, channel
  configs, org records). It is built on Better Auth impersonation: the session records `impersonatedBy`
  (the support actor stays visible), a required reason, a short auto-expiry (at most one hour), and an
  always-visible banner with one-click revoke.
- **Audit:** every support session is recorded in an append-only log you can read (who, when, why, what
  scope), and that log is never deleted when you revoke or it expires.
- **How to disable:** it is off unless you grant it; revoke any time from your account settings or the
  active-session banner.

### Local Core support access

- **Default:** off. You grant it from the desktop, and it is enforced by your local Core.
- **What it covers:** a narrow, read-only, allowlist diagnostic bundle: Core version, the active
  engine, sidecar names and running state, the **names** of which preferences are set (never their
  values), and redacted trace spans (hashed args). It never includes prompt or agent content and never
  touches credentials.
- **Transport and expiry:** the bundle egresses only over the opt-in mesh (Tailscale) and through the
  Gateway firewall, has a hard auto-expiry that is re-checked and disabled at Core startup, and writes
  to a local, append-only audit log with the actor stamped on every entry (the record is yours).
- **How to disable:** it is off unless you grant it. It writes the `support-access-local-enabled`
  preference (default `false`) with a `support-access-local-expiry` instant. With access off the
  diagnostic endpoint refuses outright.

## At a glance

| Stream | Layer | Default | Content | How to turn off |
| --- | --- | --- | --- | --- |
| Run traces | Core | On, local-only | Hashed args, no raw | Local only (never exported alone) |
| Gateway audit | Gateway | On, local-only | Tokens and meta, keys redacted | Local only (never exported alone) |
| Diagnostics export (OTLP) | Core and Gateway | **Off (opt-in)** | Same redacted records | `diagnostics-export-enabled` / `OTEL_EXPORTER_OTLP_ENDPOINT` |
| Product analytics | Desktop, Web, Extension | On (opt-out) | Content-free event enum | `product-analytics-enabled` toggle in Settings, Privacy |
| Crash reports | Desktop | On (opt-out, separate) | Scrubbed stacks | `crash-reports-enabled` toggle |
| Server logs | Control plane | On (Ryu's machines) | Wide events, no user content | Server-side only |
| Support access (cloud) | Control plane | **Off (user-granted)** | Scoped, actor recorded | Account settings, revoke any time |
| Support access (local) | Core | **Off (user-granted)** | Redacted diagnostics, secret-stripped | `support-access-local-enabled`, off by default |

<Callout type="info">
  Two separate consents (analytics vs crashes, cloud support vs local support) can each be toggled
  independently, and each has a local "what did we send" view. The full design rationale lives in the
  repo at `docs/observability-analytics-support-access.md`.
</Callout>
