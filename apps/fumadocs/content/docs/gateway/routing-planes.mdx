---
title: Routing planes
description: Model routing (Plane A, Gateway) vs agent routing (Plane B, Core), and the one rule for which plane governs chat egress.
---

Ryu has two distinct things both historically called "routing", and nothing in the code named
which is which. This page names them. **Plane A (model routing)** lives in the Gateway and picks
*which LLM* an OpenAI-compat completion uses. **Plane B (agent routing)** lives in Core and picks
*which agent* serves a chat turn, where the agent brings its own model and (for ACP subprocess
agents) makes its own provider calls. They are not the same mechanism, they do not run at the same
layer, and only one of them governs chat egress.

<Callout type="info">
  Core decides what runs. The Gateway decides what is allowed. Plane B (Core) chooses the agent;
  Plane A (Gateway) governs the LLM call - see
  [Core vs Gateway](/docs/start-here/architecture/core-vs-gateway).
</Callout>

## The two planes

| Plane | Owns | Surface | Model lifetime | Governed how |
|---|---|---|---|---|
| Plane A - Model routing | Gateway | Picks the LLM for one OpenAI-compat `/v1/chat/completions` request | Model is fixed for the whole turn (and its tool loop) once routed | Full Gateway plane: allowlist + inbound firewall + budget + audit |
| Plane B - Agent routing | Core (ACP / catalog) | Picks the agent that serves the turn; the agent brings its own model | Model is owned by the agent/subprocess, not by a single Gateway route | Only tool egress is governed (MCP bridge); ACP chat egress bypasses the Gateway unless the agent honours `OPENAI_BASE_URL` |

The crisp distinction: Plane A rewrites `body["model"]` inside one completion request. Plane B
chooses a backend (`AgentRoute`) and, for ACP agents, hands the whole turn to a subprocess that Ryu
cannot intercept at the chat-LLM level.

## Plane A: model routing

Plane A runs entirely inside the Gateway pipeline on the OpenAI-compat chat path. It is a two-stage
decision.

**Stage 1 - optional smart routing (classifier).** On the chat path, `apply_smart_routing` runs
before `pre_process` (`apps/gateway/src/pipeline/mod.rs`). It only does anything when
`SmartRouter::is_active()` is true (`apps/gateway/src/router/smart.rs`) and no per-agent chat slot is
pinned. `SmartRouter::resolve` caches the decision per session (keyed by `x-ryu-session-id`) and
dispatches on the configured strategy - a cheap classifier model, an embedding nearest-match, or a
keyword match. The classifier strategy resolves its model through the normal `ModelRouter::route` and
then issues a plain `provider.complete` call.

The load-bearing point - **model routing is NOT tool calling.** The classifier is a direct
`provider.complete`, deliberately *not* a tool call, so it bypasses the pipeline and cannot recurse
into routing. The result is a single rule index that rewrites `body["model"]` in place. It fails open
at every failure mode: inactive config, timeout, provider error, or an unparseable reply all keep the
requested model. Config is opt-in: `SmartRoutingConfig::enabled` defaults to false. The full
strategy vocabulary (llm / embedding / keyword) and per-agent overrides are documented in
[Routing → Smart routing](/docs/gateway/routing#smart-classifier-driven-routing).

**Stage 2 - deterministic model-to-provider routing.** Once the model string is fixed,
`ModelRouter::route` (`apps/gateway/src/router/mod.rs`) maps it to a provider with no hardcoded lock:
exact `model_map` hit, then longest-prefix user map, then a built-in prefix table (`claude-` to
Anthropic, `gpt-`/`o1`/`o3`/`o4` to OpenAI, `openrouter/` to OpenRouter, `gemini-` to GenAi,
`llama`/`gemma`/etc to Local, `zeroclaw`/`openclaw` to Core, `modal/` to Modal), then the configured
`default_provider`.

The model is fixed for the whole turn. Tool calls in the loop re-enter the completion API with the
already-routed model - they never re-trigger stage 1 or stage 2. The only thing that changes the
model *after* routing is budget: the lifetime token-budget block and `enforce_budget`'s Downgrade
action both rewrite `body["model"]` to a cheaper model and re-route.

## Plane B: agent routing

Plane B is the "bring-your-own-model agent" path and lives in Core. A chat turn resolves its backend
in `agent_route` (`apps/core/src/sidecar/adapters/mod.rs`), which then matches the resolved
`AgentRoute`. That match is the two-backend split:

- **Catalog / OpenAI-compat registry agents** become `AgentRoute::OpenAiCompat { via_gateway: true }`
  and stream through the Gateway. These agents feed Plane A - their chat is governed.
- **ACP subprocess agents** become `AgentRoute::Acp { spawn_cmd }` and stream via `route_acp_stream`.
  An ACP agent runs its own subprocess and makes its own provider calls, so Core cannot intercept its
  chat-LLM egress in general.

**The BYO OpenAI-compat lever.** For an ACP agent whose HTTP client honours the OpenAI base URL,
there is one interception point: `acp::openai_gateway_cmd` (`apps/core/src/sidecar/adapters/acp.rs`)
prefixes the spawn command with `OPENAI_BASE_URL` (the local gateway `/v1`) and `OPENAI_API_KEY` (the
gateway token), so the agent's chat calls flow through the Gateway. It is gated by the per-agent
toggle `agent-gateway-routing` (`apps/core/src/agent_routing/mod.rs`), which defaults OFF. The full
mechanism - base-URL injection, the subscription-preserving passthrough proxy for Claude Code and
Codex, and the bypass list - is documented in
[Gateway for any agent](/docs/gateway/gateway-for-any-agent).

**The genuine bypass.** The injection is a real no-op for Gemini CLI (Google wire format), OpenClaw
(its own WebSocket gateway), and Hermes (its own creds): they ignore `OPENAI_BASE_URL` and bypass the
Gateway for chat. Claude Code and Codex are guarded off this generic path - they have dedicated
[passthrough routing](/docs/gateway/gateway-for-any-agent#subscription-preserving-passthrough-claude-code-codex)
that preserves their subscription billing.

**Tool egress is always governed, regardless of chat.** The MCP bridge `build_ryu_mcp_server`
(`apps/core/src/sidecar/adapters/mcp_bridge.rs`) is wired into the live ACP session, and every tool
call dispatches through the bridge into `McpRegistry` allowlist enforcement - no direct-egress
bypass. The same governance is exposed over HTTP at `POST /api/mcp/tools/call`, which requires a known
agent id so the allowlist cannot fail open.

<Callout type="info">
  Plane B *backend resolution* (the `OpenAiCompat` / `Acp` split above) is shipped. The **agent-auto
  picker** - a per-turn choice of *which agent* by your own rules - is a separate, planned Plane B
  feature; see [Routing → Agent-auto](/docs/gateway/routing#agent-auto-plane-b-picking-the-agent).
</Callout>

## The ownership rule

One rule decides which plane governs a turn - **only one plane governs chat egress:**

- **OpenAI-compat requests get Plane A model routing.** The Gateway picks and fixes the LLM, and the
  full policy plane (allowlist, firewall, budget, audit) applies to that chat.
- **ACP agents self-route (Plane B).** The agent owns its model. Only its *tool egress* is governed
  (via the MCP bridge); its chat egress reaches the Gateway *only* if it honours `OPENAI_BASE_URL`
  and the per-agent toggle is ON.

Both planes feed **one** governance layer: the Ryu Gateway, the single place that decides what is
allowed, shared, measured, and paid for. In the pipeline, `pre_process` runs rate-limit then inbound
firewall then routing on every request, `enforce_budget` applies budget decisions, and every model
call is audited. The unified tool surface is deny-by-default and activates only on an explicit
per-request signal. This is where the retired `claw-patrol` tool's job now lives: patrolling and
guarding tool calls is native to the Gateway firewall and the cmdscan `/v1/exec/scan` path, not a
catalog dependency.

## How this compares

- **Zed** pushes routing *out* to a protocol boundary. ACP is bring-your-own-model: the editor talks
  to whatever agent subprocess you point it at, and that agent owns its model calls. This is Ryu's
  Plane B, and it is exactly why Plane B chat egress can bypass a central gateway.
- **Databricks Omnigent** pulls routing *up* to an agent-level meta-harness: a rule-based or
  lightweight routing model selects the model/sub-agent, with governance centralized. This is Ryu's
  Plane A smart routing plus the shared Gateway policy plane.
- **Ryu does both** and this page names them. Plane A is the Omnigent-style up-routed,
  centrally-governed model plane; Plane B is the Zed-style out-routed, bring-your-own-model agent
  plane; and both are made to feed the same Gateway policy layer.

## Where things live

| Concern | File |
|---|---|
| Smart routing classifier (Plane A stage 1) | `apps/gateway/src/router/smart.rs` |
| Model-to-provider resolution (Plane A stage 2) | `apps/gateway/src/router/mod.rs` |
| Smart routing config (opt-in, fail-open) | `apps/gateway/src/config.rs` |
| Chat pipeline: smart routing, route decision, budget, audit | `apps/gateway/src/pipeline/mod.rs` |
| Unified tool surface (deny-by-default, `tool_search`) | `apps/gateway/src/tools/mod.rs` |
| Backend resolution + dispatch (Plane B) | `apps/core/src/sidecar/adapters/mod.rs` |
| ACP spawn, session pool, `OPENAI_BASE_URL` injection, gateway-bypass registry | `apps/core/src/sidecar/adapters/acp.rs` |
| Governed tool egress (MCP bridge) | `apps/core/src/sidecar/adapters/mcp_bridge.rs` |
| Per-agent gateway-routing toggle | `apps/core/src/agent_routing/mod.rs` |

## Gotchas

- **Smart routing is off by default.** `SmartRoutingConfig::enabled` is false; `is_active` also needs
  at least one rule (and, for the `llm` strategy, a classifier model). Without those, Plane A stage 1
  is inert and the requested model is kept.
- **Everything in Plane A fails open.** Inactive config, classifier timeout, provider error, or an
  unparseable reply all keep the originally requested model. A routing misconfig can never break chat.
- **Explicit slot pins beat the classifier.** A per-agent chat-model slot override wins over smart
  routing in both `apply_smart_routing` and the `pre_process` route decision.
- **Budget can override the routed model after routing.** The lifetime token-budget block and
  `enforce_budget`'s Downgrade action both rewrite `body["model"]` and re-route *after* the normal
  route decision, so the final model may differ from the classifier's pick.
- **ACP chat egress bypasses the Gateway for non-OpenAI-compat agents.** Gemini CLI, OpenClaw, and
  Hermes ignore `OPENAI_BASE_URL`, so their chat LLM calls are never governed even with the per-agent
  toggle ON. Their *tool* egress is still governed through the MCP bridge.
