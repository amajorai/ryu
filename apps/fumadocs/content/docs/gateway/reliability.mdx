The Gateway keeps model calls fast, bounded, and resilient. Five mechanisms sit on the request path: two caches in front of the provider, a per-key rate limiter, a concurrency admission gate for the local engine, a per-provider circuit breaker that drives the routing fallback chain, and budget enforcement across user, agent, and session scopes.

## Caching

Two caches sit in front of the provider.

| Cache | Location | Keying | Default |
|---|---|---|---|
| Exact | `apps/gateway/src/cache/` | Hash of model plus messages, with a TTL and max-entries cap | On |
| Semantic | `apps/gateway/src/semantic_cache/` | Cosine similarity of the embedded request against prior requests, above a threshold | Off |

The exact cache evicts expired entries with a background task. The semantic cache embeds each request with a configured embedding model and returns a stored response when similarity clears the threshold. A cached response carries `x-cache: HIT` (`MISS` otherwise), set in `apps/gateway/src/api/chat.rs`.

<Callout type="warn">
  The semantic cache is off by default and needs a configured embedding model. It returns a prior
  response when a new request is close enough, so enable it only where near-duplicate answers are
  acceptable.
</Callout>

### Provider prompt caching

The two caches above are the Gateway's own. Separately, upstream providers (OpenRouter, OpenAI,
Anthropic) run their **own** prompt cache - a repeated prefix (system prompt, tool definitions, long
context) is served cheaper and faster on the second hit. This is a cost and latency win, not a
correctness mechanism, and the Gateway does not manage it; it passes the breakpoints through and
meters the result.

For explicit-cache providers the request must carry `cache_control` breakpoints. Managed Pi only
emits them when the model's `compat.cacheControlFormat` is `anthropic`, so Core stamps that flag for
model ids containing `claude`, `anthropic`, or `qwen` - the OpenRouter explicit-cache families
(`apply_cache_compat` / `explicit_cache_control_format`, `apps/core/src/pi_config/mod.rs`). OpenAI,
Gemini 2.5, DeepSeek, and Grok auto-cache with no marker, so they get no flag; the local llama.cpp
default stays bare (no upstream cache to hit). The Gateway's firewall leaves array-form content and
`cache_control` keys untouched, so those breakpoints survive to the provider.

On the way back, the Gateway meters cached input tokens: `pipeline/mod.rs` reads
`usage.prompt_tokens_details.cached_tokens` (non-streaming) or parses the streamed usage, and feeds
`provider_cached_input_tokens` (`add_cached_tokens`, `apps/gateway/src/metrics/mod.rs`), surfaced as
`tokens.cached_input` on the metrics snapshot.

<Callout type="info">
  Provider prompt-cache billing is already correct without extra work - OpenRouter's `usage.cost`
  nets the cache discount and the credit path bills from that authoritative figure. The
  `provider_cached_input_tokens` counter is for observability, not a second pricing input.
</Callout>

## Rate limiting

The rate limiter (`apps/gateway/src/rate_limit/`) applies token-bucket limits per key across three dimensions: requests per minute, tokens per minute, and a per-second burst cap for bot detection. Per-key overrides in the auth config let one key have a higher ceiling than the global default. Idle buckets are evicted to bound memory.

## Concurrency admission control

The concurrency gate (`apps/gateway/src/concurrency/mod.rs`) protects the resident local engine from overload. A local llama.cpp or Ollama server has a fixed number of batch slots (`--parallel N`). Without a gate, a burst of background fan-out work submitted before an interactive turn makes the user wait behind the batch jobs in the engine's internal FIFO.

The admission gate fixes this with a priority-aware async semaphore keyed to the `local` provider:

- At most `local_max_in_flight` requests run concurrently. Size this to match the engine's `--parallel` slot count so every engine slot stays busy and the engine's own queue stays empty.
- When all slots are occupied, new requests queue — up to `local_max_queued`. Interactive requests jump ahead of background ones in the queue regardless of arrival order.
- When the queue is also full, new requests are rejected immediately with `503 engine_overloaded` rather than piling up unbounded.

Priority is read from the `x-ryu-priority` request header Core stamps. Any value of `background`, `low`, or `batch` is treated as Background; anything else (including no header) is treated as Interactive, so unlabelled requests from old clients or direct callers are never penalised.

Remote providers (OpenAI, Anthropic, OpenRouter, etc.) are never gated — they scale elastically upstream.

`ConcurrencyLimiter` is a snapshot of `config.concurrency` at startup; changes take effect on the next restart. `GateSnapshot` observability data (in-flight count, queue depth per priority) is exposed on the status surface.

| Setting | `gateway.toml` | Default |
|---|---|---|
| Enabled | `[concurrency] enabled` | `true` |
| In-flight slots | `[concurrency] local_max_in_flight` | `4` |
| Queue depth | `[concurrency] local_max_queued` | `64` |

Set `local_max_in_flight = 0` to disable gating entirely (treated as unlimited).

## Circuit breaker and fallback

The circuit breaker (`apps/gateway/src/circuit_breaker/`) protects against a failing provider. After a threshold of consecutive failures it opens and rejects calls to that provider immediately. After a reset timeout it half-opens to probe with a single request, then closes on success or re-opens on failure.

An open circuit is what drives the fallback chain. In `pipeline/mod.rs` the request walks `state.router.fallback_chain(&decision.provider)`, skipping any provider whose circuit is open. When a later provider serves a request the primary was skipped for, the response carries an `x-degraded: fallback:<provider>` header (`DegradedMode`, `apps/gateway/src/pipeline/mod.rs`) so clients and operators can observe the event. Provider health is reported on `/metrics`. The chain itself is defined in [routing](/docs/gateway/routing).

## Budgets

Budgets (`apps/gateway/src/budget/`) cap token usage and apply a graduated action when a cap is crossed. `BudgetScope` has three identity dimensions - `User`, `Agent`, and `Session` (`budget/mod.rs`). When more than one scope fires, the most severe action wins (`severity` ordering in `budget/mod.rs`).

The four actions, in increasing severity (`BudgetAction`, `apps/gateway/src/config.rs`):

| Action | Effect |
|---|---|
| `notify` | Allow the call, flag it in the `x-budget-*` response headers. Never blocks. The default |
| `downgrade` | Swap the requested model for the rule's `downgrade_to` (a cheaper model). Falls back to `restrict` if no downgrade model is set |
| `restrict` | Allow the call but clamp it - strip tool definitions and cap `max_tokens` |
| `stop` | Reject with `402 budget_exceeded` |

Per-user and per-agent rules are configured as maps under `[budgets.users]` and `[budgets.agents]` in `gateway.toml` and are hot-swappable. Every decision is surfaced in the response headers (scope, action, usage, limit).

<Callout type="info">
  All budget scopes are off unless configured. The counters are in-memory `DashMap`s that reset on a
  Gateway restart, and the master key (`auth.master_key`) bypasses every budget gate. Durable,
  cross-restart accounting is the audit log's job, covered in
  [evals and audit](/docs/gateway/evals-audit).
</Callout>

### Identity headers and spoofing

Per-user and per-agent budgets key on the `x-ryu-user-id` and `x-ryu-agent-id` headers Core forwards on the request. Those headers are only honored when the API key is marked `trusted_forwarder` (`ApiKeyConfig` in `apps/gateway/src/config.rs`). For an untrusted key the budget binds to the key's own name instead, so a caller cannot rotate identity headers to escape its quota (`authenticate` in `apps/gateway/src/pipeline/mod.rs`).

### Per-session budget

A single global per-session token cap applies to every session, with the running counter keyed by the `x-ryu-session-id` header. Unlike the `users` and `agents` maps, the session scope is not a per-id rule map: Core mints a fresh conversation id per chat, so a per-id map would be unpopulatable. One rule applies to all sessions.

The session scope reuses the same machinery as the user and agent scopes. `evaluate_session` builds a transient `BudgetRule` from the `SessionBudgetConfig` and runs the shared `decide` path, so `notify` / `downgrade` / `restrict` / `stop` all behave identically (`budget/mod.rs`). Configuration:

| Setting | `gateway.toml` | Env var | Default |
|---|---|---|---|
| Token cap | `[budgets.session] limit` | `GATEWAY_SESSION_BUDGET_LIMIT` | `0` (off) |
| Action | `[budgets.session] action` | `GATEWAY_SESSION_BUDGET_ACTION` | `notify` |

A `limit` of `0` turns the session scope off. Session ids are ephemeral, so the counter map (`session_usage`) is soft-capped at `MAX_SESSION_ENTRIES` and cleared wholesale on overflow (`record_session`, `budget/mod.rs`).

### Per-key lifetime budget

Each statically configured API key can carry a lifetime token cap, `token_budget_total` (input plus output combined), with an optional `downgrade_to` model (`ApiKeyConfig`, `apps/gateway/src/config.rs`). Enforcement runs before the provider call in `pipeline/mod.rs`: it reads the key's running total from the audit log (`state.audit.token_usage`) and, once the cap is crossed, either downgrades to the configured model or rejects with `402 budget_exceeded`. Because the usage figure comes from the audit log, this cap depends on audit logging being enabled.

### Exec budget for tools and sandboxes

A separate exec budget bounds non-model executions - sandbox runs and MCP tool invocations - counting executions and wall-clock seconds over a rolling window. It is driven by the `POST /v1/exec/budget/check` pre-run gate and `POST /v1/exec/audit`, both fail-closed and `trusted_forwarder`-only. The exec budget is documented with the tool path on the [tools](/docs/gateway/tools) page.
