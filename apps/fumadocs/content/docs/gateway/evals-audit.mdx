Measurement is a Gateway concern. The Gateway scores a sampled fraction of responses to feed
eval-driven [routing](/docs/gateway/routing), and records every model call (plus sandbox and
identity events) in a local audit log you can query. Both run on the request path; neither blocks it.

## Evals

The eval runner (`apps/gateway/src/evals/mod.rs`, `EvalsRunner`) does two jobs: it passively scores
sampled live responses and folds those scores into a rolling per-provider average, and it replays a
dataset on demand for explicit eval runs. Passive scoring is **on by default** (`EvalsConfig.enabled`
defaults to `true` and `sample_rate` defaults to `1.0`).

### Dimensions

Each scored response combines three normalized dimensions into one weighted `overall` value.

| Dimension | Range | How it is computed |
|---|---|---|
| `latency_score` | 0 to 1 | `1 - (latency_ms / max_latency_ms)`, clamped. 1.0 is instant, 0.0 is at or beyond `max_latency_ms` |
| `token_efficiency` | 0 to 1 | `output_tokens / input_tokens`, normalized so 2x output caps at 1.0 |
| `policy_pass` | bool | Whether the request cleared all firewall and policy checks |

The weighted score is `0.4 * latency + 0.3 * token_efficiency + 0.3 * policy`. A sampled response
carries its `overall` value back in the `x-eval-score` response header.

### Sampling

Sampling is deterministic, with no RNG. `EvalsRunner::should_sample` increments a counter and keeps a
request every `1/sample_rate` boundary, so a `sample_rate` of `0.25` scores exactly one request in
four. A rate of `1.0` scores every request (the default), `0.0` scores none. This keeps scoring overhead
predictable and even rather than bursty.

### Rolling scores

Each sampled score is folded into a per-provider exponential moving average (`EMA_ALPHA = 0.2`),
stored lock-free so the routing path reads it without a mutex. `eval_routing` consumes these averages
to bias provider choice toward the current leader while reserving an `explore_ratio` of traffic for
non-leaders, so scores stay fresh. See [Routing](/docs/gateway/routing) for the full A/B selection
logic.

```
GET /v1/evals
```

returns the live config plus the current rolling score per provider:

```json
{
  "enabled": true,
  "sample_rate": 1.0,
  "max_latency_ms": 10000,
  "providers": { "openai": 0.84, "anthropic": 0.79 }
}
```

When evals are disabled the endpoint still returns `200` with `enabled: false` and an empty
`providers` map. It never panics and never returns a non-200 for a healthy gateway.

### Configuration

`EvalsConfig` lives under `[evals]` in `gateway.toml`. All fields have defaults so no configuration
is required out of the box.

| Field | Default | Description |
|---|---|---|
| `enabled` | `true` | Master switch. Set to `false` to disable all passive scoring |
| `sample_rate` | `1.0` | Fraction of completed requests to score, in `[0.0, 1.0]` |
| `max_latency_ms` | `10000` | Latency (ms) at which `latency_score` drops to 0 |
| `stream_usage` | `true` | Inject `stream_options.include_usage=true` into streamed requests so the provider emits a terminal usage frame. Only OpenAI-compatible providers honour this flag; non-conforming providers fall back to `estimate_prompt_tokens` |

### Dataset runs

```
POST /v1/evals/run
```

replays each prompt in `dataset` through the full pipeline (firewall, routing, provider call) and
returns per-case scores plus an aggregate. An explicit run always scores, independent of whether
passive evals are enabled, and cases run sequentially to keep provider load and budgets observable.

| Field | Purpose |
|---|---|
| `model` | Model under test, forwarded to the router as-is. Defaults to `gpt-4o-mini` |
| `models` | Run the whole dataset against each model and return a per-model breakdown |
| `dataset` | Array of `EvalCase` (`prompt`, optional `expected`, `vars`, `assertions`). Empty uses a built-in 3-case dataset |
| `system_prompt` | Run-level system prompt prepended to every case, with `{{vars}}` substituted per case |
| `judge_model` | One fixed model that judges every `llm_judge` assertion (fair cross-model compare) |
| `evaluators` | Optional run-level registry evaluator ids (from `GET /v1/evaluators`) scored against every case, unioned with any per-case `evaluators` |
| `agent_id` | Optional id forwarded for per-agent budget tracking |

Each case also supports `assertions` evaluated against the response text - `contains`, `not_contains`,
`equals`, `regex`, `json_valid` (all deterministic), and `llm_judge` (a second pipeline call whose
verdict is parsed defensively, with a missing verdict failing safe). Unlike `GET /v1/audit`, this
endpoint authenticates with any valid API key, not the master key, because a dataset run is a normal
metered call.

From Core, `POST /api/gateway/evals/run` proxies the dataset run (including optional
`evaluators` and Core-local `code_evaluators`). `GET /api/gateway/evaluators` proxies the
evaluator catalog.

<Callout type="warn">
  Dataset runs are sequential and synchronous. A multi-model run with `llm_judge` fans out
  `models * cases * (1 + judge_calls)` provider calls, all under Core's 120s proxy timeout, so a large
  matrix can time out at the proxy even though no field is dropped. Keep matrices small.
</Callout>

## Evaluator catalog

The eval runner above scores responses on three fixed dimensions. Separately, the Gateway ships a **unified evaluator catalog** - one shared taxonomy meant to power both inline guardrails and offline eval scoring from a single definition. `GET /v1/evaluators` returns the full catalog.

```
GET /v1/evaluators
```

It is a read-only computation over static seed data (`EvaluatorRegistry`, seeded by `apps/gateway/src/evaluators/catalog.rs`), so it mutates no state and exposes no secret - it is ungated like `GET /v1/evals` and `POST /v1/firewall/check`.

### The catalog

30 built-in evaluators across 8 categories (`builtin_catalog`).

| Category | Count | Examples |
|---|---|---|
| Custom (create from scratch) | 2 | `llm_as_a_judge`, `code_evaluator` (blank rubric / code templates) |
| Security | 3 | `pii_leakage`, `code_injection`, `prompt_injection` |
| Safety | 2 | `toxicity`, `bias_fairness` |
| Quality | 7 | `hallucination`, `correctness`, `answer_relevance`, `exact_match` |
| Conversation | 6 | `tone`, `task_completion`, `user_satisfaction` |
| Trajectory | 3 | `plan_adherence`, `trajectory_accuracy`, `tool_selection` |
| Image | 2 | `explicit_content`, `sensitive_imagery` |
| Voice | 5 | `audio_quality`, `transcription_accuracy`, `vocal_affect` |

Each entry carries a `capabilities` object with two booleans, `inline` and `offline`, derived from the entry's own config so they can never disagree with it. This is a first-class **gate**: an evaluator that is offline-only (every Quality, Conversation, Trajectory, and Voice entry, because it needs a reference, history, or trajectory) is never offered as an inline guardrail. An entry is inline-capable only when it defines an inline action, and offline-capable only when it defines an offline config.

Two more flags keep the catalog honest:

- `enforced` - whether the evaluator is actually wired to execution. Five text detectors (`pii_leakage`, `code_injection`, `prompt_injection`, `toxicity`, `bias_fairness`) report `enforced: true`; every other entry reports `false` so the UI can distinguish catalogued from wired.
- `higherIsBetter` - score polarity. Negative-signal judges (toxicity, bias, hallucination, and similar) score a BAD signal, so a high score is worse and this flag is `false`.

### Binding an evaluator

Evaluators are bound to a firewall scope through `FirewallConfig.evaluators`, a list of `EvaluatorBinding` entries that cascade node → org → agent with the same union-plus-lock semantics as the firewall dials (see [Hierarchical policy resolution](/docs/gateway/security)).

| Binding field | Meaning |
|---|---|
| `id` | The catalog evaluator this binding enables |
| `enabled` | Whether the binding is active in this scope |
| `inline_action` | The `FirewallPolicy` (Block / Sanitize / Warn) applied inline when it fires |
| `offline` | Offline scoring config (threshold, judge model) |
| `locked` | Freeze the binding so a narrower scope can only tighten it |

<Callout type="info">
Inline evaluator bindings (`FirewallConfig.evaluators`) cascade node → org → agent and are
enforced on the chat pipeline for input- and output-target evaluators that report
`enforced: true` (the five text detectors plus additional wired judges as they land). Image-target
evaluators are catalogued but not yet enforced on the image path. The `enforced` flag on each
catalog entry remains the honesty signal for what a surface may claim is live.
</Callout>

## Audit

The audit log (`apps/gateway/src/audit/mod.rs`, `AuditLogger`) records every model call to a SQLite
store, alongside sandbox exec events and identity-vault credential reads. Writes go to a background OS
thread over a bounded channel, so the async request path never blocks on disk I/O; reads use a
separate query-only connection under WAL mode. The store is off when `AuditConfig.enabled` is false,
in which case writes are dropped and queries return empty.

### DB path

The default path is resolved by `dirs::data_local_dir()`:

- Linux/macOS - `~/.local/share/ryu/audit.db` (respects `$XDG_DATA_HOME`)
- Windows - `%LOCALAPPDATA%\ryu\audit.db`

Override with `db_path` under `[audit]` in `gateway.toml`.

### Record schema

Each row captures the request, its tenant attribution, usage, and quality. The `event_type`
discriminator separates a `model_call` from an `exec_call` or a `credential_read`.

| Field | Notes |
|---|---|
| `request_id` | Gateway-internal id, also returned as the `x-request-id` response header |
| `api_key` | Redacted to a short prefix in query results; raw keys never leave the store |
| `user_name`, `org_id`, `team_id`, `project_id` | Tenant attribution |
| `provider`, `model` | Resolved route for the call |
| `input_tokens`, `output_tokens`, `cache_hit`, `latency_ms` | Usage and timing |
| `eval_score` | The `overall` eval score when this request was sampled, else null |
| `skill_ids` | Comma-separated active Agent Skill ids, from `x-ryu-skill-ids` |
| `session_id` | Core conversation/session id, from `x-ryu-session-id` |
| `error` | Error message when the call failed |
| `event_type`, `backend`, `command`, `duration_ms`, `exit_code` | Exec and credential-read fields, null for model calls |

### Querying

```
GET /v1/audit
```

returns matching rows newest-first with `count` and `entries`. Every filter is an optional
query-string parameter combined with `AND`. Filters cannot inject SQL (the WHERE clause is fully
parameterized), and `limit` is clamped to a maximum of 1000 (default 100).

| Filter | Matches on |
|---|---|
| `api_key` | The full (unredacted) API key |
| `org_id`, `team_id`, `project_id` | Tenant scope |
| `provider`, `model` | Resolved route |
| `request_id` | A single gateway request |
| `session_id` | All rows belonging to one Core session |
| `errors_only` | Only rows that recorded an error |
| `limit` | Row cap, 1 to 1000 |

This endpoint is master-key only. Audit data is tenant-wide and sensitive, so a per-tenant key cannot
read it. The one exception is no-auth mode from a loopback peer, and even that loopback trust is
withdrawn when [mesh](/docs/core/node-and-presence) is enabled (a tailnet peer can appear as
`127.0.0.1`). From Core, `GET /api/gateway/audit` proxies the supported filters and returns
`reachable: false` rather than erroring when the Gateway is down.

<Callout type="warn">
  The audit log records and the eval runner scores, but there is no built-in retention, rotation, or
  alerting on this data yet.
</Callout>

<Callout type="info">
  The Gateway records tokens, cost, and quality. Core records sequencing and latency. The two are
  joined by `session_id`, which Core forwards as `x-ryu-session-id`. See
  [Observability](/docs/core/observability).
</Callout>
