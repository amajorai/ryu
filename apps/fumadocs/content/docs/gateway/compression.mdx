---
title: Context Compression
description: Auto-compress every gateway-routed agent's context with headroom, applied as a Gateway egress transform.
---

Long conversations and large tool outputs burn tokens. Ryu can compress the
messages of **every model call** before they reach the provider - automatically,
for every agent that routes through the Gateway - using the
[headroom](https://github.com/chopratejas/headroom) engine (60-95% fewer tokens,
same answers).

## Why it lives in the Gateway

Compressing what gets sent to the model is deciding *what is shared* with the
provider - a **Gateway** concern, not a Core one (see the Core-vs-Gateway rule in
the architecture docs). So compression is a **swappable egress transform** on the
Gateway pipeline, applied just before the upstream provider call. Nothing is
hardcoded: it is one config toggle pointing at a compression service.

Because the default OpenAI-compat chat path and the `ryu`, `pi`, and `codex` ACP
agents already route their egress through the Gateway, enabling compression
**auto-wraps all of them** - no per-agent setup, no editing each agent's config.
This is the Ryu-native form of `headroom wrap`: one switch, every agent.

<Callout type="warn">
  Claude Code (Anthropic `/v1/messages`) and Gemini CLI run as ACP subprocesses
  that speak non-OpenAI provider formats and currently **bypass the Gateway**, so
  they are not yet covered. Routing them through a translating ingress is tracked
  as a follow-up.
</Callout>

## Enable it

1. Install the headroom proxy (the minimal extra - no serena, no ML stack):

   ```bash
   pipx install --python python3.13 "headroom-ai[proxy]"
   ```

2. Turn it on for Core:

   ```bash
   export RYU_HEADROOM_ENABLED=1
   ```

That's it. On startup Core launches the headroom proxy (`headroom proxy`) and
tells the Gateway to compress against it. Every gateway-routed model call now
flows through `POST /v1/compress` before going upstream.

## How it works

- **Core** decides *what runs*: it manages the optional `headroom proxy` process
  (`apps/core/src/sidecar/headroom.rs`) and, when enabled, sets the Gateway's
  compression env on spawn.
- **Gateway** owns the *transform*: `apps/gateway/src/compression.rs` posts the
  request messages to the compression service and swaps in the compressed result
  before the provider call - after routing/budgets, once for the whole fallback
  chain, so routing, caching, and budgets stay intact.
- **Fails open**: if headroom is absent, unhealthy, or slow, the Gateway leaves
  the original messages untouched and the call proceeds normally. Enabling
  compression can never break chat.

## Configuration

| Setting | Env var | Default |
|---|---|---|
| Enable compression (Core) | `RYU_HEADROOM_ENABLED` | off |
| Headroom proxy URL (Core) | `RYU_HEADROOM_URL` | `http://127.0.0.1:8787` |
| Core-managed proxy | `RYU_HEADROOM_MANAGED` | on |
| Gateway compression toggle | `GATEWAY_COMPRESSION_ENABLED` | set by Core |
| Gateway compression URL | `GATEWAY_COMPRESSION_URL` | set by Core |
| Min messages before compressing | `gateway.toml [compression] min_messages` | 4 |

The Gateway settings are normally managed by Core, but the Gateway is
self-contained: you can run it standalone and enable compression directly with
`GATEWAY_COMPRESSION_ENABLED=1` + `GATEWAY_COMPRESSION_URL` pointed at any
service that implements headroom's `/v1/compress` contract.
