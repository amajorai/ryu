---
title: Gateway for any OpenAI-compatible agent
description: Point Mastra, LangChain, or the raw OpenAI SDK at the Ryu Gateway with a single base-URL swap.
---

The Ryu Gateway speaks the full OpenAI-compatible API (`/v1/chat/completions`, `/v1/models`).
Any framework that accepts a custom `baseURL` works without code changes: swap the URL, add
your gateway API key, and every request runs through Ryu's firewall, PII/DLP, rate-limits,
budgets, evals, and audit log before reaching the upstream provider.

## Prerequisites

- Ryu Gateway running locally (default: `http://127.0.0.1:7981`)
- A gateway API key configured in your `gateway.toml` (see [Auth config](#auth-config) below)

## Auth config

Auth is controlled by `AuthConfig` in `apps/gateway/src/config.rs`. Add a static key to
`gateway.toml`:

```toml
[auth]
require_auth = true

[[auth.api_keys]]
key  = "ryu-sk-your-secret-key"
name = "my-app"
```

All requests must include the key as a Bearer token:

```
Authorization: Bearer ryu-sk-your-secret-key
```

When `require_auth = false` (the default for local dev) every request is accepted regardless
of the header.

## OpenAI SDK (TypeScript)

```typescript
import OpenAI from "openai";

const client = new OpenAI({
  baseURL: "http://127.0.0.1:7981/v1",
  apiKey: "ryu-sk-your-secret-key", // from gateway.toml [auth.api_keys]
});

const response = await client.chat.completions.create({
  model: "gpt-4o",        // any model listed by GET /v1/models
  messages: [{ role: "user", content: "Hello from the Ryu Gateway!" }],
});

console.log(response.choices[0].message.content);
// x-provider header tells you which upstream handled the call
```

## Mastra

```typescript
import { Mastra } from "@mastra/core";
import { openai } from "@ai-sdk/openai";
import { createOpenAI } from "@ai-sdk/openai";

// Override the OpenAI provider to point at the gateway
const ryuGateway = createOpenAI({
  baseURL: "http://127.0.0.1:7981/v1",
  apiKey: "ryu-sk-your-secret-key",
});

const mastra = new Mastra({
  llm: ryuGateway("gpt-4o"),
});
```

## LangChain (TypeScript)

```typescript
import { ChatOpenAI } from "@langchain/openai";

const model = new ChatOpenAI({
  modelName: "gpt-4o",
  openAIApiKey: "ryu-sk-your-secret-key",
  configuration: {
    baseURL: "http://127.0.0.1:7981/v1",
  },
});

const response = await model.invoke("Hello from LangChain via the Ryu Gateway!");
console.log(response.content);
```

## Available models

`GET /v1/models` returns the full list of routable models. As of this writing the built-in
list includes:

| Model ID | Provider |
|---|---|
| `gpt-4o`, `gpt-4o-mini`, `gpt-4-turbo`, `o1`, `o3-mini` | OpenAI |
| `claude-opus-4-5`, `claude-sonnet-4-5`, `claude-haiku-4-5` | Anthropic |
| `claude-3-5-sonnet-20241022`, `claude-3-5-haiku-20241022` | Anthropic |
| `zeroclaw`, `openclaw` | Core agents |
| `openrouter/auto` | OpenRouter (auto-select) |
| `llama3.2:latest`, `mistral:latest`, `phi4:latest`, `deepseek-r1:latest` | Local (Ollama) |

The list is driven by `builtin_model_list()` in `apps/gateway/src/router/mod.rs`. Add custom
mappings via `[routing.model_map]` in `gateway.toml`.

## Response headers

The gateway adds diagnostic headers to every response:

| Header | Content |
|---|---|
| `x-provider` | Which upstream provider handled the call (`openai`, `anthropic`, `local`, …) |
| `x-ryu-cached` | `true` when the response was served from the exact or semantic cache |
| `x-ryu-request-id` | Unique request ID for the audit log |

## Smoke test

A runnable smoke script is checked in at `examples/gateway/openai-compat-smoke.ts`. It sends a
chat completion to the locally running gateway and prints the `x-provider` header:

```bash
# Start the gateway first
cargo run --manifest-path apps/gateway/Cargo.toml

# Then run the smoke test
bun run examples/gateway/openai-compat-smoke.ts
```

Expected output (provider depends on your routing config):

```
> Sending chat completion to http://127.0.0.1:7981/v1 ...
  x-provider: openai
  Response: Hello! I'm running through the Ryu Gateway.
> Smoke test passed.
```

## Self-hosting checklist

1. Set upstream API keys in environment variables (`OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, …).
2. Create `gateway.toml` with `[auth]` and at least one `[[auth.api_keys]]` entry.
3. Run `cargo run --manifest-path apps/gateway/Cargo.toml -- --config gateway.toml`.
4. Set `baseURL = http://<host>:7981/v1` and `apiKey = <your-gateway-key>` in your agent.
5. Verify with `curl -H "Authorization: Bearer <key>" http://localhost:7981/v1/models`.

## Subscription-preserving passthrough (Claude Code, Codex)

The base-URL swap above works for any agent that speaks OpenAI-compatible JSON and authenticates
with an API key. Some agents do neither. Claude Code speaks the Anthropic Messages format and
authenticates with the user's own Pro/Max OAuth bearer, and Codex (in ChatGPT-login mode) speaks the
OpenAI Responses format and authenticates with an OAuth bearer plus a `ChatGPT-Account-ID` header.
The regular OpenAI-compat chat path cannot govern either, because it would substitute the gateway's
own provider key and flip the request off the user's subscription billing.

For those agents the gateway runs a **transparent reverse proxy** in the agent's native wire format,
`apps/gateway/src/passthrough/mod.rs`. The agent is pointed at the proxy through a base-URL env var.
The proxy accepts the native request, applies request-side DLP, forwards it upstream with the
caller's own `Authorization` header (and `ChatGPT-Account-ID` for Codex) unchanged, then streams the
response back through response-side DLP and audit. Because the format is never translated, the
agent's subscription billing is preserved end to end.

| Route | Upstream | Auth forwarded unchanged | Wire format |
|---|---|---|---|
| `/passthrough/anthropic/*` | `api.anthropic.com` | subscription OAuth bearer | Anthropic Messages |
| `/passthrough/openai-responses/*` | `chatgpt.com/backend-api/codex` | OAuth bearer + `ChatGPT-Account-ID` | OpenAI Responses |

Both upstreams are overridable without a rebuild, via `RYU_PASSTHROUGH_ANTHROPIC_UPSTREAM` and
`RYU_PASSTHROUGH_CODEX_UPSTREAM`.

### What the proxy redacts

The proxy is governed by the same firewall as the OpenAI-compat path (see
[Security](/docs/gateway/security)). The audit record is always emitted, even when the firewall is
off and even when the upstream call fails, so with the firewall disabled the proxy is a transparent
forwarder plus an audit log.

- **Request side**: when the firewall is enabled, the outbound body is parsed and the prompt-bearing
  fields are run through the firewall's `sanitize`. For Anthropic that is `system` and each message's
  `content`, for Codex it is `instructions` and each `input` item's `content`. A non-JSON or
  unparseable body is forwarded as-is (fail open).
- **Response side**: when the firewall is enabled, the upstream SSE stream is reassembled across
  network and UTF-8 chunk boundaries, and each assistant-text delta (`text_delta` for Anthropic,
  `output_text.delta` for Responses) is rewritten to `[REDACTED:…]` before the bytes reach the
  caller. Non-text events (`message_start`, `ping`, tool-call deltas) pass through verbatim and SSE
  framing is preserved. The accumulated redacted text is scanned once at stream end to audit any
  residual outbound violation. An unparseable event passes through verbatim (fail open).

### Enabling it

The passthrough is opt-in, configured per agent through Core preferences. Claude Code uses the
`claude-gateway-routing` preference (`apps/core/src/claude_config/mod.rs`), which injects
`ANTHROPIC_BASE_URL` pointed at the gateway's `/passthrough/anthropic` route. Codex uses
`codex-gateway-routing` (`apps/core/src/codex_config/mod.rs`), which writes an isolated `CODEX_HOME`
whose `config.toml` declares a custom provider with `wire_api = "responses"` and no `env_key` (so the
subscription credential is what reaches upstream), pointed at `/passthrough/openai-responses`. Both
default to off, because enabling them changes how the subscription credential flows.

<Callout type="warn">
  Built and compiling, but the full live round-trip is not yet exercised against a real subscription
  agent. The proxy is **loopback-only**: it refuses a request from a non-loopback peer
  (`apps/gateway/src/passthrough/mod.rs`), so a remote gateway never carries a user's subscription
  bearer. It is opt-in (off by default). Response-side redaction is per-delta, so a secret split
  across two separate text-delta events is caught only by the end-of-stream audit, not rewritten
  inline (a cross-delta hold-back buffer is a deliberate follow-on). The Gemini CLI is **not
  wirable** through either mechanism, because it exposes no base-URL hook.
</Callout>

## Generic gateway routing for a BYO agent

For an OpenAI-compatible agent the user added themselves (`engine = "acp-exec:<command>"`), Core
offers a generic per-agent toggle, `apps/core/src/agent_routing/mod.rs`. When enabled for an agent,
Core injects `OPENAI_BASE_URL` (the local gateway `/v1`) and `OPENAI_API_KEY` (the gateway token)
into that agent's spawn command via `openai_gateway_cmd`, so an agent whose HTTP client honours the
OpenAI base URL routes its model calls through the gateway's firewall, budget, and audit pipeline
with no manual env wiring. The whole set lives under a single `agent-gateway-routing` preference
holding a JSON object of agent id to enabled boolean.

<Callout type="info">
  This is the explicit M2 "point any agent at the gateway" lever, but it is honest about its limits.
  It is a genuine no-op for agents that do not read `OPENAI_BASE_URL`: Gemini speaks Google format,
  OpenClaw talks to its own gateway, Hermes uses its own credentials, and the managed Pi ignores the
  env var (it routes through its own `models.json` instead). The toggle is surfaced primarily for the
  `acp-exec:` BYO path, where it does exactly what was asked.
</Callout>
