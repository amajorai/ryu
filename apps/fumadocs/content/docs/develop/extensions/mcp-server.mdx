title: Register an MCP Server
description: Expose any tool to Ryu agents over the Model Context Protocol, using the real headroom engine as the worked example.

The **Model Context Protocol (MCP)** is an open standard for connecting AI agents to external
tools and data. A tool exposed over MCP is callable by any MCP host - including Ryu Core's MCP
registry. This is the fastest way to add a capability to Ryu: no manifest, no SDK, no restart.

Ryu speaks MCP over stdio (JSON-RPC 2.0, newline-delimited) and matches protocol version
`2024-11-05`. See the [MCP specification](https://modelcontextprotocol.io) for the full protocol.

This guide uses the **headroom** context-compression engine
([`chopratejas/headroom`](https://github.com/chopratejas/headroom)) as the worked example: it
ships a real MCP server, so you register an *existing* server rather than writing one. To build
your own server instead, jump to [Build your own](#build-your-own).

## Step 1 - Install the engine (serena-free)

Headroom's MCP **serve** path needs only the `[mcp]` extra. serena is **not** a dependency of
`headroom-ai` (it is in no extra, not even `[all]`) - it is only added as a *companion server* by
headroom's `wrap`/registry CLI, which we do not use here. Install the minimal MCP build (skip the
heavyweight `[all]` extra):

```bash
pipx install --python python3.13 "headroom-ai[mcp]"
# or: pip install "headroom-ai[mcp]"
```

This installs the `headroom` CLI and the MCP server only - no serena, no ML stack.

## Step 2 - Register with Core

Core exposes `POST /api/mcp/servers` (`apps/core/src/server/mod.rs`). It appends the server to
`~/.ryu/mcp.json` and hot-reloads the registry, so the tools become callable without a restart:

```bash
curl -X POST http://localhost:7980/api/mcp/servers \
  -H 'content-type: application/json' \
  -d '{
    "name": "headroom",
    "command": "headroom",
    "args": ["mcp", "serve"],
    "description": "Context compression: compress, retrieve, stats."
  }'
```

`headroom mcp serve` speaks the stdio MCP transport and needs no environment variables. (The
optional `--proxy-url` flag only matters if you run a separate headroom proxy.) Two rules:

- `name` must be non-empty and must **not** contain `__` - Core uses `__` as the tool-id
  separator when routing calls.
- `headroom` must be on Core's `PATH` (handled by `pipx ensurepath`); otherwise point `command`
  at the absolute path to the binary in your virtualenv.

Once registered, the tools are listed by `GET /api/mcp/servers` and callable via
`POST /api/mcp/tools/call`. Tool ids are namespaced by server name, e.g.
`headroom__headroom_compress`.

The built-in **Ghost** and **Shadow** servers are registered the same way, but as built-ins:
they are wired directly into the MCP registry (`apps/core/src/sidecar/mcp/mod.rs`) rather than
through `~/.ryu/mcp.json`, so they are available out of the box.

## Step 3 - Call it from chat

With the server registered, its tools are injected into ACP agent sessions through Core's MCP
bridge (`apps/core/src/sidecar/adapters/mcp_bridge.rs`). When an agent emits a tool call for one
of your tools, Core dispatches it through the registry and feeds the result back into the chat
loop - the same closed tool loop the built-ins use.

## Build your own

To author a new MCP server rather than register an existing one, use the SDK's `McpServer`
(`packages/sdk/src/mcp/server.ts`). You register `SdkRunnable` objects (a `name` plus a `run`
function); the SDK handles the `initialize` handshake, `tools/list`, and `tools/call` framing and
the MCP content-block envelope for you. Register the resulting stdio command with Core exactly as
in Step 2. See the [TypeScript SDK guide](/docs/develop/extensions/typescript-sdk) for the full surface.

## headroom tools

| Tool | Parameters | Returns |
|---|---|---|
| `headroom_compress` | `text: string`, `max_tokens?: number` | compressed text + a handle for reversible retrieval |
| `headroom_retrieve` | `query: string`, `k?: number` | the original / most relevant content for the query |
| `headroom_stats` | - | cumulative metrics: tokens in/out and compression ratio |

## Allowlisting

<Callout type="warn">
  Tools are not offered to an agent unless they are on that agent's **allowlist**. The MCP bridge
  filters every registered tool through a per-agent allowlist before injecting it into a session
  (`apps/core/src/sidecar/adapters/mcp_bridge.rs`): an empty allowlist means no tools are offered.
  After registering a custom MCP server, make sure its tools are allowlisted for the agents that
  should use them - registration alone does not grant access.
</Callout>

## Automatic compression vs. tool calls

Registering headroom as above gives agents compress/retrieve **as tools they choose to call**.
Compressing *every* model call automatically is a separate, complementary path: an egress
transform in the **Gateway**, enabled with `RYU_HEADROOM_ENABLED=1`. It auto-wraps every
gateway-routed agent (the default chat path plus the `ryu`/`pi`/`codex` ACP agents) with no
per-agent setup - see [Context Compression](/docs/gateway/compression). Keep MCP servers focused
on the tool surface and let the Gateway own on-the-wire transforms and policy.
