title: Extensions Overview
description: Three ways to extend Ryu - MCP servers, plugin.json manifests, and the TypeScript SDK.
---

Ryu is built so that **nothing is hardcoded and everything is swappable**. Extending it means
adding new capabilities - tools, agents, workflows, skills - that plug into the same Core and
Gateway every built-in uses. There are three paths, in increasing order of structure.

## Three ways to extend Ryu

1. **MCP server** - expose any tool over the Model Context Protocol. No manifest, no SDK, no
   restart required. Register it with Core and its tools become callable from chat immediately.
   This is the fastest path and the one the [MCP server guide](/docs/develop/extensions/mcp-server)
   worked example uses.

2. **`plugin.json` manifest** - bundle one or more Runnables (tools, agents, workflows, skills)
   plus the permission grants they need into a single installable plugin. Core validates the
   manifest, writes it to disk, and hot-reloads it with no restart.

3. **TypeScript SDK (`@ryuhq/sdk`)** - define agents, workflows, tools, and skills as typed
   Runnables in code, with the Gateway mandatory on every model call. Scaffold a project with
   `create-ryu-app`, author Runnables with `defineAgent` / `defineTool` / `defineSkill`, and
   pack them into a `plugin.json` with `bunx ryu pack`.

## When to use each

| You want to... | Use |
|---|---|
| Expose a single tool or small toolset quickly | MCP server |
| Wrap an existing process in any language (Python, Go, stdio) | MCP server |
| Ship a named, versioned, installable bundle of Runnables | `plugin.json` manifest |
| Declare permission grants the Gateway should enforce | `plugin.json` manifest |
| Author agents/workflows in typed TypeScript with Gateway-routed model calls | TypeScript SDK |
| Validate and package a plugin for distribution | TypeScript SDK (`bunx ryu pack`) |

An MCP server and a `plugin.json` manifest are not mutually exclusive. The MCP server is the
executable that exposes the tools; the manifest is the install and lifecycle wrapper for the
same surface.

## Quick start

```bash
bunx create-ryu-app my-plugin
cd my-plugin
bun install
bunx ryu pack   # produces plugin.json
```

The scaffolded project includes a starter Runnable, a gateway-pointed model config, and a
`plugin.json` manifest ready to load into Core.

## Desktop UI contributions

Plugins can also contribute **desktop UI** - a route, command, or settings section rendered
inside the desktop app. The desktop extension host (`apps/desktop/src/contributions/host/`)
renders a plugin's document in a sandboxed null-origin `<iframe>` with a capability-gated
postMessage RPC bridge. The contract a plugin codes against ships in the SDK at
`packages/sdk/src/plugin/ryu-plugin.ts`.

<Callout type="warn">
  The desktop UI host is an early MVP. The bridge and RPC gate are built and unit-tested, but
  the in-webview handshake is not yet exercised end-to-end, and the granted-capability set is
  host config rather than read from `plugin.json` grants. See
  [Desktop companion](/docs/develop/extensions/desktop-companion) for the current state.
</Callout>
