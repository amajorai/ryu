A **Ryu App** bundles one or more tools whose results render an interactive widget inline in a chat
reply - a checklist you tick, a data grid you sort, a budget dial you turn. This page is the
authoring path: how you build one and install it. For how a widget renders, the sandboxed iframe,
and the governance gate, see [Ryu Apps](/docs/desktop/user-guide/apps) - this page does not re-state
those.

<Callout type="warn">
  Ryu Apps are gated behind the plugin-runtime flag (`PLUGIN_RUNTIME_FLAG`) and the end-to-end path
  is not yet fully live-verified (headless widget render is proven; a running node whose model calls
  a widget-bearing tool in the live desktop is not). `defineApp` in v1 is **declarative
  pass-through only** - there is no `run` handler, so a widget renders from its tool's arguments and
  output; third-party tool *code* needs the [plugin runtime](/docs/develop/extensions/plugin-runtime),
  which is out of scope here. Treat this as an early surface.
</Callout>

## 1. Declare the app with `defineApp`

`defineApp` (`packages/sdk/src/runnable/app.ts`, from `@ryuhq/sdk`) assembles a complete
`plugin.json` manifest for a Ryu App from a declarative description. Each tool is one of two kinds,
and the split is derived exactly the way Core's in-process provider derives it
(`apps/core/src/sidecar/mcp/apps/mod.rs`).

| Tool kind | `accessible` | What it does |
|---|---|---|
| **render** | unset / false | Produces the widget. Gets a `contributes.widgets[]` entry binding its id to `ui://widget/<slug>.html`, and its config carries `widget: true` plus `invoking` / `invoked` status labels. |
| **companion** | `true` | A call target a mounted widget may invoke back (`callTool`). Carries `widget_accessible: true`, gets no widget template. |

```ts
import { defineApp } from "@ryuhq/sdk";

export default defineApp({
  id: "com.example.checklist",
  title: "Checklist",
  version: "1.0.0",
  slug: "checklist",
  uiEntry: "src/checklist.tsx",
  grants: [],
  tools: [
    { name: "render", description: "Render a checklist", invoking: "Building…", invoked: "Ready" },
    { name: "toggle", description: "Toggle an item", accessible: true },
  ],
});
```

The `slug` builds the widget uri (`ui://widget/checklist.html`) and, unless you set `server`,
namespaces each tool id as `<server>__<name>`. A render tool's widget may call tools only when the
app declares at least one companion tool. The result is validated through `PluginManifestSchema`, so
a malformed app throws at authoring time, not at install.

## 2. Author the widget UI

`uiEntry` points at the source for the widget itself (React, or anything that compiles to a single
HTML file). The widget renders inside a null-origin sandboxed iframe and talks to the host through
`window.openai` (the OpenAI Apps SDK surface, aliased from Ryu's native `window.ryu` bridge). It
reads its tool's validated arguments and result from `window.openai.toolInput` / `toolOutput` - Core
echoes the validated arguments back as `structuredContent`, so the widget has data to render without
any server round-trip. A companion `callTool` goes back through the host, never direct from the
frame.

## 3. Build and pack

The widget must build to a **self-contained** HTML file - all JS and CSS inlined - because it loads
under a `default-src 'none'` CSP iframe that cannot fetch an external chunk.

<Callout type="warn">
  Build **one app per Vite invocation** (single input). A multi-entry build hoists a shared React
  chunk that `vite-plugin-singlefile` leaves external (a `./chunk.js` reference), which cannot load
  under the null-origin iframe and renders a blank widget. The built-in apps loop each app through
  its own single-input build for exactly this reason
  (`packages/ryu-apps/scripts/build-all.ts`, run via `bun run build`); a Rust regression test asserts
  every bundle is self-contained.
</Callout>

Then pack: `ryu pack` bundles your `uiEntry` output into the manifest's `ui_code` field (with a
`ui_code_sha256`). At install, Core splits the bundle - `ui_code` is the widget carriage, the rest
is the manifest - and refuses a bundle whose code does not match the declared hash
(`apps/core/src/server/mod.rs`).

## 4. Install and enable

Install the packed bundle and enable it, the same lifecycle as any plugin
([App Manifest & Lifecycle](/docs/core/app-manifest-lifecycle)):

```bash
curl -X POST http://localhost:7980/api/plugins/install \
  -H 'content-type: application/json' \
  -d @my-app.bundle.json

curl -X POST http://localhost:7980/api/plugins/com.example.checklist/enable
```

Once enabled, Core's in-process app provider advertises the render tool with its widget `_meta`
binding and serves the widget HTML as a `ui://` MCP resource - no subprocess. When the agent calls
the render tool, the widget mounts in the message list.

## Which agents can emit a widget

A widget only appears if the agent actually calls the render tool through a governed tool loop.

- The batteries-included **local engine** (Gemma) and **OpenAI-compat** registry agents funnel
  through `route_openai_stream` and emit widgets with no per-agent config.
- Bridged ACP agents (Claude Code, Codex, Gemini, or a BYO agent with gateway routing on) emit via
  the MCP bridge.
- The flagship **`ryu` / Pi** agent (`AgentRoute::Acp`) cannot emit a widget from its own tool
  calls: Pi (an R1 fork) exposes no base-URL or MCP hook to inject the app tools into its
  in-subprocess loop. This is inherent to that engine, not a gap in the app.

## Governance

Widget-initiated calls are never trusted from the frame. A companion `callTool` goes to the host
(which holds the node token), through Core's provenance gate (`apps/core/src/server/widgets.rs`),
then to the Gateway's `POST /v1/exec/tool` for scan, budget, forward, and audit. The full path is
documented in [Ryu Apps](/docs/desktop/user-guide/apps#provenance-and-gateway-gating).
