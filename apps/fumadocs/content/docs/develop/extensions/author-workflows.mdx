A workflow chains agents, tools, and other Runnables to do work that no single step does alone:
research a topic, transform the result, hand it to an agent, return a report. Ryu gives you two
ways to author one, and they are different artifacts.

- **The SDK `defineWorkflow`** writes an imperative workflow: a `run` function that calls each step
  in plain TypeScript. You decide the control flow with normal code (`if`, `for`, `await`). Ships
  inside a plugin bundle, runs through the gateway.
- **The Core DAG engine** runs a declarative JSON graph of typed nodes and edges. You author it in
  the desktop canvas or by POSTing JSON to Core. The engine reference (node kinds, triggers,
  durability, template variables) lives at [/docs/core/workflows](/docs/core/workflows).

This page covers authoring in code with the SDK. For the visual canvas, see the user how-to at
[/docs/desktop/user-guide](/docs/desktop/user-guide); for the full node catalog and run
semantics, see the engine reference linked above.

<Callout type="info">
Both paths obey the same rule: every model call goes through the gateway. The SDK injects a
`ctx.gateway` client (`packages/sdk/src/runnable/runnable-types.ts`) and forbids direct provider
imports. A Runnable that needs a model must call `ctx.gateway.chat(...)`.
</Callout>

## What `defineWorkflow` gives you

`defineWorkflow` (`packages/sdk/src/runnable/workflow.ts`) returns a `Runnable` with
`kind: "workflow"`. A workflow is a peer of agents and tools, not a layer above them: it lists
other Runnables as `steps` and invokes each by calling `step.run(input, ctx)`. An agent listed as a
step can equally call a workflow as a tool. This is the one Runnable contract described at
[/docs/start-here/architecture/runnable-model](/docs/start-here/architecture/runnable-model).

The factory accepts these options (`WorkflowOptions` in `workflow.ts`):

| Field | Type | Purpose |
|---|---|---|
| `id` | `string` | Stable unique identifier, e.g. `workflow-report` |
| `name` | `string` | Human-readable display name |
| `run` | `(input, ctx) => Promise<output>` | The orchestration body; calls steps via `step.run` |
| `steps` | `readonly Runnable[]` | Optional list of Runnables this workflow orchestrates |

`ctx` is the `RunnableContext`: it always carries `ctx.gateway` (never null, fail-closed at
construction), plus an optional `ctx.sessionId` and `ctx.signal` for aborting a long run.

## A worked multi-node workflow

This example wires three nodes: a tool that fetches source text, a gateway call that summarizes it,
and an agent that drafts the final report. Each step is a Runnable; the `run` body is the DAG,
expressed as straight-line code.

```ts
import {
  defineAgent,
  defineTool,
  defineWorkflow,
} from "@ryu/sdk";

// Node 1: a tool that fetches raw text for a topic.
const fetchSource = defineTool({
  id: "tool-fetch-source",
  name: "Fetch Source",
  async run({ topic }: { topic: string }) {
    const res = await fetch(
      `https://example.com/search?q=${encodeURIComponent(topic)}`
    );
    return { text: await res.text() };
  },
});

// Node 2: an agent that drafts a report from a summary.
const reportAgent = defineAgent({
  id: "agent-reporter",
  name: "Reporter",
  async run({ summary }: { summary: string }, ctx) {
    const { content } = await ctx.gateway.chat([
      { role: "system", content: "Write a one-paragraph report." },
      { role: "user", content: summary },
    ]);
    return { report: content };
  },
});

// The workflow: fetch -> summarize (gateway) -> report (agent).
export const reportWorkflow = defineWorkflow({
  id: "workflow-report",
  name: "Topic Report",
  steps: [fetchSource, reportAgent],
  async run({ topic }: { topic: string }, ctx) {
    // Edge 1: input -> fetchSource
    const { text } = await fetchSource.run({ topic }, ctx);

    // Edge 2: fetchSource -> a gateway summarization call
    const { content: summary } = await ctx.gateway.chat([
      { role: "system", content: "Summarize the text in 3 bullets." },
      { role: "user", content: text },
    ]);

    // Edge 3: summary -> reportAgent (a conditional branch)
    if (summary.trim().length === 0) {
      return { report: "No source material found." };
    }

    const { report } = await reportAgent.run({ summary }, ctx);
    return { report };
  },
});
```

Branching (`if`), looping (`for`), and parallel fan-out (`Promise.all`) are just TypeScript here.
You do not need a special node kind for them, unlike the JSON DAG.

## Calling tools from a step

A step can reach any tool in the Core registry the same way a tool node does in the engine: by the
fully-qualified `<server>__<tool>` id (for example `spider__crawl`). In the SDK you call it through
your own `defineTool` wrapper or via the gateway tool-exec front. The tool-execution contract
(allowlist gating, exec-budget, exec-audit) is identical on both planes and is documented at
[/docs/core/unified-tool-catalog](/docs/core/unified-tool-catalog) and
[/docs/gateway/tools](/docs/gateway/tools). Do not re-implement allowlisting in your workflow; the
gateway enforces it.

## Packaging the workflow

A `defineWorkflow` runs as part of a plugin bundle. Declare it in your `plugin.json` manifest and
ship it with `ryu pack`. The manifest schema and the `contributes` block are documented at
[/docs/develop/extensions/plugin-json-manifest](/docs/develop/extensions/plugin-json-manifest); the
SDK factories and the gateway-mandatory model client at
[/docs/develop/sdk](/docs/develop/sdk).

## When to use the JSON DAG instead

Choose the declarative engine when you want triggers (cron, webhook, Composio), per-node retry and
timeout policies, durable run state that survives a Core restart, or a visual editor a non-developer
can change. That engine has 17 typed node kinds (`Input`, `Output`, `Prompt`, `Condition`,
`Transform`, `Tool`, `Webhook`, `SetState`, `Delay`, `Note`, `While`, `Guardrails`, `SubWorkflow`,
`AgentDelegate`, `Awakeable`, `Recipe`, `GhostAction`) and template variables like `{{input}}`,
`{{nodes.<id>}}`, `{{state.<key>}}`, and `{{trigger.<dotted.json.path>}}`. All of it is documented
once, in the engine reference.

<Callout type="warn">
The JSON DAG and the SDK `defineWorkflow` are separate artifacts. The SDK factory is an imperative
`run` function; it does not emit the 17-node JSON graph, and the engine does not run TypeScript
closures. Pick one per workflow. Some engine nodes carry their own maturity caveats (for example
`GhostAction` and `Recipe` are Windows-first and need an installed ghost binary). Those caveats are
flagged on [/docs/core/workflows](/docs/core/workflows).
</Callout>
