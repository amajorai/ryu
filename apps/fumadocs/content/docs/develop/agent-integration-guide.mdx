This guide consolidates everything an AI agent (or a developer building one) needs to know to
integrate with Ryu. Instead of navigating 8+ separate pages, this single reference covers the full
integration surface.

<Callout type="info">
This page is optimized for both human readers and AI agents. If you are an agent consuming this
via `/llms.txt` or the `.mdx` rewrite, the structured sections below give you everything needed
to connect, discover tools, and stay governed.
</Callout>

## Quick start: connect to Ryu

### 1. Point at the Gateway (recommended)

The Gateway exposes an OpenAI-compatible API on `:7981`. Any tool or framework that speaks the
OpenAI protocol works out of the box.

```bash
# List available models
curl http://localhost:7981/v1/models \
  -H "Authorization: Bearer $RYU_TOKEN"

# Chat completion
curl http://localhost:7981/v1/chat/completions \
  -H "Authorization: Bearer $RYU_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-4o",
    "messages": [{"role": "user", "content": "Hello"}],
    "stream": true
  }'
```

### 2. Connect via Core (for full orchestration)

Core on `:7980` provides the full agent lifecycle: conversations, memory, tools, workflows.

```bash
# Chat with the default agent
curl http://localhost:7980/api/chat/stream \
  -H "Authorization: Bearer $RYU_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "messages": [{"role": "user", "content": "Hello"}],
    "agent_id": "ryu"
  }'
```

### 3. Connect via MCP

The `ryu-mcp` server exposes Core tools over the Model Context Protocol (stdio transport).

```json
{
  "mcpServers": {
    "ryu": {
      "command": "ryu-mcp",
      "args": ["serve"],
      "env": {
        "RYU_TOKEN": "your-token",
        "CORE_URL": "http://localhost:7980"
      }
    }
  }
}
```

See [MCP quickstart](/docs/mcp/quickstart) for the full setup.

## Discover available tools

### Via the Gateway tool search

The Gateway provides a search-based tool discovery API:

```bash
# Search for tools
curl http://localhost:7981/v1/tools/search \
  -H "Authorization: Bearer $RYU_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"query": "web browsing", "limit": 5}'

# Describe a specific tool
curl http://localhost:7981/v1/tools/search \
  -H "Authorization: Bearer $RYU_TOKEN" \
  -d '{"query": "agentbrowser"}'
```

### Via Core's unified tool catalog

```bash
# Search all registered tools
curl http://localhost:7980/api/tools/search \
  -H "Authorization: Bearer $RYU_TOKEN" \
  -d '{"query": "file operations"}'

# List MCP server tools
curl http://localhost:7980/api/mcp/tools/search \
  -H "Authorization: Bearer $RYU_TOKEN" \
  -d '{"query": "screenshot"}'
```

### Via the MCP server

If you are connected via MCP, tools are automatically available:

```bash
# List all tools from a connected MCP server
ryu-mcp serve  # then use the MCP protocol to list tools
```

See [Unified tool catalog](/docs/core/unified-tool-catalog) and [MCP tools](/docs/mcp/tools) for
the full tool inventory.

## Stay governed

The Gateway enforces policy on every model call. To ensure your agent stays governed:

### OpenAI-compat agents (Plane A)

When your agent uses the Gateway's `/v1/chat/completions`, all governance is automatic:

- **Firewall** scans requests and responses for PII, prompt injection, and custom patterns
- **Budgets** enforce per-user and per-agent token caps
- **Evals** score response quality
- **Audit** logs every request

No extra configuration needed — the Gateway handles it.

### ACP agents (Plane B)

ACP agents (Claude Code, Codex, Pi, etc.) run as subprocesses. Their LLM calls are self-routed,
but **tool egress** is governed via the MCP bridge. To route ACP agent LLM calls through the
Gateway, enable per-agent gateway routing:

```bash
# Enable gateway routing for an agent
curl http://localhost:7980/api/agent-routing \
  -H "Authorization: Bearer $RYU_TOKEN" \
  -d '{"agent_id": "my-agent", "via_gateway": true}'
```

This injects `OPENAI_BASE_URL` pointing at the Gateway into the agent's subprocess. Not all agents
honour this — it works for BYO OpenAI-compatible agents but is a no-op for Gemini/OpenClaw/Hermes.

See [Gateway for any agent](/docs/gateway/gateway-for-any-agent) and
[Routing planes](/docs/gateway/routing-planes).

## Build with the SDK

The TypeScript SDK (`@ryuhq/sdk`) provides typed factories for building Runnables that route all
model calls through the Gateway.

### Define an agent

```typescript
import { defineAgent } from "@ryuhq/sdk";

const myAgent = defineAgent({
  id: "my-agent",
  name: "My Agent",
  run: async (input, ctx) => {
    const result = await ctx.gateway.chat([
      { role: "system", content: "You are a helpful assistant." },
      { role: "user", content: input },
    ]);
    return result.content;
  },
});
```

### Define a tool

```typescript
import { defineTool } from "@ryuhq/sdk";

const weatherTool = defineTool({
  id: "weather",
  name: "Get Weather",
  schema: {
    type: "object",
    properties: {
      city: { type: "string", description: "City name" },
    },
    required: ["city"],
  },
  run: async (input) => {
    const res = await fetch(`https://api.weather.com/${input.city}`);
    return res.json();
  },
});
```

### Bundle into a plugin

Create a `manifest.json`:

```json
{
  "id": "com.example.my-plugin",
  "name": "My Plugin",
  "version": "1.0.0",
  "runnables": [
    { "kind": "agent", "id": "my-agent", "name": "My Agent" },
    { "kind": "tool", "id": "weather", "name": "Get Weather" }
  ],
  "targets": ["core"]
}
```

Pack and install:

```bash
bun run pack     # creates the bundle
ryu plugins install ./dist/  # install into Core
```

See [SDK Reference](/docs/develop/sdk) for the full API surface.

## Available built-in tools

Ryu registers these tools in the MCP registry by default when their sidecars are active:

| Tool | Source | Description |
|---|---|---|
| `web_fetch` | Core | Fetch a URL and return its content |
| `exa` | Core | Web search via Exa |
| `research` | Core | Deep research with multi-source gathering |
| `screenshot` | Shadow | Capture screen content |
| `ocr` | Shadow | Extract text from images |
| `ghost_snapshot` | Ghost | Desktop screen perception |
| `ghost_type` | Ghost | Simulate keyboard input |
| `ghost_click` | Ghost | Simulate mouse clicks |
| `ghost_record` | Ghost | Record a recipe of actions |
| `spider` | Core | Web scraping |
| `sandbox` | Core | Execute code in a sandboxed environment |
| `notify` | Core | Send notifications |
| `delegate` | Core | Delegate to sub-agents |
| `skills` | Core | List and search agent skills |

See [MCP tools](/docs/mcp/tools) for the complete list and
[Unified tool catalog](/docs/core/unified-tool-catalog) for the full registry.

## Configuration quick reference

### Core environment variables

| Variable | Purpose |
|---|---|
| `RYU_TOKEN` | Node admittance token (required for non-loopback) |
| `RYU_DIR` | Data directory (default `~/.ryu`) |
| `RYU_DEFAULT_AGENT` | Default agent ID (default `ryu`) |
| `RYU_BIND` | Bind address (default `127.0.0.1:7980`) |

### Gateway environment variables

| Variable | Purpose |
|---|---|
| `GATEWAY_MASTER_KEY` | Admin key for config and audit endpoints |
| `OPENAI_API_KEY` | OpenAI provider credentials |
| `ANTHROPIC_API_KEY` | Anthropic provider credentials |
| `GATEWAY_FIREWALL_ENABLED` | Toggle firewall |

See [Environment variables](/docs/start-here/environment-variables) for the full 689-variable
reference.

## LLM-friendly documentation

Ryu's docs are served in machine-readable formats for agent consumption:

| Endpoint | Format | Use case |
|---|---|---|
| `GET /llms.txt` | Plain text index | Discover all pages |
| `GET /llms-full.txt` | Full text dump | Bulk consumption (large) |
| `GET /llms.mdx/docs/{slug}` | Per-page Markdown | Targeted page fetch |
| `GET /docs/{path}.mdx` | Raw MDX source | Raw content access |

These endpoints are designed for agents that need to read documentation programmatically.
