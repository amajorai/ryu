When something goes wrong in Ryu, the first question is *where*: Core, Gateway, plugin runtime,
or the network between them. This page covers the tools and patterns for each layer.

## Debugging Core

### Logs

Core logs to stdout. When running via `bun run dev:core`, logs appear in the terminal. Key log
modules to watch:

| Module | What it covers |
|---|---|
| `core::server` | HTTP request handling, chat stream lifecycle |
| `core::sidecar` | Sidecar spawn, health-check, restart |
| `core::plugin_host` | Turn hook dispatch, sandbox errors |
| `core::plugins::lifecycle` | Install/enable/disable validation |
| `core::workflow` | DAG execution, node failures |
| `core::tool_registry` | Tool search, catalog updates |

Set `RUST_LOG=debug` for verbose output:

```bash
RUST_LOG=debug bun run dev:core
```

### Health endpoint

```bash
curl http://localhost:7980/api/health
```

Returns sidecar status, engine readiness, and loaded plugin count.

### Tracing

Core emits per-run traces accessible at `GET /api/traces`. Each trace records:

| Field | Description |
|---|---|
| `run_id` | Unique run identifier |
| `conversation_id` | Parent conversation |
| `agent_id` | Agent that handled the turn |
| `spans` | Timing spans for each phase (routing, model call, tool execution, hooks) |
| `status` | `completed` / `failed` / `aborted` |

View traces in the desktop Observability page or via the API.

## Debugging the Gateway

### Logs

Gateway logs to stdout. Key modules:

| Module | What it covers |
|---|---|
| `gateway::router` | Model routing, smart classifier |
| `gateway::pipeline` | Request/response pipeline |
| `gateway::firewall` | PII/DLP scanning |
| `gateway::budgets` | Budget enforcement |
| `gateway::evals` | Eval scoring |

```bash
RUST_LOG=debug bun run dev:gateway
```

### Health endpoint

```bash
curl http://localhost:7981/health
```

### Audit log

Every model call is recorded in the audit log. Query it:

```bash
curl http://localhost:7981/v1/audit?limit=10 | jq .
```

Each entry includes model, provider, tokens, latency, and any policy decisions (blocked, downgraded,
etc.).

### Tool execution

The Gateway's tool-search front is at `POST /v1/exec/tool`. To debug tool governance:

```bash
curl -X POST http://localhost:7981/v1/exec/tool \
  -H 'content-type: application/json' \
  -d '{"tool": "web_search", "args": {"query": "test"}}'
```

The response includes the governance decision (allowed, denied, budget-exceeded).

## Debugging plugins and turn hooks

### Plugin status

```bash
curl http://localhost:7980/api/plugins | jq .
```

Returns all installed plugins with their enabled state and declared hooks.

### Sandbox errors

Turn hooks run in a Deno sandbox. If a hook fails, the runtime degrades to `{ kind: "none" }` —
the turn continues without the hook's effect. To see the error:

1. Check Core logs for `plugin_host` entries
2. Look for `is_error: true` in the hook response
3. Verify `deno` is on PATH (`which deno`)

### Common hook failures

| Symptom | Likely cause | Fix |
|---|---|---|
| Hook never fires | Plugin not enabled | `POST /api/plugins/:id/enable` |
| Hook fires but no effect | Hook returns `{ kind: "none" }` | Check `ctx.flags` — toggle may be off |
| `capability not granted` | Missing `permission_grants` in manifest | Add the required grant (e.g. `hook:side-model`) |
| `deno: command not found` | Deno not installed or not on PATH | Install Deno, ensure it's on PATH |
| Hook returns `is_error` | Runtime error in sandbox code | Check logs; simplify hook body |

### Testing hooks locally

Use the bundled fixtures as reference:

```bash
# The double-check fixture
cat apps/core/src/plugin_manifest/fixtures/double-check.plugin.json
```

These are live-tested via `plugin_host::tests::live_*` in the Core test suite.

## Debugging MCP servers

### Check registered servers

```bash
curl http://localhost:7980/api/mcp/servers | jq .
```

### Test a tool call

```bash
curl -X POST http://localhost:7980/api/tools/search \
  -H 'content-type: application/json' \
  -d '{"query": "web search"}' | jq .
```

### Common MCP failures

| Symptom | Likely cause | Fix |
|---|---|---|
| Tool not found | MCP server not registered | Register via `POST /api/mcp/servers` |
| Tool call times out | Server process crashed | Restart the MCP server |
| `ECONNREFUSED` | Server not running on expected port | Check the server's port configuration |

## Debugging ACP agents

### Agent status

```bash
curl http://localhost:7980/api/sidecars | jq .
```

Returns running sidecars with their health status and process info.

### ACP chat issues

ACP agents run as subprocesses. If an agent fails to respond:

1. Check `GET /api/sidecars` for the agent's process status
2. Check Core logs for `sidecar` entries
3. Verify the agent binary is on PATH or downloaded

### Gateway routing for ACP agents

If an ACP agent's model calls bypass the Gateway:

- **BYO OpenAI-compat agents**: Check the per-agent `agent-gateway-routing` toggle is ON
- **Claude Code / Codex**: Verify the passthrough proxy is enabled in `gateway.toml`
- **Gemini / OpenClaw / Hermes**: These agents ignore `OPENAI_BASE_URL` and bypass by design

See [Routing Planes](/docs/gateway/routing-planes) for the full picture.

## Debugging workflows

### Workflow run status

```bash
curl http://localhost:7980/api/workflows/runs | jq .
```

### Common workflow failures

| Symptom | Likely cause | Fix |
|---|---|---|
| Node stuck in `running` | Long-running tool or model call | Check agent status; check timeout config |
| Node fails with `timeout` | Exceeded `timeout_secs` | Increase timeout or optimize the step |
| `While` loop runs forever | Bounded loop hit `max_iterations` | Check loop condition |
| Durable resume fails | Checkpoint corrupted | Check workflow state in DB |

## Debugging the mesh

### Mesh status

```bash
curl http://localhost:7980/api/mesh/status | jq .
```

Returns `enabled`, `reachable`, `up`, `backend`, and `peers` info.

### Common mesh issues

| Symptom | Likely cause | Fix |
|---|---|---|
| `mesh disabled` | `RYU_MESH_ENABLED` not set | Set the env var |
| `unreachable` | Tailscale not running | Start Tailscale: `tailscale up` |
| Auth failure | Token mismatch | Regenerate mesh token |

## Performance debugging

### Engine resource sampling

Core samples engine resources (CPU, memory) per turn. View via:

```bash
curl http://localhost:7980/api/engines/resources | jq .
```

### Gateway latency

The audit log includes per-request latency. High latency can indicate:

- Slow model provider response
- Rate limiting (check `X-RateLimit-*` headers)
- Circuit breaker tripping (check gateway logs for `circuit_breaker`)
- Budget cap hit (check `budget_exceeded` in response)
