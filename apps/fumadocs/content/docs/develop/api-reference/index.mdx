# API reference

Ryu exposes two HTTP surfaces. Both are documented below as **interactive OpenAPI
references** - every operation has its request/response schema, example code in
several languages, and a live "try it" playground.

<Cards>
  <Card href="/docs/develop/api-reference/gateway" title="Gateway API" description="The OpenAI-compatible control layer: chat, image, and audio generation plus routing, evals, audit, and governance. Point any OpenAI client at it by swapping the base URL." />
  <Card href="/docs/develop/api-reference/core" title="Core API" description="The local orchestration backend: chat, agents, conversations, models, skills, MCP tools, spaces/RAG, sidecars, and more. Generated directly from Core's Rust handlers." />
</Cards>

## How these specs are produced

- The **Gateway** spec is hand-authored at `apps/fumadocs/specs/gateway-openapi.yaml`
  (its surface is a stable, OpenAI-compatible superset).
- The **Core** spec is generated from Core's Axum handlers with
  [`utoipa`](https://docs.rs/utoipa) - run `bun run generate:openapi` in
  `apps/core` (or fetch `GET /api/openapi.json` from a running Core), which writes
  `apps/fumadocs/specs/core-openapi.json`.

The reference pages themselves are produced from those specs by
`bun run generate:docs` (Fumadocs OpenAPI), so they never drift from the schema.

## Base address and auth

- **Core** default bind: `127.0.0.1:7980` (override with `--bind=<addr>` or
  `RYU_BIND`). When `RYU_TOKEN` is set, protected routes require
  `Authorization: Bearer <RYU_TOKEN>`; health and auth routes are public.
- **Gateway** default bind: `0.0.0.0:7981` (override with `RYU_GATEWAY_BIND`).
  Authenticate with `Authorization: Bearer <api-key>`; admin endpoints
  (`/v1/config`, `/v1/audit`) additionally require the master key or a loopback peer.
