description: "The shared Rust kernel - manifest validation, the RunnableKind model, and the gateway-mandatory model client"
---

`ryu-sdk` (`crates/ryu-sdk/`) is the **shared Rust kernel** every Ryu language binding builds on. It
is not a Rust-only SDK; it is the one implementation of the parts that must never drift across
languages:

- **Shared local logic** (bound out via FFI): the `plugin.json` manifest / Runnable model + its
  validation (`manifest.rs`, `runnable.rs`) and the gateway egress rules (`gateway.rs`).
- **Transport** (per-language, contract from OpenAPI): the gateway-mandatory model and embedding
  clients (`model.rs`, `embedding.rs`) speaking the gateway's OpenAI-compat API.

For the manifest *format* and the full Runnable kind table, see
[plugin.json manifest](/docs/develop/extensions/plugin-json-manifest) and
[Plugin API](/docs/develop/sdk/plugin-api). For the Runnable *concept*, see
[the Runnable model](/docs/start-here/architecture/runnable-model). This page documents the Rust
surface only.

<Callout type="warn">
  The crate version is `0.0.1`. It is not yet published to crates.io; depend on it by path inside the
  monorepo.
</Callout>

## The Runnable model

`RunnableKind` (`crates/ryu-sdk/src/runnable.rs`) is the union of every executable thing in Ryu. It
mirrors `RunnableKind` in `apps/core/src/runnable/mod.rs` byte-for-byte (same
`#[serde(rename_all = "snake_case")]` wire form), so a manifest authored against this enum
deserializes identically in Core. The eight kinds and their meaning are documented once on
[the Runnable model](/docs/start-here/architecture/runnable-model);
`RunnableKind::as_str()` returns the stable lowercase identifier, and a
`RunnableMeta { id, name, kind }` is the kind-agnostic identity snapshot.

### Per-kind config and validation

A `RunnableEntry` is one entry inside a manifest: identity fields plus an optional typed `config`
blob whose required shape is driven by `kind`. Each kind has a dedicated config struct
(`AgentConfig`, `WorkflowConfig`, `ToolConfig`, `SkillConfig`, `CompanionConfig`, `ChannelConfig`,
`EngineConfig`, `PolicyConfig`).

```rust
pub struct RunnableEntry {
    pub id: String,
    pub name: String,
    pub kind: RunnableKind,
    pub config: Option<serde_json::Value>,
}
```

`validate_runnable(entry)` enforces each kind's contract and returns a descriptive `String` error
when it fails. It mirrors `validate_runnable` in Core's
`apps/core/src/plugin_manifest/schema.rs` (same error substrings, so Core's tests and any binding's
tests agree). The per-kind required-field table is owned by
[Plugin API](/docs/develop/sdk/plugin-api); the Rust call site is the same contract:

```rust
use ryu_sdk::{validate_runnable, RunnableEntry, RunnableKind};

let entry = RunnableEntry {
    id: "tool-web-search".into(),
    name: "Web Search".into(),
    kind: RunnableKind::Tool,
    config: Some(serde_json::json!({ "slug": "web_search" })),
};
validate_runnable(&entry)?; // Ok(()) - tool has a non-empty slug
```

## Manifest validation

`PluginManifest` (`crates/ryu-sdk/src/manifest.rs`) is the in-memory model of a `plugin.json`,
mirroring `PluginManifest` in `apps/core/src/plugin_manifest/mod.rs`.

```rust
pub struct PluginManifest {
    pub id: String,
    pub name: String,
    pub version: String,
    pub runnables: Vec<RunnableEntry>,
    pub permission_grants: Vec<String>,
    pub companion: Option<CompanionSurface>,
}
```

The single entry point for loading an untrusted manifest is `parse_and_validate`, which parses JSON
then runs full validation (id, semver, every Runnable entry):

```rust
use ryu_sdk::PluginManifest;

let manifest = PluginManifest::parse_and_validate(raw_json)?;
for meta in manifest.runnable_metas() {
    println!("{} ({})", meta.name, meta.kind.as_str());
}
```

`validate()` checks three things: `validate_plugin_id` (reverse-domain, no path separators / `..` /
leading-trailing dots - safe as both an identity key and a filesystem directory name), semver via
the `semver` crate, and `validate_runnable` for every entry.

### Loading installed plugins

`plugins_dir()` resolves the scan directory exactly as Core does (`RYU_PLUGINS_DIR`, then legacy
`RYU_APPS_DIR`, then `~/.ryu/plugins`, with a legacy `~/.ryu/apps` fallback). `load_user_plugins()`
scans it for `plugin.json` (or legacy `ryu.json`) manifests and returns only those that pass
id/semver/duplicate validation:

```rust
use ryu_sdk::load_user_plugins;

for manifest in load_user_plugins() {
    println!("{} v{}", manifest.id, manifest.version);
}
```

### JSON Schema export

`ryu_sdk::json_schema::plugin_manifest_schema()` emits the JSON Schema for `PluginManifest` (derived
from the Rust types via `schemars`). This is the "validate everywhere, bind only where it pays" path:
any language can validate a `plugin.json` against this schema without a Rust FFI binding, and the
schema stays in lockstep with the Rust types.

## The gateway-mandatory model client

Every model call routes through the Ryu Gateway. The crate enforces this by construction.

`DEFAULT_GATEWAY_URL` is `http://127.0.0.1:7981` (Core 7980 / Gateway 7981). `resolve_gateway_url()`
reads `RYU_GATEWAY_URL` (falling back to the default); `resolve_gateway_token()` reads
`RYU_GATEWAY_TOKEN`.

`assert_allowed_egress(base_url)` (`crates/ryu-sdk/src/gateway.rs`) rejects known direct-provider
hosts (`api.openai.com`, `api.anthropic.com`, `openrouter.ai`, Google, Cohere, Mistral, Groq,
Together, Replicate, Perplexity) with an `EgressNotAllowed` error. Loopback, the configured gateway,
and private addresses are allowed. Provider credentials belong in the Gateway, never in SDK config.

`ModelClient` (`crates/ryu-sdk/src/model.rs`) POSTs to `{base}/v1/chat/completions`. It is
constructed via `ModelClient::new(model, options)` or the `define_model(model, options)` convenience
factory (mirroring the TS `defineModel`). The egress check runs at construction time, so a
direct-provider base URL fails before any request.

```rust
use ryu_sdk::{define_model, ChatMessage, ModelClientOptions};

let model = define_model("gemma4", ModelClientOptions::default())?;

// Non-streaming:
let result = model.chat(&[ChatMessage::user("hello")]).await?;
println!("{}", result.content);
if let Some(usage) = result.usage {
    println!("tokens: {}", usage.total_tokens);
}

// Streaming (OpenAI-compat SSE, hand-parsed):
use futures_util::StreamExt;
let messages = [ChatMessage::user("write a haiku")];
let mut stream = model.stream(&messages);
while let Some(delta) = stream.next().await {
    let delta = delta?;
    print!("{}", delta.content);
}
```

`ChatMessage` has `system` / `user` / `assistant` constructors. `ChatResult` carries `content`, an
optional `finish_reason`, and optional `Usage`. `ChatDelta` carries an incremental `content` fragment
and an optional `finish_reason` on the final chunk. `ModelError` distinguishes `Egress` (blocked
URL), `Transport` (network failure), and `Http { status, url, body }` (non-2xx from the gateway).

## The gateway-mandatory embedding client

`EmbeddingClient` (`crates/ryu-sdk/src/embedding.rs`) is the retrieval-side sibling of `ModelClient`.
It POSTs to `{base}/v1/embeddings` on the gateway and is constructed via
`EmbeddingClient::new(model, options)` or the `define_embedding(model, options)` factory. The same
`assert_allowed_egress` rule runs at construction, so a direct-provider base URL is rejected before
any request, exactly as for chat.

```rust
use ryu_sdk::{define_embedding, EmbeddingClientOptions};

let embedder = define_embedding("nomic-embed-text-v1.5", EmbeddingClientOptions::default())?;
let result = embedder.embed(&["first text".into(), "second text".into()]).await?;
for vector in result.vectors() {
    println!("dims: {}", vector.len());
}
```

The input is always sent as an array, so one call embeds a single text or a batch. `EmbeddingResult`
holds `embeddings` (one `Embedding { index, vector }` per input, in input order) plus optional
`EmbeddingUsage`, with `vectors()` and `first_vector()` convenience accessors. `EmbeddingError`
mirrors `ModelError` with `Egress`, `Transport`, and `Http { status, url, body }` variants.

## OpenAPI codegen (optional)

The canonical OpenAPI specs are vendored under `crates/ryu-sdk/specs/`
(`core-openapi.json`, `gateway-openapi.yaml`) as the contract. The crate ships a hand-written
`reqwest` client as the shipping transport; an optional `codegen` Cargo feature can generate a client
from the vendored spec.

<Callout type="info">
  `codegen` is **off by default**: the vendored specs are OpenAPI 3.1, which the `progenitor`
  generator does not fully support, so the default build never depends on the generator and can never
  be reddened by a generator change. The spec is still exercised by the `spec_conformance` test.
</Callout>

## Crate API at a glance

```rust
// Re-exported from crates/ryu-sdk/src/lib.rs:
pub use embedding::{ define_embedding, Embedding, EmbeddingClient, EmbeddingClientOptions,
                     EmbeddingError, EmbeddingResult, EmbeddingUsage };
pub use gateway::{ assert_allowed_egress, resolve_gateway_token, resolve_gateway_url,
                   EgressNotAllowed, DEFAULT_GATEWAY_URL };
pub use manifest::{ load_user_plugins, plugins_dir, validate_plugin_id,
                    CompanionSurface, PluginManifest };
pub use model::{ define_model, ChatDelta, ChatMessage, ChatResult, ModelClient,
                 ModelClientOptions, ModelError, Usage };
pub use runnable::{ validate_runnable, AgentConfig, ChannelConfig, CompanionConfig, EngineConfig,
                    PolicyConfig, RunnableEntry, RunnableKind, RunnableMeta, SkillConfig,
                    ToolConfig, WorkflowConfig };
```

## Language bindings

The shared local logic (manifest model + validation + egress rules) is implemented once in this
crate and bound out to other languages, while each target reuses its own first-class HTTP for the
model transport (the vendored OpenAPI spec is the contract). Three binding crates wrap the same
`ryu_sdk::*` functions, so manifest rules, the egress blocklist, and the model/embedding transport
never drift across languages:

| Crate | Target | Mechanism |
|---|---|---|
| `crates/ryu-sdk-napi` | TypeScript / JS | napi-rs native addon (keeps the streaming closure boundary via `ThreadsafeFunction`) |
| `crates/ryu-sdk-ffi` | Go (today) | hand-written raw C-ABI |
| `crates/ryu-sdk-uniffi` | Python / Swift / Kotlin (and C# / Go via third-party generators) | `uniffi-bindgen` generated |

The UniFFI surface (`crates/ryu-sdk-uniffi/src/lib.rs`) exports `validate_plugin_id`,
`parse_and_validate_manifest`, `plugin_manifest_json_schema`, the gateway resolvers,
`assert_allowed_egress`, and blocking `ModelClient` / `EmbeddingClient` objects. The
`uniffi::setup_scaffolding!("ryu_sdk")` namespace makes the emitted package import as `ryu_sdk`
in Python. Bindings are emitted on demand by the `uniffi-bindgen` driver
(`crates/ryu-sdk-uniffi/src/bin/uniffi-bindgen.rs`).

<Callout type="warn">
  Only the **Python** binding is generated by `uniffi-bindgen` and smoke-tested (exit 0) against the
  real Rust kernel. Swift / Kotlin (native UniFFI) and C# / Go (via the third-party
  `uniffi-bindgen-cs` / `uniffi-bindgen-go`) are the same-pipeline follow-on and are not yet verified.
</Callout>

<Callout type="info">
  The UniFFI surface is **blocking, value-in / value-out only**. Streaming chat
  (`ModelClient::stream`) is deliberately omitted because UniFFI has no closure type; it is the
  deferred `ChatSink` callback-interface slice. See `docs/multi-language-bindings-spec.md` for the
  streaming design.
</Callout>

For TypeScript, prefer the published packages over the napi addon directly:
[`@ryuhq/sdk`](/docs/develop/sdk) for authoring and [`@ryuhq/client`](/docs/develop/sdk/client) for
calling a Core node. For any language without a binding, validate manifests against the exported JSON
Schema and call Core's [HTTP API](/docs/develop/api-reference) and the Gateway's
`/v1/chat/completions` directly.
