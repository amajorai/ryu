title: Language bindings (UniFFI)
description: One shared Rust kernel bound out to Python, Swift, Kotlin, C#, and Go via UniFFI. What ships today (Python) and what is planned.
---

Ryu's SDK is split by **layer, not language**. The shared local logic - the `plugin.json`
manifest model plus validation, the gateway egress rules, and the gateway-mandatory model and
embedding clients - lives once in the [`ryu-sdk`](/docs/develop/sdk/rust) Rust kernel. Every
language binding is a thin marshalling shell over that same kernel, so manifest rules, the egress
blocklist, and the transport never drift across languages.

This page covers the **UniFFI** binding path (`crates/sdk/uniffi/`), which generates idiomatic
packages for Python, Swift, Kotlin, and (via third-party generators) C# and Go from one Rust
surface. For the kernel itself see [Rust SDK](/docs/develop/sdk/rust); for the TypeScript client
see [Client SDK](/docs/develop/sdk/client).

<Callout type="warn">
  Only the **Python** binding is generated and smoke-tested today. The UniFFI crate
  (`crates/sdk/uniffi`) and the Python packaging shell (`bindings/python/`) are shipped, and CI
  builds the cdylib, generates the Python module, and runs an exit-0 smoke test against the real
  kernel. **Swift, Kotlin, C#, and Go via UniFFI are planned** - the same pipeline, not yet wired or
  verified. Go ships today through a separate hand-written C-ABI path (`bindings/go/`), not UniFFI.
</Callout>

## Architecture

There is one Rust kernel and three binding shells around it. They all wrap the **same** `ryu_sdk::*`
functions, which is what keeps the rules identical everywhere.

| Crate | Mechanism | Targets | Status |
|---|---|---|---|
| `crates/sdk/napi` | napi-rs native addon | TypeScript / JS | Shipped (keeps the streaming closure via `ThreadsafeFunction`) |
| `crates/sdk/ffi` | hand-written raw C-ABI | raw C, Go (cgo) | Shipped (`bindings/go/ryusdk/ryusdk.go`) |
| `crates/sdk/uniffi` | `uniffi-bindgen` generated | Python (shipped), Swift / Kotlin / C# / Go (planned) | Python shipped |

UniFFI uses **proc-macro mode** (`uniffi::setup_scaffolding!("ryu_sdk")` plus `#[uniffi::export]`),
not a `.udl` IDL. This matches the codebase's "derive the schema from the Rust types" philosophy, so
there is no second source of truth to drift from the Rust signatures. The foreign package is emitted
on demand from the compiled cdylib by the `uniffi-bindgen` driver
(`crates/sdk/uniffi/src/bin/uniffi-bindgen.rs`).

The namespace passed to `setup_scaffolding!` is `ryu_sdk` (not the crate name `ryu_sdk_uniffi`), so
the generated Python package imports as `import ryu_sdk` and the Swift / Kotlin equivalents follow
the same name.

### Blocking surface only

Every UniFFI export is **blocking, value-in / value-out**. UniFFI has no closure type, so streaming
chat (`ModelClient::stream`) is deliberately omitted - it needs a `ChatSink` callback interface,
which is a named future slice. The blocking `chat` and `embed` calls cover Python, Swift, and Kotlin
the same way they cover Go today. See `docs/multi-language-bindings-spec.md` for the deferred
streaming design.

The exported surface (`crates/sdk/uniffi/src/lib.rs`):

- `validate_plugin_id(id)`
- `parse_and_validate_manifest(json)`
- `plugin_manifest_json_schema()`
- `resolve_gateway_url()` / `resolve_gateway_token()`
- `assert_allowed_egress(url)`
- `ModelClient(model, base_url?, token?)` with `.chat(messages)`
- `EmbeddingClient(model, base_url?, token?)` with `.embed(inputs)`

All fallible calls raise a single `RyuError` carrying the kernel's human-readable message.

## Python (shipped)

The Python binding lives in `bindings/python/`. What is committed is the packaging shell
(`pyproject.toml`), the `smoke_test.py` pipeline proof, and a tracked `ryu_sdk/__init__.py`. The
importable `ryu_sdk/` module and the matching compiled library are **generated** by `uniffi-bindgen`
and gitignored.

<Callout type="info">
  The distribution name is `ryu-sdk` (importable as `ryu_sdk`, version `0.1.0`), but it is **not yet
  published to PyPI**. Today you generate it locally or consume the CI artifact; there is no
  `pip install ryu-sdk` yet.
</Callout>

### Generate locally

`sdk/uniffi` is a standalone crate (its own `Cargo.lock`, no workspace), so `cargo` runs from
inside it. From `crates/sdk/uniffi/`:

```sh
# 1. Build the cdylib.
cargo build --release

# 2. Generate the Python module from the compiled library.
cargo run --release --bin uniffi-bindgen -- \
  generate --library target/release/libryu_sdk_uniffi.so \
  --language python --out-dir ../../bindings/python/ryu_sdk

# 3. Copy the compiled library next to the generated module so it loads.
cp target/release/libryu_sdk_uniffi.so ../../bindings/python/ryu_sdk/

# 4. Prove it.
cd ../../bindings/python && PYTHONPATH=. python smoke_test.py
```

On Windows the artifact is `target/release/ryu_sdk_uniffi.dll` (no `lib` prefix, not `.so`); swap the
two `libryu_sdk_uniffi.so` paths for it.

### Minimal usage

```python
import ryu_sdk

# Manifest validation (shared kernel rules - path-traversal-safe ids).
ryu_sdk.validate_plugin_id("io.ryu.ok")          # ok
ryu_sdk.validate_plugin_id("../evil")            # raises ryu_sdk.RyuError

# The gateway egress moat: direct providers are blocked, loopback is allowed.
ryu_sdk.assert_allowed_egress("http://127.0.0.1:7981")    # ok
ryu_sdk.assert_allowed_egress("https://api.openai.com")   # raises ryu_sdk.RyuError

# A gateway-mandatory model client. The egress check runs at construction, so a
# direct-provider base URL fails before any request.
client = ryu_sdk.ModelClient("gemma4", "http://127.0.0.1:7981", None)
result = client.chat([
    ryu_sdk.ChatMessage(role="user", content="Write a haiku about Rust."),
])
print(result.content)
if result.usage:
    print("tokens:", result.usage.total_tokens)
```

`ModelClient` and `EmbeddingClient` default `base_url` and `token` to `RYU_GATEWAY_URL` and
`RYU_GATEWAY_TOKEN` when passed `None`. Streaming is not exposed (see [Blocking surface
only](#blocking-surface-only)); for token-by-token output today, use the
[TypeScript](/docs/develop/sdk/client) path or call the gateway's `/v1/chat/completions` directly.

## Swift, Kotlin, C#, and Go (planned)

These targets are the same pipeline as Python - one cdylib, regenerated per language - and are not
yet wired or verified:

| Language | Generator | Provenance | Status |
|---|---|---|---|
| Swift | `uniffi-bindgen --language swift` | built into UniFFI | Planned (needs an Apple CI runner + xcframework packaging) |
| Kotlin | `uniffi-bindgen --language kotlin` | built into UniFFI | Planned (needs JNA + a JVM packaging story) |
| C# | `uniffi-bindgen-cs` | third-party | Planned (generator lags UniFFI minors) |
| Go | `uniffi-bindgen-go` | third-party | Planned; Go ships today via the cgo C-ABI path instead |

Python, Swift, and Kotlin generators are first-party to UniFFI, so they move in lockstep with the
`uniffi` pin in `crates/sdk/uniffi/Cargo.toml`. The C# and Go generators are maintained
out-of-tree and pinned to specific UniFFI minors, so bumping the `uniffi` dependency is a coordinated
cross-language event rather than a routine update.

Go already works today through hand-written cgo over `crates/sdk/ffi` (`bindings/go/`), with no
generator-lag risk. A migration to `uniffi-bindgen-go` would buy one source surface at the cost of
inheriting that lag, so the cgo path stays as-is for now.

For any language without a binding, validate manifests against the exported JSON Schema
(`plugin_manifest_json_schema()`) and call Core's [HTTP API](/docs/develop/api-reference) and the
gateway's `/v1/chat/completions` directly - both are plain JSON.
