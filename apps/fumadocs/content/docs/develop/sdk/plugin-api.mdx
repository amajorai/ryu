This is the canonical reference for the `manifest.json` manifest: every field, the **eight**
Runnable kinds and their required config, the install/enable lifecycle and its endpoints, Gateway
grant validation, ed25519 manifest signing, and the activation runtime.

This page **owns** the manifest schema (the 8-kind table lives only here). For the conceptual model
see [Runnable / Session / App](/docs/start-here/architecture/runnable-model); for a guided worked
example see the [Plugin manifests tutorial](/docs/develop/extensions/plugin-json-manifest).

<Callout type="warn">
  The VS-Code-style runtime is **partly live, not purely design-only**. The manifest carries a
  `contributes` block, `activation_events`, and an `engines.ryu` version pin
  (`apps/core/src/plugin_manifest/mod.rs`), and an activation runtime (`apps/core/src/runnable/mod.rs`)
  wakes a plugin's runnables when its events fire - but only the `onStartup` event is fired today.
  Still design-only: the desktop extension host, the in-process plugin API, and live UI/Core
  modification by a plugin. See [activation runtime](#plugin-activation-runtime) and
  [planned runtime](#planned-runtime) below (`docs/rfc-plugin-runtime.md`).
</Callout>

## What a plugin is

A **plugin** is a bundle of one or more **Runnables** plus the **permission grants** they need,
packaged so Core can install, enable, disable, and update it as a unit. The canonical manifest file
name is `manifest.json`; the legacy `ryu.json` name is still read for back-compat (the apps -> plugins
rename, #457). The SDK's bundled output is `plugin.bundle.json`.

The authoritative schema lives in Core at `apps/core/src/plugin_manifest/{mod.rs,schema.rs}`, mirrored
in Rust at [`crates/sdk/core/src/manifest.rs`](/docs/develop/sdk/rust) and in TypeScript at
`packages/sdk/src/manifest.ts` (a Zod `PluginManifestSchema`). All three deserialize the same bytes.

## Manifest schema

### Top-level fields

```json
{
  "id": "io.ryu.headroom",
  "name": "Headroom",
  "version": "1.0.0",
  "runnables": [],
  "permission_grants": [],
  "companion": null
}
```

| Field | Type | Required | Rule |
|---|---|---|---|
| `id` | string | yes | Reverse-domain id; must contain a `.`, only `a-z A-Z 0-9 . - _`, no `..`, no leading/trailing `.`, no leading `-`, max 128 bytes. Doubles as a filesystem directory name. |
| `name` | string | yes | Human-readable display name. |
| `version` | string | yes | Valid semver (`MAJOR.MINOR.PATCH`). A non-semver version is rejected. |
| `runnables` | `RunnableEntry[]` | no (default `[]`) | The Runnables this plugin bundles. May be empty for a surface-only plugin. |
| `permission_grants` | `string[]` | no (default `[]`) | Grant scopes the plugin declares it needs. Declarations only - the Gateway enforces. |
| `companion` | `CompanionSurface \| null` | no | Optional in-desktop overlay or sidebar panel. |

`validate_plugin_id` (`apps/core/src/plugin_manifest`, mirrored in `ryu-sdk`) enforces the id rules
above so an `id` can never escape `~/.ryu/plugins/<id>/`.

### RunnableEntry

Each entry in `runnables` is a `RunnableEntry`: identity fields plus an optional typed `config` whose
required shape is driven by `kind`.

```json
{
  "id": "io.example.researcher",
  "name": "Researcher",
  "kind": "agent",
  "config": { "system_prompt": "You are a careful researcher.", "model": "gemma4", "tools": ["web_search"] }
}
```

Core's schema defines **eight** kinds (`apps/core/src/runnable/mod.rs`). Each has a required-field
contract enforced by `validate_runnable` (`apps/core/src/plugin_manifest/schema.rs`):

| `kind` | What it is | Config required? | Required field(s) |
|---|---|---|---|
| `agent` | Multi-turn model loop with swappable model/tools/persona slots | optional (all default) | none |
| `workflow` | A DAG of typed nodes run by Core's workflow engine | yes | `entry` |
| `tool` | A callable function (wraps an MCP tool slug) | yes | `slug` |
| `skill` | An Agent Skill (versioned, shareable) | yes | `skill_id` (non-empty) |
| `companion` | An in-desktop overlay or sidebar panel | yes | `label` |
| `channel` | A messaging-platform bot adapter | yes | `platform` |
| `engine` | A model/inference backend binding | yes | `engine_type` (non-empty) |
| `policy` | A Gateway-enforced rule fragment | yes | `policy_type` (non-empty) + `definition` |

<Callout type="warn">
  The TypeScript SDK's `RunnableKindSchema` (`packages/sdk/src/manifest.ts`) enums only the four most
  common kinds (`agent`, `workflow`, `tool`, `skill`). Core validates all eight. To author the other
  four (companion / channel / engine / policy), write the `manifest.json` directly rather than relying
  on the TS type. The Rust kernel (`sdk/core`) covers all eight.
</Callout>

### CompanionSurface

```json
{ "label": "Research Panel", "icon": "search", "shortcut": "ctrl+shift+r" }
```

`label` is required; `icon` and `shortcut` are optional and resolved by the desktop shell. Rendering
the panel is part of the [planned runtime](#planned-runtime), not yet executed.

## Lifecycle

Core decides **what runs**: it persists install/enable/disable/update state and compares semver.
The Gateway decides **what is allowed**, and `enable` is the only transition that calls it. The full
state machine, the on-disk record, and verify-on-install live in
[App manifest and lifecycle](/docs/core/app-manifest-lifecycle); this page lists only the endpoints
and the per-operation contract a manifest author needs.

| Operation | Effect | Gateway involved? |
|---|---|---|
| install | Creates a `PluginRecord` with `enabled = false`; validates semver first. Fails if already installed. | no |
| enable | Validates declared grants via the Gateway, then flips `enabled = true` and stores the approved grants. **Fails closed** if the Gateway denies a grant or is unreachable. | yes |
| disable | Flips `enabled = false` and clears the approved grants. | no |
| update | Sets a new version. **Refuses a downgrade** (new < installed) unless `force = true`. A same-version update is a no-op. Does *not* re-enable a disabled plugin. | no |

### Endpoints

The canonical routes are `/api/plugins*` (`apps/core/src/server/mod.rs`). The deprecated `/api/apps*`
aliases point at the same handlers for one release of back-compat.

| Method + path | Purpose |
|---|---|
| `GET /api/plugins` | List loaded manifests (built-ins + user-installed). |
| `GET /api/plugins/catalog` | Browse installable plugins from the remote registry. |
| `POST /api/plugins/install` | Install by fetching a `manifest.json` from a URL, validate, write to `~/.ryu/plugins/`, hot-reload. |
| `POST /api/plugins/reload` | Re-scan built-ins + `~/.ryu/plugins/*/manifest.json`. |
| `POST /api/plugins/:id/install` | Record an already-known plugin as installed (disabled). |
| `POST /api/plugins/:id/enable` | Validate grants via the Gateway, then enable. |
| `POST /api/plugins/:id/disable` | Disable and clear approved grants. |
| `POST /api/plugins/:id/update` | Update the installed version (body may carry `force`). |

Install hot-reloads Core's manifests, so a newly installed plugin appears in `GET /api/plugins`
immediately with **no restart needed**.

## Grant validation

`permission_grants` are scope strings the plugin declares (e.g. `mcp.tools`, `model.chat`,
`memory.read`). They are **declarations only**. On `enable`, Core calls the Gateway's
`POST /v1/grants/validate` with the declared grants (`apps/gateway/src/api/governance.rs`):

```json
// request
{ "app_id": "io.ryu.headroom", "grants": ["mcp.tools", "model.chat"] }
// response
{ "approved": ["mcp.tools", "model.chat"], "denied": [], "all_approved": true }
```

The Gateway checks each grant against an allowlist (`apps/gateway/src/governance/mod.rs`). The
built-in default allowlist covers the common capability scopes - `mcp.tools`, `tools.read`,
`tools.invoke`, `memory.read`, `memory.write`, `spaces.read`, `spaces.write`, `files.read`,
`model.chat`, `model.embed`, `network.fetch` - and is overridable via
`RYU_MARKETPLACE_GRANT_ALLOWLIST`. A grant outside the allowlist lands in `denied`, which blocks the
enable.

**Fail-closed behavior** (`apps/core/src/plugins/lifecycle.rs`):

- Any `denied` grant -> the plugin stays disabled (`GrantsDenied`).
- Gateway unreachable or non-2xx -> the plugin stays disabled (`GatewayUnreachable`). Core applies
  no inline policy; it never silently allows.
- Empty grant list -> trivially approved (nothing to validate).

For environments where the Gateway is not yet running, `RYU_STUB_GRANT_VALIDATION=1` makes Core's
side allow-all (logged at WARN so it is always visible). This is an explicit seam, not a default.

## Manifest signing (ed25519)

Publishing a plugin to the Ryu Marketplace is a governed action, so the Gateway owns the signing key
and the sign/verify endpoints. The algorithm is **ed25519**, and both sign and verify
**canonicalize** the manifest (recursively sorted object keys) before hashing so re-serialization
across the stack (TS -> Mongo -> Core JSON round-trips) never changes the signed bytes.

The full signing ceremony - loopback-only signing, the trust-anchor rules, and where the key resolves
from - is documented in [Gateway governance](/docs/gateway/governance). The manifest author only needs
the three endpoints and the verify-on-install contract:

| Method + path | Purpose | Access |
|---|---|---|
| `POST /v1/manifests/sign` | Sign a manifest; returns `{ algorithm, signature, public_key }`. | **Loopback only** (and neutralized under mesh). The only legitimate caller is the co-located control-plane server on publish. |
| `POST /v1/manifests/verify` | Verify a signature; returns `{ valid }`. | Open (read-only check over caller data). |
| `GET /v1/manifests/pubkey` | The public verifying key; returns `{ algorithm, public_key }`. | Open (no secret). |

### Verify-on-install

When Core installs a Ryu Marketplace item it verifies the manifest signature **before any install
work** (`apps/core/src/catalog_source/sources.rs`):

- A pinned, operator-configured `RYU_MARKETPLACE_PUBLIC_KEY` is the trust anchor when set; otherwise
  no key is sent and the Gateway verifies with its own resident key.
- The manifest's own `signingPublicKey` field is **never** the trust anchor (that would be
  self-attested signing - a tamperer ships their own key and a matching signature and "passes"). It
  is for display / rotation cross-checks only.
- Signature present + invalid -> **reject** (the tamper case).
- Signature absent -> allow, logged (unsigned first-party seed items).
- Gateway unreachable while a signature is present -> **reject (fail closed)**, so a tampered manifest
  cannot install merely because the verifier is down.

## Plugin activation runtime

The manifest carries three VS-Code-style fields that drive a now-live activation runtime. They live
on `PluginManifest` (`apps/core/src/plugin_manifest/mod.rs`) and are validated by the loader at load
time.

### Activation fields

| Field | Type | Default | Rule |
|---|---|---|---|
| `contributes` | `Contributes \| null` | absent | A declare-by-id block naming which of the manifest's `runnables` contribute to each surface (`commands`, `tools`, `agents`, `workflows`, `policies`). Every referenced id **must** exist in `runnables` - the loader cross-validates and rejects a typo at load. |
| `activation_events` | `string[]` | `[]` | Lazy-activation tokens. Recognised: `"*"` (eager), `"onStartup"`, `"onChat"`, `"onCommand:<id>"`. An **empty** list means **eager** (every existing manifest keeps activating on enable). |
| `engines` | `{ "ryu": string } \| null` | absent | A semver **requirement** the running Core version must satisfy (e.g. `">=0.3.0"`). An unparseable or unsatisfied requirement makes the loader **reject** the manifest. |

A `contributes` entry is declare-by-id, not a second copy of the runnable:

```json
{
  "runnables": [
    { "id": "tool-x", "name": "Tool X", "kind": "tool", "config": { "slug": "web_search" } }
  ],
  "contributes": { "tools": [ { "id": "tool-x", "title": "Search the web" } ] },
  "activation_events": ["onStartup"],
  "engines": { "ryu": ">=0.0.1" }
}
```

The version pin is gated against `core_version()` (`apps/core/src/plugin_manifest/mod.rs`), derived
from the Core crate version - the single version of record.

### How activation fires

The activation contract is in `apps/core/src/runnable/mod.rs`:

- `RunnableRegistry::register_active(manifest, fired_events)` activates a plugin's runnables **only**
  if it should wake. `manifest_should_activate` returns `true` for an eager plugin (empty
  `activation_events`), for a `"*"` declaration, or when any declared event is in `fired_events`.
- A process-global `FIRED_ACTIVATION_EVENTS` set tracks which events have fired. It is monotonic, so a
  plugin enabled **after** an event still sees it via the snapshot.
- `fire_activation_event(state, event)` (`apps/core/src/server/mod.rs`) is the live driver: it records
  the event, then runs every **enabled** plugin through `register_active` against the new snapshot.
- `onStartup` is fired once from `apps/core/src/main.rs` after `ServerState` exists.
- The `enable` lifecycle path uses the same `register_active` against the live fired set, so enable and
  event-driven activation share one contract (an eager plugin activates immediately on enable; a gated
  plugin is held back until its event fires).

<Callout type="warn">
  **Only `onStartup` is actually fired today.** The driver, the fired-event set, and the gated
  dispatch are real and unit-tested, but `onChat` / `onCommand:<id>` / `onRunnable:<id>` are never
  fired from the chat or command-palette paths yet - that wiring is a documented follow-on. A plugin
  declaring those events will not wake until it lands.
</Callout>

### Gateway policy runnables

A `policy`-kind runnable is the generalization of the headroom-compression pattern
(`apps/core/src/sidecar/gateway_policy.rs`): a Policy is an on/off switch that flips a process-global
flag, and `gateway_spawn_env` (`apps/core/src/sidecar/gateway.rs`) re-reads that flag on the next
gateway spawn. `apply_policy` (`apps/core/src/server/mod.rs`) dispatches on each Policy runnable's
`policy_type` when the plugin is enabled or disabled:

| `policy_type` | Effect | Gateway respawn? |
|---|---|---|
| `compression` | Flip the headroom flag and start/stop the headroom proxy. | yes |
| `firewall` | Force the gateway firewall on (flag only - the Gateway enforces). | yes |
| `routing` | Force smart (classifier) routing on (flag only). | yes |
| `sandbox` | Toggle the Core wasmtime sandbox availability (an in-process flag, no gateway). | no |

The gateway-affecting arms set a `gateway_dirty` flag and `apply_policy` calls `gateway.refresh()`
**once** at the end, so enabling several gateway policies at once is one restart, not N. An unknown
`policy_type` is a logged no-op. `main.rs` seeds the flags from each plugin's persisted state at
startup, so a gateway restart never silently reverts what a Policy plugin set.

<Callout type="info">
  Per the Core-vs-Gateway rule, the Core Policy handler is **validate-only**: Core decides whether a
  feature is active for this install, the Gateway owns enforcement (firewall scanning, smart routing).
  The rich policy *definition* - firewall pattern set, routing model map - stays in the Gateway's
  `/v1/config`, not the flag. Like all routing config, a Policy change takes effect on the next
  gateway restart. See [Gateway governance](/docs/gateway/governance).
</Callout>

The built-in Policy plugins (`io.ryu.firewall`, `io.ryu.routing`, `io.ryu.sandbox`, `io.ryu.headroom`)
are Core-tier and opt-in. Their dev seed env vars (`GATEWAY_FIREWALL_ENABLED`,
`GATEWAY_SMART_ROUTING_ENABLED`) default off.

### External-runtime provisioning (scaffold)

A plugin may declare an external language runtime - today a Python venv with pip dependencies - via
an optional `runtime` block, modeled on the `apps-store/voice/` precedent. The provisioner is
`apps/core/src/sidecar/external_runtime.rs`: it derives OS-correct venv paths
(`.venv/Scripts/python.exe` on Windows, `.venv/bin/python` elsewhere) and a best-effort `provision()`
that creates the venv and runs `pip install`. The enable path calls `provision_external_runtime`
(`apps/core/src/server/mod.rs`), which spawns the work so a slow `pip install` never blocks the enable
response.

<Callout type="warn">
  External-runtime provisioning is **scaffold**: only `kind: "python"` is provisionable, asset fetch
  through the DownloadCenter is not wired yet, and provisioning is gated to **Core-tier** plugins
  (running `pip install` from a manifest is a network plus arbitrary-code surface, so a Community
  plugin must reach it through a Gateway grant - that grant path is the documented follow-on). Live
  provisioning is not exercised here; the pure path-derivation and pip-args logic are unit-tested.
</Callout>

## Planned runtime

The activation runtime above is partly live, but the following are **still not implemented**. They
are design-only; do not build against them as if they exist. The RFC is `docs/rfc-plugin-runtime.md`
(epic #425 P3, feeding #444 / #446 / #443):

- **Contribution points that mount real UI.** The `contributes` block declares and cross-validates by
  id today (see [above](#activation-fields)), but no surface yet consumes a contribution to register a
  route, command, or panel.
- **A desktop extension host and an in-process plugin API.** The activation runtime fires events and
  activates a plugin's Core runnables; there is no host that hands a plugin an in-process API or
  renders its UI.
- **Runtime UI/Core modification** by a plugin (route rewrites, panels, in-process hooks, sandboxing
  of third-party code).
- **Firing `onChat` / `onCommand` / `onRunnable`.** Only `onStartup` fires today.

What exists today is the install-time and activation layer: a validated `manifest.json`, the lifecycle
above, Gateway grant validation, signing/verification, the `onStartup` activation runtime, and Gateway
policy runnables. A plugin that wants to govern egress declares a `policy` Runnable that the Gateway
activates on install; every model call a plugin causes still routes through the Gateway. The runtime
grants capabilities; the Gateway decides what is allowed, shared, measured, and paid for.
