Both `apps/native` (Expo / React Native) and `apps/desktop` (Tauri) are **thin GUIs over the same
Core HTTP API** (`:7980`, routed through the active node). So parity is mostly a shared client layer
plus thin screens, not 28 reimplementations. The full status lives in `docs/mobile-parity.md`; this
page summarizes it.

## The leverage: packages/core-client

Desktop's `apps/desktop/src/lib/api/*.ts` modules split into two groups:

- **Pure Core clients** (roughly 33 modules) that only import `./client.ts` (`request(target,
  path)`) plus types. Platform-agnostic.
- **Coupled modules** (billing, channels, credits, marketplace, seller, teams-billing, secrets,
  managed-nodes, update, shadow) that reach `localStorage`, `@tauri-apps/api`, or the control plane.
  These stay desktop-only.

The pure modules are extracted into **`packages/core-client`**: a platform-agnostic `client.ts`
(`ApiTarget` / `request` / `apiUrl` / `makeHeaders` plus an injectable `buyerTokenProvider`) plus the
pure modules copied **verbatim** from desktop. Mobile imports them by subpath, for example
`import { fetchAgents } from "@ryuhq/core-client/agents"`, and adapts its node store to `ApiTarget`
through `apps/native/lib/api-target.ts` (`toTarget`).

Why verbatim copies matter:

- The copies are **byte-identical** to desktop's `lib/api` (only the leading path comment and one
  `ModelOption` type import differ), so request and response **shapes cannot silently drift at
  birth**, and the package carries the same accepted lint baseline desktop's `lib/api` already
  commits with.
- The package's `tsc --noEmit` is clean.
- **Follow-up, not done yet:** re-point desktop's pure `lib/api` modules to re-export from
  `@ryuhq/core-client` so there is one source. Today the package is the canonical seed and desktop
  still has its own copies.

<Callout type="info">
  The package now lives under the `@ryuhq` npm scope (commit `6f1662bb`), so imports read
  `@ryuhq/core-client`. Some older notes still say `@ryu/core-client`.
</Callout>

## Status at a glance

Across the 28 desktop pages: the chat path and the data-list screens are working against Core, a set
of control-plane and deep-edit screens are pending, and a group of desktop-only surfaces are excluded
on purpose (Tauri IPC, browser extensions, first-run downloads, and so on).

| Area | Mobile | State |
|---|---|---|
| Chat (`/api/chat/stream`, agents, `/btw`) | `ai.tsx` | Working |
| Marketplace | `marketplace.tsx` | Working |
| Monitors (plus Expo push tokens) | `monitors.tsx` | Working |
| Settings (preferences) | `settings.tsx` | Working |
| Agents, Skills, Models, Workflows, Schedules, Spaces, Tools, Apps | shipped this pass | Working (lists) |
| Services, Meetings | shipped this pass | Working (viewer / read-only) |
| AgentEditPage (model / routing / skills / Composio) | remaining | Pending |
| Channels, Credits (control plane) | remaining | Pending (need session token wired into `setBuyerTokenProvider`) |
| Gateway config (routing / firewall / budget) | remaining | Pending |
| Timeline (Shadow frames), SpaceDocEditor | remaining | Pending (viewer) |
| Calendar, Companion, Extensions, FileEditor, Fleet, Onboarding | not applicable | Excluded (desktop-only) |

## Remaining for full parity

1. Re-point desktop `lib/api` to `@ryuhq/core-client` (single source).
2. Control-plane screens (Channels, Credits): wire the Better-Auth / Expo session token into
   `setBuyerTokenProvider` plus a control-plane target.
3. AgentEditPage, the deepest screen (model / routing / skills / Composio editing).
4. Gateway config surface (routing / firewall / budget) inside Settings.
5. Viewer-only: Timeline (Shadow frames) and SpaceDocEditor (read).
6. On-device build plus a smoke test of every new screen against a live Core node.

## Verification ceiling

Per screen, the request shapes are matched to desktop's, and the package and the new native files
type-check cleanly (the only `tsc` noise is the repo-wide import-extension baseline that metro
resolves, identical to existing committed native files). What is **not** confirmed here is the live
round-trip: the screens need a `bunx expo install` plus an Expo / metro **device or simulator build**
to exercise, and the repo's existing native features carry the same caveat. On-device inference is not
wired.
