title: "Predictive Typing"

---

Predictive typing suggests the next few words as you type and lets you accept them with **Tab**.
It runs in two places: everywhere on your machine (any text field) and inside Spaces documents (the
editor copilot). Both ask a model for a continuation, and every request routes through the Gateway
like any other model call.

<Callout type="warn">
  The system-wide overlay is **Windows-first**, **opt-in**, and ships **off**. It needs a real
  display to runtime-verify, so treat it as built-but-unverified. Turn it on in
  <strong>Settings → Plugins</strong> before anything appears.
</Callout>

## Turn it on

Predictive typing everywhere is a built-in **plugin** - a single on/off switch, not a pile of
toggles. It is Core-tier and ships disabled (it is not in `CORE_DEFAULT_ON`), so you opt in once:

<Steps>
<Step>Open **Settings → Plugins** and enable **Predictive Typing**.</Step>
<Step>A **Predictive typing** tab appears under Settings once the plugin is on. Open it to tune the
model, the per-app allowlist, and the debounce.</Step>
<Step>Start typing in any allowed app. When a suggestion appears inline as dim ghost text, press
**Tab** to accept it or keep typing to dismiss it.</Step>
</Steps>

The plugin is the one source of truth for on/off. The three older duplicate switches (in Core, in
the desktop settings tab, and in the native overlay itself) were removed, so enabling the plugin is
the only thing that starts predictions. To stop it everywhere, disable the plugin again.

## The predict brain (Core)

The suggestions come from Core, not the overlay. Core owns the model call:

```
POST /api/predict/complete
GET  /api/predict/config
PUT  /api/predict/config
```

`POST /api/predict/complete` takes the text before your cursor and returns a continuation
(`apps/core/src/server/predict_api.rs`). It reuses the same `call_side_model` path as `/btw`, so the
prediction goes through the Gateway and is governed like every other model call. The refusal check
runs first: with the plugin disabled, Core returns an empty suggestion instead of calling a model.

Nothing about the model is hardcoded. Core resolves it in order - the picked agent's bound
chat model, then `predict-config.model`, then the `RYU_PREDICT_MODEL` / `RYU_DEFAULT_LLM_MODEL` env
vars, then the built-in default. Leave the model blank in the settings tab to use the system default.

Core also enforces privacy server-side (`apps/core/src/predict/mod.rs`): password and secure fields
are on a denylist and are never read, and the per-app allowlist is matched on the app basename. The
denylist always wins, whatever you put in the allowlist.

## The system-wide overlay

The inline ghost text is drawn by a separate companion, `apps/predict/` - a Win32 / UIA process that
watches the caret, debounces, calls `POST /api/predict/complete`, and paints a click-through overlay
at the cursor. A low-level keyboard hook (`apps/predict/src/hook.rs`) swallows **Tab** only while a
suggestion is showing, so Tab still tabs normally when there is nothing to accept.

Use the **Predictive typing** settings tab to shape it:

- **Agent / model** - pick an agent to use its own model, or choose Default model and set one (or
  leave it blank for the system default).
- **App allowlist** - one app name per line, or empty to allow every app. Secure fields stay
  excluded regardless.
- **Debounce** - how long after you stop typing before a suggestion is requested (0 to 5000 ms).

## The in-editor copilot

Inside a Spaces document, the editor has its own ghost-text completion (PlateJS). This is a
**separate surface** from the system-wide overlay, and it is always available - it is not gated by
the predict plugin. It is non-streaming: Core returns the completion as plain JSON and the editor
inserts it inline.

Configure the model behind it under **Settings → Editor & Embeddings → Inline AI editing**, not on
the Predictive typing tab. That same card drives autocomplete inside Spaces documents.

<Callout type="info">
  Both surfaces route through the Gateway, so your routing rules, budgets, and firewall apply to
  predictions exactly as they do to chat. See [The Gateway](/docs/gateway).
</Callout>
