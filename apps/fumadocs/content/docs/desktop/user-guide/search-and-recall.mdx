title: "Search & Auto-Recall"
description: "Search past conversations semantically and let agents auto-recall relevant facts on every turn"
---

Ryu indexes your chat history so you can find earlier discussions by meaning, not just exact words, and so an agent can pull the relevant bits of past chats into context automatically. There are two surfaces:

- **Search** - ask an agent to find earlier messages with the `search_conversations` tool.
- **Auto-recall** - a per-node toggle that injects relevant past messages and memory facts into every turn for you.

Both run on the local embedder (the `llamacpp-embed` sidecar on :8081 by default), so your chat content never leaves the node to be searched. The index itself is internal to Core: see [Memory & message index](/docs/core/memory) for how it is stored and kept encrypted.

<Callout type="warn">
Both surfaces need a running embedder. Out of the box Core uses a local hashing embedder, but real semantic quality needs the `llamacpp-embed` sidecar on :8081 (downloaded during onboarding). If the embedder is down, search reports the index as unavailable and auto-recall fails open inside a 4s timeout, so a missing embedder never blocks a chat turn. This was verified network-free; live semantic quality across a running Core is not yet exercised in CI.
</Callout>

## Search your past conversations

Searching is done by an agent, through the built-in `search_conversations__search` tool (`apps/core/src/sidecar/mcp/search_conversations.rs`). Just ask in plain language:

```
Search our past chats for what we decided about the database schema
```

The tool runs a semantic search across your stored messages and returns matching snippets, each with the conversation id, the role, a timestamp, and a relevance score. Because it rides the unified tool path, it works on both the ACP and OpenAI-compatible chat planes, appears in `GET /api/tools/search`, and is allowlist-gated plus audited like any other tool.

<Callout type="info">
The tool is allowlist-gated. If an agent reports it cannot search your history, grant it the `search_conversations__search` tool in the agent's tools list. See [Tools & connections](/docs/desktop/user-guide/tools-and-connections).
</Callout>

### What the tool accepts

| Argument | Type | Default | Meaning |
|---|---|---|---|
| `query` | string | required | Natural-language search query |
| `limit` | integer (1-20) | 5 | Maximum number of matching snippets to return |
| `conversation_ids` | string array | all | Restrict the search to specific conversations; omit to search everything |

Snippets are truncated to keep tool output compact, so a long matched message comes back shortened. To read the full message, open the conversation the snippet points to.

## Auto-recall

Auto-recall does the searching for you. On every chat turn, Core semantically retrieves relevant past chat messages (excluding the current conversation) plus relevant long-term memory facts, and merges them into the system context before the model runs. It is the same injection seam skills use, so both chat planes inherit it.

Auto-recall is **on by default** and is persisted in Core, not in your client, so it applies to every chat turn the node serves rather than just the device you toggled it on.

### Turn it on or off

Open **Settings -> Memory** and use the auto-recall switch (`MemoryTab.tsx`). The toggle writes the `auto-recall-enabled` preference (`apps/desktop/src/lib/api/preferences.ts`).

The underlying knobs:

| Knob | Where | Default | Effect |
|---|---|---|---|
| `auto-recall-enabled` | Settings -> Memory, or env `RYU_AUTO_RECALL_ENABLED` | on | Master switch for per-turn recall |
| `auto-recall-top-k` | preference key | implementation default | Caps how many recalled snippets are injected per turn |

<TryInRyu page="settings" />

### How it stays clean

Auto-recall only adds what the existing recency window missed. Memory facts injected by recall are deduplicated by fact id against the facts the recency path already added, so a fact never appears twice. The recency path is unchanged and remains the fail-open fallback, so disabling auto-recall does not lose your long-term memory entirely. The full mechanism lives on one page: [Memory & message index](/docs/core/memory).

<Callout type="info">
Auto-recall is a separate setting from your long-term memory facts. Turning recall off stops the per-turn semantic injection but leaves your stored facts and the recency-based fallback intact.
</Callout>
