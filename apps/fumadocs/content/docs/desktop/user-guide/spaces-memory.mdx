Ryu gives agents two kinds of grounding. A **Space** is a document store you build and attach to
an agent so its answers cite your own material. **Long-term memory** is what an agent records
about you over time, used as background on every reply. Both run locally in Core, and both are
swappable defaults rather than locks.

This page is the how-to. For how retrieval is scored and assembled, see
[Spaces & RAG](/docs/core/spaces-rag); for how memory is captured, encrypted, and recalled, see
[Memory](/docs/core/memory).

## Create a Space and add documents

A Space is a vector store backed by sqlite-vec inside Core. To build one:

1. Open the **Spaces** page in the desktop app.
2. Create a Space and give it a name. This calls `POST /api/spaces`.
3. Add content. Each document is embedded on ingest (`POST /api/spaces/:id/documents`, the
   `spaces.ingest_document` path) so it can be retrieved by meaning, not just keyword.
4. Or start a blank **page** (`POST /api/spaces/:id/pages`) and write notes directly in the
   editor.

| Action | Route |
|---|---|
| List Spaces | `GET /api/spaces` |
| Create a Space | `POST /api/spaces` |
| Ingest a document | `POST /api/spaces/:id/documents` |
| Create a blank page | `POST /api/spaces/:id/pages` |
| Search a Space | `POST /api/spaces/:id/search` |
| Delete a Space | `DELETE /api/spaces/:id` |

## Edit documents in the PlateJS editor

Open any document to edit it in Ryu's Notion-style PlateJS editor
(`apps/desktop/src/pages/SpaceDocEditorPage.tsx`, route `/spaces/:spaceId/doc/:docId`). Saving
sends `PUT /api/spaces/:id/documents/:doc_id`, which re-embeds the document so retrieval always
reflects the latest text. Delete a single document with
`DELETE /api/spaces/:id/documents/:doc_id`.

<Callout type="info">
  Meeting notes reuse this exact surface. When a meeting is finalized, Core writes the notes and
  transcript as a markdown document into an auto-created "Meetings" Space, so you view and edit
  them in the same PlateJS editor. See [Meeting notes](/docs/core/meeting-notes).
</Callout>

## Beyond pages: databases, whiteboards, and a wiki graph

A Space holds more than ingested documents and markdown pages. Three richer document kinds live
alongside them, and all of them feed the same retrieval so an agent can still ground its answers in
them.

### Databases

A database is a Notion-style data grid (`kind="database"`), created with
`POST /api/spaces/:id/databases`. Columns are typed (text, number, checkbox, date, url, select, and
multi-select), each row can open as its own page, and you can view the same data as a **table**, a
**board / kanban** grouped by a select column, a **gallery**, or a **list** - views are a display
layer, so switching never mutates the data (`apps/desktop/src/components/spaces/views/ViewBar.tsx`,
`BoardView.tsx`). For retrieval, Core flattens each row's labels and cell text before embedding, so
a database is searchable the same way a page is.

### Whiteboards

A whiteboard is an Excalidraw canvas stored as a third kind (`kind="whiteboard"`), created with
`POST /api/spaces/:id/whiteboards`. You can also generate one from a prompt: `POST
/api/whiteboard/generate` runs a single side-model call, which is Gateway-routed like every other
model call, and the diagram is drawn client-side from the model's output
(`apps/desktop/src/lib/whiteboard/`). For search, Core extracts the text on the canvas elements and
embeds that.

### Wiki links, backlinks, and a graph

Pages can link to each other with `[[wikilinks]]` and `@mentions`. Ryu tracks these as a
document-to-document graph (the `document_links` store), which powers a backlinks panel under the
editor (`apps/desktop/src/components/spaces/BacklinksPanel.tsx`,
`GET /api/spaces/:id/documents/:doc/backlinks`) and a knowledge-graph view (`GET /api/spaces/:id/graph`,
or `GET /api/graph` across all Spaces). It also feeds retrieval: link-expansion pulls chunks from
linked documents into the candidate set before reranking, on by default and controllable with
`RYU_SPACES_LINK_EXPANSION`.

<Callout type="info">
  This document-link graph is separate from the chunk-level entity GraphRAG (`graph_nodes`). Both
  feed search independently. For how retrieval scoring and GraphRAG work, see
  [Spaces & RAG](/docs/core/spaces-rag).
</Callout>

## Attach a Space to an agent

Attach a Space from the agent's configuration (the **Agent** edit page). Once attached, the agent
retrieves relevant passages from that Space and grounds its answers in them instead of relying on
the model's training alone. This is how you build an agent that answers from your own knowledge
base. To manage agents, see [Agents](/docs/desktop/user-guide/agents).

<Callout type="info">
  The embedding model and retrieval strategy are configurable, not hardcoded. Core ships a local
  default (the `nomic-embed-text-v1.5` GGUF served by the `llamacpp-embed` sidecar on `:8081`) so
  RAG works on install. Change the embedding model and re-index via
  `GET/POST /api/embeddings/model`. See [Spaces & RAG](/docs/core/spaces-rag).
</Callout>

## Long-term memory

Separate from Spaces, Core keeps an encrypted long-term memory store (`server/memory.rs`,
`MemoryStore`). It is captured as you converse, so you do not curate it by hand, and it is
encrypted at rest on your machine.

Before each reply Core retrieves relevant snippets from long-term memory and past conversations
and passes them to the model as background context. This **auto-recall** is on by default, never
blocks a reply if retrieval is unavailable, and skips the current conversation. The internals are
documented in [Memory](/docs/core/memory).

### View, search, and add memory

Manage memory from **Settings -> Memory** in the desktop app (`MemoryTab.tsx`):

- **Long-term memory** is an opt-in toggle, off by default for privacy.
- **Auto-recall** is the on-by-default similarity search across long-term memory and indexed
  Space documents, controlled by the `auto-recall-enabled` preference.
- **Search memory and Spaces** runs a ranked similarity search via `POST /api/retrieval/search`,
  the same retrieval path the desktop uses to surface context.
- **Index a memory chunk** manually adds a piece of text so Core can retrieve it later
  (`POST /api/retrieval/index`).

```
POST /api/retrieval/search
POST /api/retrieval/index
```

These are available to any client that talks to Core on `:7980`.

<Callout type="warn">
  Memory and Spaces are single-tenant. Conversations and memory carry no per-user identity, so
  people sharing one node share the same memory and Space documents. The "Connected" presence
  panel answers who is on a node, not who may see what. See
  [Node & presence](/docs/core/node-and-presence).
</Callout>
