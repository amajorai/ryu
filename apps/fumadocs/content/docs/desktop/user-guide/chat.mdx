Beyond plain back-and-forth, the Ryu chat composer carries a set of power features modeled on the
ergonomics of Claude Code. They live in the chat view: some as slash commands you type, some as
composer toggles and hover actions.

## Pursue a goal

A goal is a persistent completion condition attached to a conversation. After each turn a separate
model judges whether the goal has been met; if not, the conversation keeps going on its own until
it is met or a safety cap is reached.

- Type `/goal <condition>` to set a goal, or use the composer **+** menu and pick **Pursue goal**.
- An active-goal chip and a goal bar appear above the composer, showing the editable condition, a
  live elapsed timer, and the latest judge reason. The Target icon on the chip turns into an
  **✕** on hover to clear it.
- Type `/goal` alone to open the bar, and `/goal clear` to remove the goal.
- The **Stop** button halts the loop at any time.

The judge runs as its own model call, separate from your chat model. Choose which model judges
goals in **Settings → Goals**. The continuation loop is capped at a fixed number of turns so a goal
can never run away. For how the goal state and judge primitive work, see
[Goals](/docs/core/goals).

## Ask a side question with `/btw`

`/btw <question>` asks an ephemeral question about the current conversation. The side model sees the
conversation context but has no tools, returns a single answer in a dismissible overlay, and the
exchange never enters chat history. It is the inverse of a sub-agent - full context, no tools,
nothing persisted.

Type `/btw` once you have at least one message in the thread. Pick the model and reasoning effort it
uses in **Settings** (the `btw-model` and `btw-effort` preferences). For the underlying primitive,
see [Side Questions](/docs/core/side-questions).

## Double-check answers

Toggle **double-check** in the composer to have a second model review each answer as it arrives. It
fires once per assistant message and flags problems it finds. Configure which model reviews and at
what reasoning effort in **Settings → Double-check**. For the review primitive, see
[Double-Check](/docs/core/double-check).

## Voice input

Click the **microphone** in the composer to dictate. A live waveform (an `AnalyserNode`-driven
wave) shows you are being heard; when you stop, the 16 kHz mono WAV is posted to
`POST /api/voice/transcribe` and the transcript is appended to the composer
(`apps/desktop/src/hooks/useVoiceRecorder.ts`). The default speech engine (whisper) is bundled and
auto-downloaded, and is swappable. Pick your default microphone in **Settings → Audio**.

For the speech engines you can swap in, see [Voice Engines](/docs/desktop/engines/voice-engines).

## Agent session controls

When the active agent is an ACP agent, the composer surfaces per-agent session controls: pickers for
the agent's permission mode, reasoning effort, and model. Nothing here is hardcoded by Ryu. Each
picker is built from what the agent advertises at `session/new`, so a control only appears when that
agent actually reports it (`apps/desktop/src/lib/api/acp.ts`,
`apps/core/src/sidecar/adapters/acp.rs`).

| Control | When it appears | Source |
|---|---|---|
| **Permission mode** | The agent reports modes (for example `default`, `acceptEdits`, `plan`, `bypassPermissions`) | The agent's advertised mode set and active mode |
| **Reasoning effort** | The agent exposes a `thoughtLevel`-style select config option | That config option |
| **Model** | The agent supports the unstable ACP session-model capability | That capability (null and hidden otherwise) |

Core fetches this advertisement once per agent via `GET /api/agents/:id/acp-config` (cached per
agent) and returns all-null for non-ACP agents. Your choice is applied to the next chat turn, which
Core re-applies to that turn's session.

### Interactive tool-permission prompts

When an agent in a gating permission mode wants to run a tool, Core streams a `data-ryu-permission`
part and pauses the turn. The desktop renders the agent's offered options (allow once, allow always,
reject once, reject always) above the composer. Your choice is sent to `POST /api/chat/permission`,
keyed by the request id, which unblocks the awaiting turn (`apps/desktop/src/pages/ChatPage.tsx`,
`apps/core/src/sidecar/adapters/acp.rs`).

<Callout type="warn">
  These controls are data-driven, so they only appear when the ACP agent advertises the capability.
  The model picker in particular depends on the agent's support for the unstable ACP session-model
  feature, so many agents show no model picker at all.
</Callout>

## Branch a conversation

Every message has a **branch** action in its hover toolbar. Branching forks the conversation up to
that message into a new, independent conversation in a new tab, so you can explore an alternative
direction without disturbing the original thread. Conversations stay linear - there is no in-place
message tree. For how the fork is stored, see
[Conversations and Sessions](/docs/core/conversations-sessions).

## Edit and regenerate with versions

Every user message has an **edit** action, and every assistant reply a **regenerate** action, in
its hover toolbar. Editing a message or regenerating a reply forks a new version in place rather
than overwriting: a `< n / m >` pager appears on the affected turn so you can page between versions
and pick the one to keep. Unlike branching, this stays in the same conversation. The backend keeps a
lazy tree over the flat history (`POST /api/conversations/:id/messages/:mid/edit`, `/regenerate`,
`/select`); the pager and inline editor live in
`packages/blocks/src/desktop/agent-elements/user-message-editor.tsx`.

## Quote a selection to reply

Select any text in a message and a floating **Quote** button appears. Clicking it drops the
selection into the composer as a pending quote; when you send, it is prepended to your message as a
leading markdown blockquote, so the model sees exactly what you quoted and the quote survives a
reload (`packages/blocks/src/desktop/agent-elements/quote.tsx`).

## Message timestamps

Hover any message, user or assistant, and its send time shows in the hover toolbar beside Copy and
Branch. The time comes from Core's stored `created_at` (stamped at persist), formatted in place - no
extra round-trip.

## Rate replies with 👍 / 👎

Each assistant reply carries thumbs-up and thumbs-down buttons in its hover toolbar
(`POST /api/conversations/:id/messages/:mid/feedback`). One vote fans out to three sinks: it is
persisted on the message, it upgrades the continual-learning reward from a model guess to your human
label, and (on by default, local and private) it writes a memory fact. Vote again to clear it.

<Callout type="info">
  Voting a reply you just received works via a latest-turn fallback. Voting an older reply in a
  session you have not reloaded may need a reload first - a known v1 limitation.
</Callout>

## Token-speed and context stats

On a live reply, a small stats readout shows inference speed (tokens per second) with a hover card
breaking down token counts, and a context-usage ring shows how full the window is
(`packages/blocks/src/desktop/agent-elements/message-stats.tsx`, fed by a `data-ryu-stats` stream
part). These stats are live-only - they show while a reply streams, not on a reloaded conversation.

## Jump around with the TOC rail

A Notion-style table-of-contents rail runs down the left gutter of the message list, one marker per
user message. Hover the rail to reveal each message's text, click to scroll that turn into view; the
active turn's marker is highlighted (`packages/blocks/src/desktop/agent-elements/chat-toc.tsx`). The
rail hides on narrow windows and short threads.

## Queue messages while a reply streams

You can keep typing and sending while a reply is still streaming: extra messages stack in a
**queue** above the composer and drain automatically, one per turn, when the run finishes
(`packages/blocks/src/desktop/agent-elements/queue/`). Each queued item has send-now and remove
actions, plus Send all. Choose the drain order (oldest-first, latest-first, or send-all-combined) in
**Settings -> General -> Chats**.

## Tune how much tool detail you see

Agent tool calls can render compactly or in full. **Settings -> Appearance -> Chat** has a
**Tool detail** knob with three presets - Compact, Minimal, and Detailed - a monotonic ramp from
grouped-and-collapsed to fully expanded (`apps/desktop/src/components/settings/AppearanceTab.tsx`).
The individual toggles it drives sit under **Advanced** for fine control.

## The composer picker

The composer's agent and model picker is one grouped dropdown with a search box and three sections
(`packages/blocks/src/desktop/agent-elements/input-bar.tsx`):

- **Ryu Portal** - the flagship `ryu` agent, with its live model and reasoning-effort pickers when active.
- **Providers** - your configured Pi providers; an unconfigured one links straight to credential setup.
- **External Agents** - installed ACP agents, each probed lazily for its model and permission options when you drill in; agents you have not added yet show an Install button.

## Attach a Clip as context

**Clips** are agent-native screen recordings. Record your screen (with audio) or paste a video URL,
and Ryu turns it into a frame-plus-transcript context bundle you can attach to a chat message so the
agent can watch what happened (`apps/desktop/src/components/clips/`). You can also attach the last
few minutes of ambient activity without a discrete recording. Finished clips are filed into a
default **Clips** Space.

<Callout type="warn">
  Recording sources (display and window enumeration) are Windows-first, and the full
  desktop-to-Core-to-Shadow live path is not yet fully runtime-verified. Frame extraction and local
  ingest are proven; treat live capture as early.
</Callout>

## Image generation

Image generation runs on a local engine you install from the Store. The backend is live
(stable-diffusion.cpp via `POST /api/images/generate`), and the engine ships install, start, and
stop lifecycle in the Store's **Engines → Image** group.

<Callout type="warn">
  There is no "generate an image" action in the chat composer yet. Image generation is reachable
  through its backend and the engine lifecycle, but it is not surfaced as a chat composer feature
  today (this is a known gap in `CLAUDE.md` §4). To install the image engine, see
  [Voice Engines](/docs/desktop/engines/voice-engines) and the Store's Engines section.
</Callout>

## Speak replies aloud

Text-to-speech has a working backend (the OuteTTS engine, swappable, via `POST /api/voice/speak`)
and the engine installs from the Store.

<Callout type="warn">
  There is no "speak this reply" action on assistant messages in the desktop chat UI yet. TTS
  playback is wired in surfaces like the Island companion, but not on chat messages (a known gap in
  `CLAUDE.md` §4). The voice engines you can swap in are documented under
  [Voice Engines](/docs/desktop/engines/voice-engines).
</Callout>
