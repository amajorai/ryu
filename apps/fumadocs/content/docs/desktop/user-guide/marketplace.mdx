The Marketplace is the money layer on top of the catalog: buy paid skills, plugins, MCP servers, and
models, see what your organization owns, sell your own work, and keep a prepaid credits balance. It
runs in the control-plane server (`:3000`), authed with your Better Auth session, and opens hosted
checkout pages in your browser.

<TryInRyu page="marketplace" />

Free browse and install is a separate, anonymous flow covered in
[Store](/docs/desktop/user-guide/store). This page is only about the paid surface. Publishing your
own item lives in [Develop: Marketplace](/docs/develop/extensions/marketplace).

<Callout type="info">
Buying and selling are **org-level**. Wallets, licenses, and seller accounts all belong to your
active organization, not to your personal login, so create or select an organization first. The
desktop **Marketplace** page and the web `/marketplace` page render the same three tabs over the same
endpoints, differing only in auth (browser cookie session vs the desktop bearer).
</Callout>

## The three tabs

| Tab | What it shows | Backing route |
|---|---|---|
| **Browse** | Catalog cards; paid items show a price badge plus Buy, owned items show Owned, free items show no price or buy affordance | `GET /api/marketplace/catalog?kind=` |
| **My licenses** | Every paid item your org owns, newest first, with status (active / refunded / disputed) | `GET /api/marketplace/licenses` |
| **Sell** | Become a seller: Stripe Connect onboarding and payout status | `GET /api/seller/status`, `POST /api/seller/onboard` |

Free items never carry pricing and never write a license, so they never appear under My licenses. To
install one, use the free flow in [Store](/docs/desktop/user-guide/store).

## Buy a paid item

1. Open **Browse** and pick a kind (Skills, Plugins, MCP, or Models).
2. A paid card shows its price. Click **Buy**.
3. The desktop calls `POST /api/marketplace/purchase` with the item `kind` and `id`. The server reads
   the price, platform fee, and seller from the **stored** pricing (never the client) and creates a
   Stripe Checkout session as a destination charge.
4. The hosted Stripe page opens in your browser. Complete payment there.
5. Your license is granted asynchronously by the server webhook once the payment clears, so the app
   re-fetches your licenses on window focus. The new item then shows **Owned** in Browse and appears
   under **My licenses**.

If your org already holds an active license, Buy is a no-op: the purchase route returns
`alreadyLicensed: true` and the existing license, with no new charge.

<Callout type="info">
The price you pay is the stored price plus tax. Checkout calculates and collects sales tax / VAT / GST
via Stripe Tax. The platform takes a commission per sale (default 20%, `DEFAULT_PLATFORM_FEE_BPS =
2000` in `packages/api/src/routers/marketplace.ts`); Stripe routes the rest to the seller's connected
account.
</Callout>

### How licensing is enforced

The install handoff (`GET /api/marketplace/catalog/detail`) is the single fail-closed seam. For a
**free** item it stays anonymous and returns the descriptor and signature. For a **paid** item, if
your org does not hold an active license the route returns `402` with `requiresPurchase: true` and the
pricing, so Core can prompt you to buy rather than install. See
[Develop: Marketplace](/docs/develop/extensions/marketplace) for the publish and signing side.

## Sell your own items

Selling a paid item requires your org to complete Stripe Connect onboarding (payouts enabled). Free
listings need none of this.

1. Open the **Sell** tab and click **Set up payouts**.
2. The desktop calls `POST /api/seller/onboard`, which creates (or reuses) your org's Stripe Connect
   Express account and returns a fresh hosted onboarding URL.
3. Complete Stripe's identity, bank, and tax verification in your browser. Ryu never sees those
   details; Stripe owns KYC.
4. Return to the Sell tab. Payout status updates from the `account.updated` webhook, so refresh to see
   **Payouts enabled**.

Once payouts are enabled, publish a paid item with the publish API. The actual publish flow,
including the per-sale commission and grant signing, is documented in
[Develop: Marketplace](/docs/develop/extensions/marketplace).

<Callout type="warn">
Onboarding and purchasing need Stripe configured on the server. With no Stripe keys, the Sell tab
shows "Stripe is not configured on this server" and `POST /api/marketplace/purchase` returns `503`.
Managing the seller account is **admin-only**: `POST /api/seller/onboard` requires the caller be at
least an org admin (`roleSatisfies(mapBaRole(...), "admin")` in `packages/api/src/routers/seller.ts`).
</Callout>

## The credits wallet

Credits are a closed-loop, prepaid balance ("Ryu $") your org spends on first-party managed
inference. The wallet is org-level and lives in the control plane.

| Route | Purpose | Access |
|---|---|---|
| `GET /api/credits/wallet` | Org balance plus recent ledger | Any org member (read) |
| `POST /api/credits/topup` | Buy a credit pack through Polar checkout | Org owner / admin |
| `POST /api/credits/debit` | Internal atomic debit by micro-USD (idempotent) | Gateway service secret |

Top-up packs are `10`, `25`, or `100` (US dollars), or a custom amount between `$5` and `$1000`. The
buyer is **charged the face value plus a deposit fee** (the platform's managed-inference margin) and
the wallet is **credited the face value only**. The top-up route returns a fee quote so the desktop
can show the breakdown before you check out. Payment runs through Polar; the wallet is credited
asynchronously by the server webhook, so the balance re-fetches on focus and on the Refresh button.

<Callout type="info">
Topping up spends the shared org balance, so it is owner/admin only, enforced server-side. A plain
member or viewer has read-only access to the wallet. Credits are first-party only at launch; paid
third-party marketplace items are bought with a card through Stripe (above), not with credits.
</Callout>

<Callout type="warn">
The credits **debit loop is dormant by default**. `POST /api/credits/debit` is an internal
service endpoint the gateway calls with a service secret (`x-ryu-internal-secret` matching
`RYU_CREDITS_INTERNAL_SECRET`), and that metering hook is not yet wired into the Core-managed gateway
spawn. Until it is, a top-up credits a wallet that is not yet debited for usage. Top-up, purchase, and
license flows are independent and work.
</Callout>

## Where each surface lives

```
POST /api/marketplace/purchase   buy a paid item (Stripe Connect destination charge)
GET  /api/marketplace/licenses   the org's owned items
GET  /api/marketplace/catalog/detail  install handoff; 402 for an unlicensed paid item
POST /api/seller/onboard         create/reuse the org's Connect account, return onboarding URL
GET  /api/seller/status          stored onboarding + payouts-enabled state
GET  /api/credits/wallet         org balance + recent ledger
POST /api/credits/topup          Polar checkout for a credit pack + fee quote
```

Desktop code: `apps/desktop/src/pages/MarketplacePage.tsx` and `CreditsPage.tsx`. Web mirror:
`apps/web/src/app/marketplace/page.tsx`. Control-plane routers: `packages/api/src/routers/`
(`marketplace.ts`, `seller.ts`, `credits.ts`).
