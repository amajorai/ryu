---
title: Chat engines
description: The local inference engines Core can run, and how to swap the active one.
---

A chat engine is an OpenAI-compatible inference server bound to a model. Core can manage several, but
**only one is the resident, active local engine at a time**. The Gateway routes local model calls to
whichever engine is active.

## The engines

| Engine | Default port | Process | Default model |
|---|---|---|---|
| `llamacpp` | `8080` | `llama-server` | Gemma 4 E2B-it Q4_K_M (the bundled default) |
| `llamacpp-embed` | `8081` | `llama-server --embeddings` | nomic-embed-text v1.5 (for RAG) |
| `ollama` | `11434` | `ollama serve` | user-managed |
| `vllm` | `8000` | vLLM OpenAI server (Python) | Qwen2.5-1.5B-Instruct |
| `sglang` | `30000` | SGLang server (Python) | Qwen2.5-1.5B-Instruct |
| `docker-model-runner` | `12434` | Docker Model Runner (adopt-only) | any HF GGUF / Docker Hub model you pull |

`llamacpp` is the default chat engine, installed and downloaded on first run. `llamacpp-embed` is a
second llama.cpp instance dedicated to embeddings; it runs alongside the chat engine and is in the
startup order, so RAG gets real semantic vectors on install. `vllm` and `sglang` are Python-based and
require their packages installed; each resolves its model from an environment variable
(`RYU_VLLM_MODEL`, `RYU_SGLANG_MODEL`) or a built-in default.

`docker-model-runner` is **adopt-only**: Ryu downloads nothing - it routes to Docker's built-in model
server, which serves any Hugging Face GGUF (Llama, Qwen, …) or Docker Hub model you pull. You install
Docker Desktop and enable it yourself. See **[Docker Model Runner](/docs/desktop/engines/docker-model-runner)**
for the full setup.

<Callout type="info">
  Apple Silicon also has three MLX engines (`mlx`, `mlx-vlm`, `omlx`) that appear only on arm64 macOS
  nodes. They follow the same active-engine swap as the engines above.
</Callout>

<Callout type="info">
  These ports are configured defaults. The Gateway runs on its own port (`7981`), so it no longer
  contends with `llamacpp` on `8080`. If you point two engines at the same port, change one through
  its configuration (the Gateway via `RYU_GATEWAY_URL`, llama.cpp via the registry).
</Callout>

## The active engine

Core tracks one resident local engine and persists the choice in `~/.ryu/active-engine.json`, so it
survives restarts (`apps/core/src/sidecar/active_engine.rs`).

| Action | Endpoint |
|---|---|
| Report the active engine and what is available | `GET /api/engine/active` |
| Swap the active engine | `POST /api/engine/active` |

Swapping stops the incumbent and starts the new engine under a lock, so at most one resident chat
engine runs at a time. The choice is persisted before the start completes, so your intent is durable
even if the start fails. When Core spawns the Gateway, it sets `LOCAL_LLM_URL` to the active engine's
URL so the engine registers as a routable provider.

## Swappable defaults

The default chat and embedding models are resolved through the registry, in order of precedence:

1. An environment variable (for example `RYU_LOCAL_CHAT_MODEL_URL`).
2. A field in `~/.ryu/registry.json`.
3. A built-in constant.

This is how you change the bundled model without recompiling. To browse and install other GGUF
models, see [Models](/docs/desktop/models).
