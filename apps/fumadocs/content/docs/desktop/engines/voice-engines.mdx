---
title: Voice engines
description: Speech-to-text and text-to-speech engines that run alongside the chat engine.
---

Voice engines do speech-to-text (STT) and text-to-speech (TTS). Unlike chat engines, they run
**alongside** the active chat engine rather than instead of it, and they are opt-in: they are not in
the startup order, so you start them on demand from the Store or the API. Both STT engines can run at
once.

## The engines

| Engine | Default port | Model | Status |
|---|---|---|---|
| `whispercpp` | `8090` | `ggml-base.en` (whisper.cpp) | Built; downloaded on first run |
| `parakeet` | in-process | Parakeet v3 ONNX bundle | Built, but transcription is behind a cargo feature |

The whisper.cpp server is the default voice engine. Its model path defaults to
`~/.ryu/models/ggml-base.en.bin` and is overridable with `RYU_WHISPER_MODEL`.

Parakeet v3 is a second, ONNX-based engine that runs in-process via the `transcribe-rs` crate. Its
model bundle downloads from a public host. In-process inference is gated behind the off-by-default
`voice-parakeet` cargo feature (it pulls a heavy native ONNX Runtime dependency). Without the feature,
the lifecycle, download, and routing are all present, but a transcription request returns a clear
"feature not built" error.

## Transcribing

```
POST /api/voice/transcribe?engine=whisper|parakeet
```

The request is a multipart form with the audio bytes in a `file` field, and the response is the
transcribed `text`. The `engine` query parameter selects which voice engine handles the request.

In the desktop app, the composer's microphone button captures 16 kHz mono WAV through the Web Audio
graph and POSTs it to this endpoint, then appends the transcript to your message. See
[the desktop tour](/docs/desktop/user-guide) for the UI.

## Text-to-speech (TTS)

TTS is the generative counterpart to whisper and parakeet's transcription. Like the STT engines, TTS
engines are opt-in and run alongside the chat engine. There are two paths: a built-in OuteTTS engine
and a universal sidecar that fronts many other engines behind one contract.

<Callout type="warn">
  The TTS engines are opt-in and not in the startup order. Built-in OuteTTS synthesis has not been
  exercised live here. On the universal sidecar, Pocket TTS was verified producing a WAV, but
  KittenTTS install is blocked by an upstream `misaki` dependency pin. The sidecar needs Python.
</Callout>

### Built-in OuteTTS

The built-in engine (`outetts`) has **no resident server**. Each `POST /api/voice/speak` shells out
to the `llama-tts` CLI, which ships in the same llama.cpp release as `llama-server` and is extracted
by `LlamaCppDownloader::ensure_tts_binary` (`apps/core/src/sidecar/providers/outetts/downloader.rs`).
`llama-tts` runs an OuteTTS GGUF plus a WavTokenizer vocoder GGUF and writes a WAV file, which Core
reads back and returns as `audio/wav` (`apps/core/src/sidecar/providers/outetts/mod.rs`). Because
there is no server, the sidecar lifecycle maps to "binary plus models present": `start` ensures the
binary and both models are installed, `stop` marks it not ready.

The two models are swappable defaults, not locks. The OuteTTS language model and WavTokenizer vocoder
paths are overridable with `RYU_OUTETTS_MODEL` and `RYU_OUTETTS_VOCODER`.

| File | Default | Override |
|---|---|---|
| Language model | `OuteTTS-0.2-500M-Q4_K_M.gguf` (~400 MB) | `RYU_OUTETTS_MODEL` |
| Vocoder | `WavTokenizer-Large-75-F16.gguf` (~130 MB) | `RYU_OUTETTS_VOCODER` |

Both default to `~/.ryu/models/`. If the binary or either model is missing, `/api/voice/speak`
returns a clear error pointing you to install the OuteTTS voice engine from the Store first.

### Universal Ryu TTS sidecar

For engines beyond OuteTTS, Core manages the `ryutts` sidecar (`apps/tts-sidecar`), a small Python
FastAPI server on port `8085` that fronts many TTS engines behind one normalized HTTP contract:
`/health`, `/engines`, `/generate`, `/unload`. Adding an engine is one backend file plus one registry
row in that app, so Core never grows a per-engine branch
(`apps/core/src/sidecar/providers/ryutts/mod.rs`).

Two engines are seeded:

| id | Model | Notes |
|---|---|---|
| `kitten` | KittenTTS (15-80M ONNX) | CPU-only, fixed expressive voices |
| `pocket` | Kyutai Pocket TTS (100M) | ~6x realtime on CPU, voice cloning via `reference_audio` |

Core adopts an already-running sidecar (for example `bun run dev:tts`) rather than spawning a
competing process; otherwise it spawns `python -m ryu_tts` from the installed sidecar directory
(overridable with `RYU_TTS_DIR`, interpreter with `RYU_TTS_PYTHON`). Heavy per-engine inference
dependencies are the user's to `pip install`. Core points the sidecar's `HF_HOME` at
`~/.ryu/models/hf` so every downloaded model's bytes live under `~/.ryu` and Core's catalog can detect
installs (`RYU_TTS_HF_HOME` overrides).

### Speaking

```
POST /api/voice/speak
```

The body is `{ text, engine?, voice?, speed?, language?, reference_audio? }` and the response is an
`audio/wav` body (`apps/core/src/server/voice.rs`). The `engine` field selects the path: omitted or
`"outetts"` runs the built-in `llama-tts` path, and any other id (for example `"kitten"` or
`"pocket"`) is proxied to the sidecar's `/generate`. The remaining fields are passed through to the
sidecar when present.

Engine and model discovery is data-driven, so nothing is hardcoded:

| Action | Endpoint |
|---|---|
| List available TTS engines (built-in plus whatever the sidecar serves) | `GET /api/voice/tts-engines` |
| List the curated, installable TTS model catalog | `GET /api/voice/tts-models` |
| Install a curated model into the Core-managed HF cache | `POST /api/voice/tts-models/install` |

`GET /api/voice/tts-engines` always includes the built-in `outetts` and appends the sidecar's
`/engines` catalog when it is reachable. When the sidecar is down, only the built-in is returned. The
curated model catalog is the voicebox-style known-good set, distinct from the raw Hugging Face
`text-to-speech` browse in the Models tab. Curated installs run `huggingface_hub.snapshot_download`
into the Core-managed cache and are registered with the download overlay as a spinner entry, since
Hugging Face reports no byte total there.

In the desktop app, Settings then Voice has an engine and voice picker plus a "Test voice" button
(`apps/desktop/src/components/settings/TtsEngineSettings.tsx`). The engine set comes from
`/api/voice/tts-engines`, so it reflects whatever Core and the sidecar registry serve. The choice is
persisted to localStorage so any future in-chat "speak" surface can read the same default.
