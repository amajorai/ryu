This guide walks through adding your own agent to Ryu. "BYO" means you bring the agent CLI; Ryu
provides the session, tools, memory, and governance.

## Prerequisites

- Ryu Core running (`cargo run --manifest-path apps/core/Cargo.toml`)
- Your agent CLI installed and callable from PATH
- The CLI accepts an OpenAI-compatible base URL (or you accept that chat egress bypasses the Gateway)

## Step 1: Register the agent

Register your agent CLI with Core's catalog:

```bash
curl -X POST http://localhost:7980/api/agents/catalog/install \
  -H 'Content-Type: application/json' \
  -d '{
    "id": "my-agent",
    "name": "My Custom Agent",
    "engine": "acp-exec:my-agent --stdio",
    "transport": "acp",
    "description": "My custom coding agent"
  }'
```

The `engine` field is the spawn command. Core will run this command when a user selects your agent.

## Step 2: Test it locally

Select your agent in the desktop picker or send a chat with `agent_id: "my-agent"`:

```bash
curl -X POST http://localhost:7980/api/chat/stream \
  -H 'Content-Type: application/json' \
  -d '{
    "messages": [{"role": "user", "content": "Hello from my agent!"}],
    "agent_id": "my-agent"
  }'
```

You should see SSE events streaming back.

## Step 3: Route through the Gateway (optional)

If your agent reads `OPENAI_BASE_URL`, enable gateway routing so model calls are governed:

```bash
# Get current routing preferences
curl http://localhost:7980/api/preferences/agent-gateway-routing

# Enable for your agent
curl -X PUT http://localhost:7980/api/preferences \
  -H 'Content-Type: application/json' \
  -d '{
    "key": "agent-gateway-routing",
    "value": { "my-agent": true }
  }'
```

When enabled, Core injects `OPENAI_BASE_URL=http://localhost:7981/v1` and
`OPENAI_API_KEY=<gateway-token>` into your agent's spawn environment.

## Step 4: Declare tool access

Your agent can use tools through the MCP bridge. Declare which tools it needs in its tool
allowlist. The MCP bridge offers `tool_search` and `describe` to discover tools, and `execute` to
call them (governed by the Gateway's allowlist).

## Step 5: Add a plugin manifest (optional)

For distribution, create a `manifest.json` manifest:

```json
{
  "id": "com.example.my-agent",
  "name": "My Custom Agent",
  "version": "1.0.0",
  "runnables": [
    {
      "id": "my-agent-main",
      "name": "My Agent",
      "kind": "agent",
      "config": {
        "model": "gpt-4o",
        "tools": ["tool_search", "describe"]
      }
    }
  ],
  "permission_grants": ["tools.read", "tools.invoke", "model.chat"]
}
```

See [manifest.json manifest](/docs/develop/extensions/plugin-json-manifest) for the full schema.

## What your agent gets for free

When running inside Ryu, your agent automatically gets:

| Capability | How |
|---|---|
| **Tool access** | MCP bridge → unified tool catalog → Gateway-governed execution |
| **Memory** | Long-term conversation memory via `/api/memory/*` |
| **Spaces/RAG** | Document ingestion and semantic search |
| **Gateway governance** | Firewall, budgets, rate limiting, audit (if routed) |
| **Skills** | Agent skills loaded on demand via `skills__load` |
| **Sessions** | Conversation persistence across turns |

## Troubleshooting

| Symptom | Fix |
|---|---|
| Agent not showing in picker | Check `GET /api/agents/catalog` — ensure the agent is registered |
| 401 on tool calls | Ensure Gateway allowlist includes your tool IDs |
| Chat egress bypasses Gateway | Enable `agent-gateway-routing` for your agent (Step 3) |
| Agent crashes on spawn | Verify the spawn command works standalone: `my-agent --stdio` |
