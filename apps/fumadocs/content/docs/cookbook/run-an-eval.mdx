---
title: "Run an eval from the evaluator catalog"
description: "Discover the shared evaluator catalog, read which entries are wired to execution, then run an offline eval set over a dataset."
---

The gateway ships one shared evaluator catalog (30 entries across Security, Safety, Quality,
Conversation, Trajectory, Image, and Voice, plus two create-from-scratch templates). This recipe
discovers that catalog, reads the honesty flags that tell you what is wired versus catalogued, and
runs an offline eval set over a small dataset. It is for anyone who wants to score model output
against a reference rather than eyeball it.

Two distinct things live behind the word "eval" here, and this recipe keeps them apart. The catalog
(`GET /v1/evaluators`) is the shared definition surface. The offline runner (`POST /v1/evals/run`)
replays a dataset and scores each case — pass registry `evaluators` ids (run-level and/or per-case)
to score with catalog judges, or use `assertions` for deterministic checks.

## What you'll use

- [Evals and audit](/docs/gateway/evals-audit): the runner, its dimensions, and the audit log it feeds.
- [Security and governance](/docs/gateway/governance): the policy surface the inline evaluators back.

## Prerequisites

- **A running Ryu Core**, which runs the gateway as a managed sidecar on `http://127.0.0.1:7981`.
- A model the gateway can route, so a dataset run has something to call.

## Steps

<Steps>

<Step>

### Discover the catalog

The catalog is a read-only computation over the seed table (`apps/gateway/src/evaluators/catalog.rs`),
so it needs no auth and mutates no state. List it straight from the gateway:

```bash
curl http://127.0.0.1:7981/v1/evaluators
```

The response is `{ "evaluators": [...] }`. Each entry carries an `id`, a `category`, a `target`
(`Input` / `Output` / `Conversation` / `Trajectory` / `Image` / `Audio`), and a `capabilities`
object with `inline` and `offline` booleans. `capabilities` is the first thing to read - the
Quality, Conversation, Trajectory, and Voice categories are **offline-only** and are never offered
as inline guardrails, because they need a reference, a full conversation, or a trajectory to score.

</Step>

<Step>

### Read the honesty flags

Two flags tell you what actually runs today, so you never mistake a catalogued entry for a wired one:

- `enforced` - `true` on exactly the five text detectors wired to real **inline** execution today:
  `pii_leakage`, `code_injection`, `prompt_injection`, `toxicity`, and `bias_fairness`. Every other
  entry ships `enforced: false`, including the two image judges, which need a vision-capable judge
  that is not wired yet. `enforced` is an inline-wiring flag, not an offline-capability flag.
- `higherIsBetter` - `false` on negative-signal judges (toxicity, bias, hallucination, and the like),
  where a high score means a worse output. The security regex detectors keep `true` because their
  scorer already inverts, so `1.0` means clean.

<Callout type="info">
Filter the list to the entries whose `enforced` is `true` to see the guardrails that block, sanitize,
or warn on the request path right now. Treat the rest as catalogued definitions, not live execution.
</Callout>

</Step>

<Step>

### Run an offline eval set

The runner that works today is `POST /v1/evals/run`. It replays each prompt in `dataset` through the
full pipeline (firewall, routing, provider call) and scores each case. Pass registry evaluator ids
via a run-level `evaluators` array and/or per-case `evaluators`, or use `assertions` for deterministic
checks. An explicit run always scores, independent of passive evals, and cases run sequentially:

```bash
curl -X POST http://127.0.0.1:7981/v1/evals/run \
  -H "Authorization: Bearer $RYU_GATEWAY_TOKEN" \
  -H "content-type: application/json" \
  -d '{
    "model": "gpt-4o-mini",
    "evaluators": ["exact_match"],
    "dataset": [
      {
        "prompt": "What is the capital of France?",
        "expected": "Paris"
      },
      {
        "prompt": "Reply with valid JSON only.",
        "assertions": [{ "kind": "json_valid" }]
      }
    ]
  }'
```

Assertions are deterministic (`contains`, `not_contains`, `equals`, `regex`, `json_valid`) plus
`llm_judge`, a second pipeline call whose verdict is parsed defensively. Set `judge_model` to hold
one judge fixed across every case. The response returns per-case scores (including per-evaluator
breakdowns when `evaluators` are requested) plus an aggregate. An empty `dataset` uses a built-in
3-case set. See [Evals and audit](/docs/gateway/evals-audit) for the full field list.

From Core, `POST /api/gateway/evals/run` accepts the same body; add optional
`code_evaluators: [{ id, lang, source }]` (`lang` = `js` | `python`) for user functions Core runs
locally and merges into each case's `evaluators` array.

</Step>

</Steps>

## Check your work

- `GET /v1/evaluators` returns 30 entries. Exactly five report `enforced: true`
  (`pii_leakage`, `code_injection`, `prompt_injection`, `toxicity`, `bias_fairness`); every other
  entry, including both image judges, reports `false`.
- `POST /v1/evals/run` returns per-case scores and an aggregate. Registry `evaluators` add
  per-evaluator breakdowns; assertions still work for deterministic checks.
- A multi-model run with `llm_judge` fans out `models * cases * (1 + judge_calls)` provider calls
  under Core's 120s proxy timeout, so keep the matrix small or it times out at the proxy.
