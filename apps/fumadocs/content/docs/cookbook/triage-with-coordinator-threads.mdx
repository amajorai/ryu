---
title: "Triage a backlog with a coordinator agent"
description: "Have one agent spin up parallel worker threads to chew through a list of tasks, each isolated in its own git worktree, then read the results back."
---

You have a pile of independent tasks: a stack of bug reports to investigate, a list of files to
summarize, a set of repos to audit. Instead of working them one at a time, you let a single
coordinator agent create a worker thread per task, run them in the background in parallel, and
collect the answers when they land. This recipe is for anyone who wants real fan-out without hand
driving each run.

## What you'll use

- [Coordinator threads](/docs/core/coordinator-threads): the built-in `threads__*` tools a
  coordinator uses to create, run, and read worker threads.
- [Conversations and sessions](/docs/core/conversations-sessions): each worker thread is a real
  conversation, so it is durable, searchable, and resumable.
- [Git workspace](/docs/desktop/user-guide/git-workspace): per-conversation worktrees keep
  parallel workers from colliding on the same files.
- [Delegation](/docs/core/delegation): the ephemeral alternative, contrasted at the end.

## Steps

<Steps>

<Step>

### Grant the coordinator the `threads` tools

The `threads` tools are allowlist-gated. Only an agent whose tool allowlist grants `threads__*` can
spawn and manage workers, so pick the agent that will act as your coordinator and add the threads
tools to its allowlist on its edit page Tools tab.

If the agent's allowlist is left empty (the default for the flagship `ryu` agent), all enabled
built-in tools are offered, so `threads__*` is already reachable with no configuration. Narrow the
allowlist only if you want to scope the coordinator down to a specific set.

<Callout type="info">
This is an agent-facing capability today. There is no dedicated desktop coordinator screen yet: you
drive it by chatting with the coordinator agent, and it calls the tools through its normal tool
loop.
</Callout>

</Step>

<Step>

### Ask the coordinator to spin up a worker per task

In a chat with the coordinator, hand it the list and ask it to create one worker thread per item and
dispatch each one. Under the hood it calls `threads__create_thread` for each task, then
`threads__send_message_to_thread` to run that thread's agent on the instruction.

By default `send_message_to_thread` runs in the background and returns immediately, so the
coordinator fans out across the whole list rather than blocking on each worker:

```json
{
  "thread_id": "<worker-id>",
  "message": "Investigate issue #482 and propose a fix",
  "isolate": true
}
```

The response confirms the dispatch without waiting for the work to finish:

```json
{ "ok": true, "thread_id": "<worker-id>", "dispatched": true, "status": "running" }
```

Background fan-out is bounded by a process-global semaphore of 8 permits, so if you queue more than
eight at once the extras wait for a slot rather than being rejected.

</Step>

<Step>

### Let worktree isolation keep workers from colliding

When the working directory is a git repo and `isolate` is true (the default), each worker runs in
its own per-conversation git worktree. Two workers editing the same repo never step on each other's
files, because each has its own checkout. The worktree is created lazily on the first
`send_message_to_thread` and reused across that thread's later turns, keyed on its conversation id,
so a worker keeps the same checkout as you give it follow-up instructions.

See [Git workspace](/docs/desktop/user-guide/git-workspace) for how the per-conversation worktrees
are created and how their diffs surface.

</Step>

<Step>

### Read the results back

Since the workers run in the background, the coordinator polls them. `threads__list_threads` shows
every thread with its `run_status`, title, message count, and pinned/archived flags;
`threads__read_thread` returns a thread's most recent messages plus its `run_status`.

```json
{ "thread_id": "<worker-id>", "limit": 20 }
```

Each message body is truncated to keep a long worker reply from flooding the coordinator's context,
and `read_thread` reports `run_status` so the coordinator knows whether a worker is still running,
done, or failed. Once a worker is finished, the coordinator can summarize it, archive it with
`threads__set_thread_archived`, or pin the ones worth keeping in view with
`threads__set_thread_pinned`.

If you want a blocking call instead, pass `wait: true` to `send_message_to_thread` and it returns
the reply inline rather than dispatching in the background.

</Step>

</Steps>

## Threads vs delegate: which one

Both fan work out, but they answer different needs.

- Use [coordinator threads](/docs/core/coordinator-threads) when the work is **durable**: each worker
  is a real, persistent conversation you can read, resume, and revisit. This is the right tool for a
  backlog the coordinator checks back on over time.
- Use [`delegate__fanout`](/docs/core/delegation) when the work is **ephemeral**: parallel subtasks
  with clean contexts whose results come back in one call, with nothing to revisit afterward.

If a worker should run, finish, and be forgotten, delegate. If it should stick around as a thread you
can open later, use coordinator threads.
