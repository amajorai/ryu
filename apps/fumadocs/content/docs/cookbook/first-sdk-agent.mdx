---
title: "Build your first agent in code"
description: "Write an agent with the Ryu SDK whose every model call routes through the gateway, then stream its reply and run it locally."
---

You want to build an agent in TypeScript, not click one together in the app. This recipe scaffolds a
project with the Ryu SDK, writes a `defineAgent` whose every model call goes through the gateway (no
provider key in your code), streams its reply token by token, and runs it locally. It is for
developers who want the Runnable contract and gateway-mandatory model access from the first line.

This is the *write-your-own-agent* path (`defineAgent` + `ctx.gateway`). If you want the standalone
`defineModel` script path, where the model is a swappable string you change in one place, start from
[Quickstart](/docs/develop/quickstart) instead. The two are complementary: the same gateway governs
both.

## What you'll use

- [Quickstart](/docs/develop/quickstart): scaffold a plugin with `create-ryu-app`.
- [The SDK](/docs/develop/sdk): the `defineAgent` factory and the gateway-mandatory model client.
- [Gateway](/docs/gateway): what governs every model call your agent makes.

## Prerequisites

- **Bun** on your PATH (`bun` and `bunx`). Ryu uses [Bun](https://bun.sh) across the monorepo.
- **A running Ryu Core**, which runs the gateway as a managed sidecar on `http://127.0.0.1:7981`. A
  fresh install ships a local engine and a default model, so there is a model to route to out of the
  box.
- If your gateway requires auth, the `RYU_GATEWAY_TOKEN` for it. This is the gateway token, never a
  provider API key.

## Steps

<Steps>

<Step>

### Scaffold a project

Scaffold a new plugin, passing a lowercase project name:

```bash
bunx create-ryu-app my-first-agent
cd my-first-agent
bun install
```

`create-ryu-app` writes a valid `manifest.json`, a starter `src/agent.ts`, and a `package.json` that
depends on `@ryuhq/sdk`. See [Quickstart](/docs/develop/quickstart) for what each file is.

</Step>

<Step>

### Write the agent

An agent is a [Runnable](/docs/start-here/architecture/runnable-model): an object with an `id`, a
`name`, and a `run()` method. The one rule is that every model call goes through `ctx.gateway`,
never a provider SDK. Replace `src/agent.ts` with:

```ts
import { defineAgent } from "@ryuhq/sdk";

export const researcher = defineAgent({
  id: "agent-researcher",
  name: "Researcher",
  async run({ query }: { query: string }, ctx) {
    const result = await ctx.gateway.chat([
      { role: "system", content: "You answer concisely." },
      { role: "user", content: query },
    ]);
    return { answer: result.content };
  },
});
```

There is no API key and no base URL here. `ctx.gateway` is injected for you, and it is the only
allowed path to a model. The gateway resolves the provider, applies routing and budgets, and
returns the reply. Your agent code never names a provider or a model id, that decision lives in the
gateway config, which is exactly what makes it swappable without touching this file. See
[the gateway overview](/docs/gateway) for what that buys you.

</Step>

<Step>

### Run it locally

The `ryu` CLI runs a Runnable with an interactive chat loop:

```bash
bunx ryu dev src/agent.ts
```

The model client targets `RYU_GATEWAY_URL` (default `http://127.0.0.1:7981`). If your gateway
requires auth, set `RYU_GATEWAY_TOKEN` first. A running local Core runs the gateway as a managed
sidecar, so with a fresh install there is already something to call.

<Callout type="warn">
The agent needs a model the gateway can route. A fresh install ships a local engine and a default
model, but if the gateway cannot resolve the model id the turn fails. Point your call at a model
your gateway can route, for example a local model id or an `openrouter/...` slug.
</Callout>

</Step>

<Step>

### Stream the reply token by token

`ctx.gateway.chat()` waits for the whole reply. To show tokens as they arrive, swap it for
`ctx.gateway.stream()`, which yields `{ content, finishReason }` deltas. Accumulate them into the
return value and write each fragment to stdout as it lands:

```ts
import { defineAgent } from "@ryuhq/sdk";

export const researcher = defineAgent({
  id: "agent-researcher",
  name: "Researcher",
  async run({ query }: { query: string }, ctx) {
    let answer = "";
    for await (const delta of ctx.gateway.stream([
      { role: "system", content: "You answer concisely." },
      { role: "user", content: query },
    ])) {
      if (delta.content) {
        process.stdout.write(delta.content);
        answer += delta.content;
      }
    }
    return { answer };
  },
});
```

Nothing else changes: the same gateway governs the streaming call, so routing, budgets, and audit
still apply. Run `bunx ryu dev src/agent.ts` again and the reply now prints as it generates instead
of appearing all at once.

</Step>

<Step>

### Bundle it for the store

When the agent works, bundle the plugin so the desktop plugin store can install it:

```bash
bun run pack
```

This runs `bunx ryu pack .`, validates `manifest.json` against the same schema the store reads, and
writes `dist/plugin.bundle.json`. An invalid manifest exits non-zero and names the failing field.

</Step>

</Steps>

## Check your work

- `bunx ryu dev src/agent.ts` opens an interactive loop. Type a question and you get a concise
  answer back, proof the call reached a model through the gateway.
- After the streaming step, the reply prints **incrementally**, fragment by fragment, rather than
  appearing in one block when the turn finishes.
- If the gateway cannot resolve the model, the turn fails with a routing error instead of hanging,
  point it at a model your gateway can route (see the callout above).
- `bun run pack` exits `0` and writes `dist/plugin.bundle.json`. A broken `manifest.json` exits
  non-zero and names the failing field, so a bad manifest never reaches the store.
