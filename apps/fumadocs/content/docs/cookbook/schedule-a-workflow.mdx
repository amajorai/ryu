---
title: "Run a workflow on a schedule"
description: "Give a workflow a cron or interval trigger so it fires on its own, with optional human approval before each run."
---

You have a workflow you run by hand, a daily digest, a nightly cleanup, a recurring check, and you
want it to fire on its own. This recipe adds a schedule trigger to a workflow so the scheduler runs
it on a cron expression or a fixed interval, and optionally gates each run behind your approval. It
is for anyone who wants set-and-forget automation that survives restarts.

## What you'll use

- [Workflows](/docs/core/workflows): the DAG engine, triggers, and durable resume.
- [Scheduler](/docs/core/scheduler): the always-on tick loop that fires the job.

## Steps

<Steps>

<Step>

### Have a workflow to run

Start from a workflow you have built on the **Workflows** page, or assemble one. You can drag nodes
on the canvas, or describe it to the builder chat and let it assemble the DAG for you.

<TryInRyu page="workflows" />

</Step>

<Step>

### Add a schedule trigger

A workflow declares zero or more triggers. Above the canvas, the trigger panel lets you add a
**schedule** trigger and choose either a cron expression or an `every` interval:

| Kind | Example | Fires |
|---|---|---|
| `cron` | `0 9 * * *` | At most once per matching minute, evaluated in UTC (here, 9am daily) |
| `every` | `30m` | On a fixed interval parsed like `30s`, `5m`, `1h` |

On save, Core reconciles the trigger into a scheduler job with a deterministic id
(`wf-sched-<workflow_id>-<idx>`) and target `JobTarget::Workflow`. The job is file-backed, so the
schedule survives a Core restart: on boot the scheduler reloads it and resumes ticking.

<Callout type="info">
Triggers are reconciled idempotently on every save: editing the schedule updates the same job, and
deleting the trigger (or the workflow) tears the job down. A cron expression wins over `every` if you
set both.
</Callout>

</Step>

<Step>

### (Optional) Require approval before each run

If the workflow does something you want to eyeball first, mark its scheduled job to require approval.
A flagged job's firing raises a single, deduped entry in the approval inbox instead of running, and
approving it runs the exact same run the autonomous tick would have. Rejecting or letting it expire
skips that run. The flag is `require_approval` on the scheduled job (settable on `POST /api/jobs`).

</Step>

<Step>

### Let it ride the tick loop

You do not start anything. The scheduler's tick loop fires the job on its cron or interval, runs the
workflow, and checkpoints every node to disk so a crash mid-run resumes rather than restarting. Watch
runs on the Workflows page, or trigger one immediately to test the wiring:

```http
POST /api/workflows/:id/run
```

</Step>

</Steps>
