---
title: "Give a model a tool it can call"
description: "Define a typed tool with the Ryu SDK, validate its input, and let an agent call it during its turn."
---

A model can only answer from what it knows unless you give it a tool. This recipe defines a tool
with the Ryu SDK, the kind a model can call mid-turn, with a typed input schema that is validated
before your code runs, then hands it to an agent. It is for developers who want their agent to *do*
something, not just talk.

## What you'll use

- [The SDK](/docs/develop/sdk): the `defineTool` and `defineAgent` factories.
- [MCP registry](/docs/core/mcp-registry): how tools are discovered and dispatched once installed.

## Prerequisites

- **Bun** on your PATH (`bun` and `bunx`).
- A scaffolded plugin project. If you do not have one, start from
  [Build your first agent in code](/docs/cookbook/first-sdk-agent) and come back.
- **A running Ryu Core** (it runs the gateway sidecar) with a model the gateway can route, so the
  agent that calls the tool has something to answer with.

## Steps

<Steps>

<Step>

### Define the tool

A tool is a [Runnable](/docs/start-here/architecture/runnable-model) with `kind = "tool"` and a JSON
Schema describing its input. The schema is the same shape Core's tool layer expects, so it forwards
without transformation. The SDK validates input against it before your `run()` body executes, so a
missing required field or a type mismatch throws a descriptive error rather than reaching your code:

```ts
import { defineTool } from "@ryuhq/sdk";

export const wordCount = defineTool({
  id: "tool-word-count",
  name: "Word Count",
  schema: {
    type: "object",
    properties: {
      text: { type: "string", description: "The text to count words in" },
    },
    required: ["text"],
  },
  async run({ text }: { text: string }) {
    return { words: text.trim().split(/\s+/).filter(Boolean).length };
  },
});
```

A tool does not need a model, so `ctx.gateway` is optional in its body. If a tool *does* need model
help, it must still call through `ctx.gateway` like everything else.

</Step>

<Step>

### Give it to an agent

List the tool in the agent's `tools` so the agent can call it as a named tool during its turn:

```ts
import { defineAgent } from "@ryuhq/sdk";
import { wordCount } from "./word-count";

export const editor = defineAgent({
  id: "agent-editor",
  name: "Editor",
  tools: [wordCount],
  async run({ draft }: { draft: string }, ctx) {
    const result = await ctx.gateway.chat([
      { role: "system", content: "You are an editor. Use your tools when counting helps." },
      { role: "user", content: draft },
    ]);
    return { feedback: result.content };
  },
});
```

The agent and the tool are peers under the one Runnable contract: an agent exposes a workflow or a
tool as something it can call, not as a subordinate. See [the Runnable model](/docs/start-here/architecture/runnable-model).

</Step>

<Step>

### Run and bundle

Run the agent locally to watch it call the tool, then bundle the plugin:

```bash
bunx ryu dev src/agent.ts
bun run pack
```

`pack` validates the manifest and writes `dist/plugin.bundle.json` for the store.

</Step>

</Steps>

<Callout type="info">
Input validation is enforced at `run()` time. Calling the tool with a missing `text`, or with a
number where a string is expected, throws before the body runs, so a malformed call from the model
never silently does the wrong thing.
</Callout>

## Check your work

- Run `bunx ryu dev src/agent.ts` and give the agent a draft. When counting helps, it emits a tool
  call for `Word Count` and its feedback reflects the real count, proof the model reached your code.
- Call the tool with a missing `text` (or a number in its place) and `run()` throws a descriptive
  validation error **before** the body executes, rather than returning a wrong count.
- `bun run pack` exits `0` and writes `dist/plugin.bundle.json` for the store.
