This page consolidates the domain-specific vocabulary used throughout Ryu. Each term includes its
definition, where it lives in the system, and a cross-reference to the deeper documentation.

## Core concepts

### Runnable

The unified contract for anything Ryu executes: one interface with `input → run → output`. The
eight kinds are peers, not a hierarchy. An agent invokes a workflow by exposing it as a named tool;
a workflow orchestrates agents by calling them as steps.

| Kind | What it does |
|---|---|
| `agent` | Multi-turn model loop with memory, tools, and policy slots |
| `workflow` | DAG of typed nodes with durable execution |
| `tool` | Callable function with JSON Schema input/output |
| `skill` | Reusable prompt pattern (SKILL.md format) |
| `companion` | In-desktop overlay/panel surface |
| `channel` | Messaging-platform bot adapter |
| `engine` | Inference backend (llama.cpp, OpenAI, etc.) |
| `policy` | Gateway-enforced rule |

See [Runnable model](/docs/start-here/architecture/runnable-model).

### Session

Owns a conversation's run history and state. Lives in Core. A session maps to one agent and
persists across turns. Sessions are stored in SQLite under `~/.ryu/`.

See [Conversations & sessions](/docs/core/conversations-sessions).

### Conversation

A user-visible thread of messages. One conversation may span multiple sessions (e.g., when an
agent is swapped). Conversations are the UI-level abstraction; sessions are the backend-level
abstraction.

### App / Plugin

A manifest (`plugin.json`) bundling Runnables + an optional Companion surface. Apps are
installable and enableable on-the-fly with permission grants. The legacy name "plugin" is still
used in code (`apps/core/src/plugins/`), but the user-facing term is "app."

See [Plugin manifests](/docs/develop/extensions/plugin-json-manifest).

## Architecture terms

### Ryu Core

The local Rust backend (`apps/core`, default `127.0.0.1:7980`). Owns the **"what runs"** half of
the system: engine processes, conversation storage, long-term memory, tool routing, workflow
execution, scheduling, and MCP registry.

See [Core Internals](/docs/core).

### Ryu Gateway

The LLM control layer (`apps/gateway`, default `127.0.0.1:7981`). Owns the **"what is allowed,
shared, measured, or paid for"** half: routing, firewall/DLP, budgets, evals, audit, and tool
governance. Exposes an OpenAI-compatible API.

See [Gateway](/docs/gateway).

### Core vs Gateway rule

The single most important design rule:
- If code decides **what runs** → Core
- If code decides **what is allowed/shared/measured/paid** → Gateway

Core never enforces policy inline; it routes every model call through the Gateway.

See [Core vs Gateway](/docs/start-here/architecture/core-vs-gateway).

### Sidecar

A child process managed by Core (engine, Gateway, MCP server, etc.). Core downloads,
checksum-verifies, spawns, and health-checks ~16 sidecars. The Gateway itself runs as a
Core-managed sidecar in the default desktop setup.

### Capability layer

Ryu decomposes into ~40 self-contained packages in layers (L0-L3 + Gateway). Each layer depends
only on layers below it, forming a DAG with no cycles.

| Layer | Contents |
|---|---|
| L0 | Storage, crypto, engines, downloads |
| L1 | RAG, TTS, STT, sandbox, mesh |
| L2 | Memory, spaces, retrieval, tool-exec |
| L3 | Meetings, monitors, workflows, skills |
| Lg | Gateway: router, firewall, budgets, evals |
| Kernel | Sessions, registry, sidecars, auth |

See [Capability layers](/docs/start-here/architecture/capability-layers).

### Platform decomposition

The process of breaking the monolith into ~40 swappable capability crates (`crates/core/*`, `crates/gateway/*`) and
self-contained apps under `apps-store/<app>/`. The canonical record is
[`docs/full-decomposition-program-2026-07-16.md`](https://github.com/anomalyco/ryu-closed/blob/main/docs/full-decomposition-program-2026-07-16.md).

## Routing terms

### Plane A — Model routing

The Gateway picks *which LLM* one OpenAI-compatible completion uses. Smart routing (classifier,
embedding, or keyword strategy) rewrites `body["model"]`, then `ModelRouter::route` maps model
to provider. Optional and fail-open.

See [Routing](/docs/gateway/routing).

### Plane B — Agent routing

Core picks *which agent* serves the turn. Two paths:
- `AgentRoute::OpenAiCompat` — feeds Plane A (fully governed)
- `AgentRoute::Acp` — own subprocess, own provider calls (only tool egress governed via MCP bridge)

See [Routing planes](/docs/gateway/routing-planes).

### Smart routing

A Gateway feature that uses a cheap classifier (LLM, embedding, or keyword strategy) to pick the
best model for a request before routing. Opt-in and fail-open. The classifier runs as a direct
`provider.complete` call (not a tool call, so it never recurses).

### Auto routing

A Core feature where `AUTO_AGENT_ID` resolves to a concrete agent at runtime based on
configuration. Per-agent gateway-routing toggles inject `OPENAI_BASE_URL`/`OPENAI_API_KEY` into
ACP agents at spawn.

### Fallback chain

An ordered list of models the Gateway tries when the primary model fails. Configured in
`gateway.toml` under `[routing]`.

## Protocol terms

### ACP (Agent Communication Protocol)

The protocol used by agents like Claude Code, Codex, Pi, OpenClaw, and Hermes. ACP agents run as
subprocesses and communicate via their own protocol. Core bridges ACP agents to the Gateway via
the MCP bridge for tool governance.

See [ACP agents](/docs/start-here/architecture/acp-agents).

### OpenAI-compatible

Any endpoint that speaks the OpenAI `/v1/chat/completions` wire format. Core's OpenAI-compat
adapter and the Gateway both expose this interface.

### MCP (Model Context Protocol)

A standard protocol for exposing tools to LLM agents. Ryu supports MCP servers both as a registry
(Core registers MCP servers and exposes their tools) and as a host (the `apps/mcp` package
exposes Core to external MCP hosts like Claude Desktop).

See [MCP](/docs/mcp).

### SKILL.md

The agent skills format: a Markdown file with YAML frontmatter that teaches an agent how to
perform a specific task. Skills are reusable prompt patterns installed into `~/.claude/skills/`.

See [Agent Skills](/docs/skills).

## Gateway subsystems

### Firewall

Hierarchical policy system that inspects both requests and responses for PII, prompt injection,
jailbreak attempts, and custom patterns. Policies cascade from org-level → agent-level → default.
Enforcement actions: `warn_and_continue`, `block`, `sanitize`.

See [Security](/docs/gateway/security).

### DLP (Data Loss Prevention)

Part of the firewall that scans outbound content for sensitive data (SSN, credit cards, API keys,
etc.) and redacts or blocks it. Operates on both request and response sides.

### Budgets

Per-user, per-agent, and per-session token spending caps. When a budget is exceeded, the Gateway
can downgrade the model, reject the request, or trigger an alert.

See [Reliability](/docs/gateway/reliability).

### Evals

Automated quality measurements. The Gateway has 30+ built-in evaluators (relevance, coherence,
safety, etc.) that can score individual requests or run against datasets. Evals drive A/B routing
decisions.

See [Evals & audit](/docs/gateway/evals-audit).

### Audit

Per-request trail of what the Gateway did: which model was selected, budget checks, firewall
decisions, cache hits, and token usage. Stored in SQLite and queryable via `GET /v1/audit`.

### Circuit breaker

Per-provider failure tracking. When a provider exceeds a failure threshold, the Gateway opens the
circuit and routes to the next provider in the fallback chain. Automatically resets after a
timeout.

### Semantic cache

Cache that matches requests by embedding similarity (not exact string match). When a semantically
similar recent request is found, the cached response is returned without hitting the provider.

### Passthrough proxy

A Gateway feature that proxies Claude Code and Codex requests while preserving the caller's
subscription. Adds request-side firewall/DLP and audit without modifying the bearer token.

See [Gateway for any agent](/docs/gateway/gateway-for-any-agent).

## Core subsystems

### Memory

Long-term memory stored in SQLite. Agents can write facts, preferences, and context that persist
across sessions. Scoped to agent, user, or globally.

See [Memory](/docs/core/memory).

### Spaces (RAG)

Knowledge spaces containing documents that agents can search via semantic retrieval. Each space
has its own embedding index. Documents can be uploaded, re-embedded, and searched.

See [Spaces & RAG](/docs/core/spaces-rag).

### Workflow engine

A DAG-based execution engine with 22+ node kinds (Prompt, Condition, Transform, Tool, While,
SubWorkflow, AgentDelegate, Awakeable, etc.). Supports durable execution via an in-process
`FallbackEngine` with per-node checkpoints that survive restarts.

See [Workflows](/docs/core/workflows).

### Capability broker

A registry that resolves abstract capability requirements (`requires.capabilities: ["rag"]`) to
concrete provider apps. The binding registry mirrors the Gateway's `ModelRouter` ladder: override
> single > explicit `Ambiguous` refusal.

See [Platform decomposition](/docs/start-here/architecture/platform-decomposition).

### MCP registry

Core's catalog of registered MCP servers and their tools. Tools from registered servers appear in
the unified tool catalog and can be searched, described, and called by agents.

See [MCP registry](/docs/core/mcp-registry).

### Identity Vault

Per-domain credential storage. Agents can store and retrieve API keys, tokens, and other secrets
scoped to specific domains. Credentials are encrypted at rest.

See [Identity Vault](/docs/core/identity-vault).

### Delegation

The ability for one agent to fan out subtasks to other agents (coordinator threads). The
coordinator manages context isolation and result aggregation.

See [Delegation](/docs/core/delegation).

### Goals

A chat feature (`/goal`) that lets users set a persistent objective for an agent. The agent
pursues the goal across turns, reporting progress.

See [Goals](/docs/core/goals).

### Side questions

A `/btw` command that lets users ask a synchronous side question without interrupting the main
conversation flow. The answer is returned inline.

See [Side questions](/docs/core/side-questions).

## Plugin terms

### plugin.json

The manifest format for Ryu apps. Contains: `id` (reverse-domain), `name`, `version`,
`runnables[]`, `permission_grants[]`, `companion`, `activation_events[]`, `contributes`,
`requires`, `targets[]`, and listing metadata (`description`, `tagline`, `iconUrl`, etc.).

See [Plugin manifests](/docs/develop/extensions/plugin-json-manifest).

### Permission grants

Declarations in `plugin.json` that a plugin needs to function (e.g., `memory:read`,
`spaces:write`, `gateway:budget`). Grants are validated at enable-time and routed to the Gateway.

### Activation events

VS Code-style events that control when a plugin loads: `*` (always), `onStartup`, `onChat`,
`onCommand:<id>`.

### Contribution points

Declarative UI extensions a plugin can provide: `composer_controls`, `settings_tabs`,
`slash_commands`, `widgets`.

### Companion

An in-desktop overlay or panel surface that a plugin can contribute. Rendered in a dedicated
panel in the desktop app.

## Configuration terms

### gateway.toml

The Gateway's primary configuration file. Contains sections for `[auth]`, `[providers.*]`,
`[routing]`, `[firewall]`, `[budgets]`, `[rate_limit]`, `[cache]`, `[semantic_cache]`,
`[circuit_breaker]`, `[evals]`, `[audit]`, `[compression]`, `[channels.*]`, `[concurrency]`,
`[exec_budget]`, `[tools]`, `[composio]`, `[credits]`, `[control_plane]`, and more.

See [Configuration](/docs/gateway/configuration).

### Environment variables

~689 environment variables across the entire monorepo, documented in
[Environment variables](/docs/start-here/environment-variables). Key ones:
- `RYU_TOKEN` — Node admittance token
- `GATEWAY_MASTER_KEY` — Gateway admin key
- `RYU_DIR` — Data directory (default `~/.ryu`)

### Preference keys

~30+ runtime preference keys stored in `~/.ryu/preferences.db` that control feature behavior.
Examples: `agent-gateway-routing`, `embedding-model`, `approval-mode`, `hf-token`.

## Protocol and data terms

### OKF (Open Knowledge Format)

A bundle format for exporting knowledge (spaces, documents, embeddings) from one Ryu node to
another. Created via `ryu okf export`.

### Worktree

A per-run isolated copy of the active working folder. When an agent runs, it gets its own git
worktree so file modifications don't conflict with other runs or the user's live files.

See [Git workspace](/docs/desktop/user-guide/git-workspace).

### Prompt injection

An attack where malicious instructions are embedded in content that an LLM processes, causing it
to ignore its system prompt or take unintended actions. The Gateway's firewall detects and blocks
these.

### HITL (Human-in-the-Loop)

Approval gates where an agent pauses execution and waits for human confirmation before proceeding.
Implemented via the `Awakeable` node kind in workflows and the approval inbox.

See [Approvals](/docs/core/approvals).

### PTC (Programmatic Tool Calling)

The ability for agents to call tools programmatically via `POST /api/tools/exec` through a
sandboxed execution environment (Deno subprocess).

See [Programmatic tool calling](/docs/core/programmatic-tool-calling).
