The firmware is one ESP-IDF codebase shared across all three devices. A set of reusable components in `apps/hardware/firmware/shared/` does the heavy lifting, and each device app (`watch/`, `necklace/`, `desk/`) supplies a `main.c`, a board config, and a display implementation, then calls `rhp_app_run()` with its capability profile.

## Toolchain

- **ESP-IDF >= 5.1** with the ESP32-S3 toolchain.
- The first build auto-pulls managed components: `chmorgan/esp-libopus`, `espressif/esp_websocket_client`, `lvgl/lvgl ^9.2`, `espressif/esp_lcd_gc9a01`, `espressif/esp_lvgl_port`, `espressif/qrcode`, and (necklace only, when `CONFIG_RYU_NECKLACE_CAMERA` is set) `espressif/esp32-camera`.

<Callout type="info">
  CMake is also needed on the **node** side, not just for firmware. Core's `opus` crate vendors and builds libopus through CMake, and the repo's `.cargo/config.toml` sets `CMAKE_POLICY_VERSION_MINIMUM=3.5` so a too-new CMake 4.x still builds the old vendored libopus. Keep that file. See [Deployment](/docs/hardware/deployment).
</Callout>

## Shared components

Each is an ESP-IDF component with a clean header so the three device apps reuse it.

| Component | Responsibility |
|---|---|
| `protocol` | RHP JSON encode and decode (cJSON) plus binary framing. Mirrors the [protocol spec](/docs/hardware/protocol). |
| `ws_client` | WebSocket connect, Bearer auth, send and receive text and binary, auto-reconnect with backoff. |
| `audio_io` | I2S mic capture and speaker playback, with a codec abstraction (ES8311, or INMP441 + MAX98357A). |
| `opus_codec` | Opus encode at 16 kHz and decode at 24 kHz, fixed-point, low complexity. |
| `vad` | Voice-activity gate plus the pre-roll ring buffer (WebRTC-style). |
| `ble_prov` | The provisioning and relay GATT services (see [Pairing](/docs/hardware/pairing)). |
| `wifi_mgr` | NVS creds, connect, reachability probe, and the Mode B versus Mode A selection signal. |
| `pairing` | device_id and nonce generation, the QR payload, and NVS storage of the token and URL. |
| `display` | Display abstraction: `none` (necklace), `lcd_round` (watch), `eink+lcd` (desk). |
| `face` | The island-eyes emotion renderer for LVGL displays, mapping state to animation. |
| `app_core` | The device state machine (idle, chat, ambient) that wires every component together, exposing `rhp_app_run()`. |
| `dash_client` | The TRMNL poll-bitmap client: HTTP poll, image fetch, and blit. Added for the [dashboard](/docs/hardware/dashboard). |

## Per-device apps

Each device directory holds a `main.c`, a board config (SPI and I2S pins, codec selection), an `sdkconfig.defaults` (octal PSRAM, LVGL, and similar), and a display implementation. `EXTRA_COMPONENT_DIRS` already points at `../shared`, and the CMake `REQUIRES` are wired, so the per-device app is thin.

The reference panels assumed where unspecified are GC9A01 for the watch, Waveshare UC8179 7.5-inch e-ink plus optional ST7789 for the desk, and OV2640 for the necklace camera.

## Build and flash

Run these per device, with `watch`, `necklace`, or `desk` as the target directory.

<Steps>
  <Step>
    Source the ESP-IDF environment in your shell, then set the chip target once:

    ```bash
    idf.py -C apps/hardware/firmware/watch set-target esp32s3
    ```
  </Step>
  <Step>
    Build. The first build pulls the managed components listed above.

    ```bash
    idf.py -C apps/hardware/firmware/watch build
    ```
  </Step>
  <Step>
    Flash and monitor over the serial port.

    ```bash
    idf.py -C apps/hardware/firmware/watch -p <PORT> flash monitor
    ```
  </Step>
</Steps>

## Board configuration

Adjust the board config to your actual hardware before flashing:

- **Pins and codec** - each device's `main/` carries the SPI and I2S pin map and the codec choice. The reference values assume the reference panels; change them to match your board.
- **PSRAM and LVGL** - `sdkconfig.defaults` enables octal PSRAM and LVGL. Keep these for displays and the camera path.
- **Necklace camera** - the OV2640 path is gated behind `CONFIG_RYU_NECKLACE_CAMERA`. Enable it through `idf.py menuconfig` for a camera build, or leave it off for an audio-only necklace.

<Callout type="warn">
  The firmware is flash-ready but not yet hardware-tested. Treat these steps as first bring-up, not a proven appliance image. Verify pins against your board, and bring up [Mode B first](/docs/hardware/deployment) to isolate the core path before testing the BLE relay.
</Callout>
