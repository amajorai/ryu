title: "Pairing and provisioning"
description: "How a device pairs with your account over QR and BLE, the GATT provisioning and relay services, the canonical UUID table, and the pairing trust model."
---

Pairing binds an unprovisioned device to your account and hands it everything it needs to connect: a `device_token`, the node URL, and (optionally) WiFi credentials. The flow uses a QR code and a BLE provisioning channel, driven by the Ryu mobile app.

## The pairing flow

<Steps>
  <Step>
    **Unprovisioned boot.** The device generates a `device_id` and a `pairing_nonce`, starts BLE GATT advertising the RHP Provisioning Service, and (watch and desk) renders a QR code encoding `ryu-pair://<device_id>?n=<nonce>&t=<device_type>`. The necklace has no display, so it advertises the same payload in a GATT characteristic.
  </Step>
  <Step>
    **App scans.** In the Ryu mobile app, signed in with better-auth, you open Devices then Pair and scan the QR (or pick the device over BLE).
  </Step>
  <Step>
    **App pairs with the node.** The app calls `POST /api/hardware/pair` with `{ device_id, pairing_nonce, device_type }`. The node verifies the nonce, registers the device to your account, and returns `{ device_token, node_url }`.
  </Step>
  <Step>
    **Provision over BLE.** The app writes to the device's provisioning characteristic: `{ device_token, node_url, wifi_ssid?, wifi_psk?, prefs }`. The device persists this to NVS.
  </Step>
  <Step>
    **Connect.** The device tries Mode B (direct WiFi) if creds are present and the node is reachable, otherwise Mode A (relay through the app). The pairing characteristic closes and the device reboots into normal operation, using `device_token` as the Bearer on the WebSocket upgrade.
  </Step>
</Steps>

## GATT services

Pairing and the Mode A relay each have their own BLE GATT service. The phone is the BLE central; the device is the peripheral.

### Provisioning service (`RHP_PROV`)

Used only while pairing.

- `pairing_info` (read / notify) - device_id, nonce, and type.
- `provision` (write, chunked) - the token, URL, and WiFi JSON.
- `status` (notify) - the provisioning result.

### Relay service (`RHP_RELAY`)

Used at runtime in Mode A. The phone bridges these characteristics to and from the node WebSocket frames one to one.

- `ctrl_up` (notify) - JSON control, device to node.
- `ctrl_down` (write) - JSON control, node to device.
- `audio_up` (notify) - Opus, device to node.
- `audio_down` (write) - Opus, node to device.

The direction rule is simple: every `*_up` characteristic is a notify (the phone subscribes), and every `*_down` is a write (the phone writes).

## Canonical UUID table

These 128-bit UUIDs are the single source of truth. Both sides use them verbatim - the firmware (`apps/hardware/firmware/shared/ble_prov/ble_prov.c`) and the app (`apps/native/lib/hardware/ble.ts`). A divergence here breaks discovery, so never edit one side only.

| Service / characteristic | Direction (firmware) | UUID |
|---|---|---|
| `RHP_PROV` (service) | - | `7e1a0001-9b34-4d21-8f5a-2c6b1d3e0001` |
| `RHP_PROV` pairing_info | read / notify | `7e1a0001-9b34-4d21-8f5a-2c6b1d3e0002` |
| `RHP_PROV` provision | write (chunked) | `7e1a0001-9b34-4d21-8f5a-2c6b1d3e0003` |
| `RHP_PROV` status | notify | `7e1a0001-9b34-4d21-8f5a-2c6b1d3e0004` |
| `RHP_RELAY` (service) | - | `7e1a0002-9b34-4d21-8f5a-2c6b1d3e0001` |
| `RHP_RELAY` ctrl_up | notify | `7e1a0002-9b34-4d21-8f5a-2c6b1d3e0002` |
| `RHP_RELAY` ctrl_down | write | `7e1a0002-9b34-4d21-8f5a-2c6b1d3e0003` |
| `RHP_RELAY` audio_up | notify | `7e1a0002-9b34-4d21-8f5a-2c6b1d3e0004` |
| `RHP_RELAY` audio_down | write | `7e1a0002-9b34-4d21-8f5a-2c6b1d3e0005` |

<Callout type="warn">
  The pair endpoint trusts `device_id` entropy plus the nonce. It has no proof-of-control: anyone who can reach the endpoint and present a valid `device_id` and nonce could attempt to pair. That is safe on a LAN or tailnet, where the endpoint is not publicly reachable. **Before internet-exposing the pair endpoint**, add out-of-band verification (a code the user confirms in-app) to stop a race-to-pair. See [Security](/docs/hardware/security).
</Callout>
