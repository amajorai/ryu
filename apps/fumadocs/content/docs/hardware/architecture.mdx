The hardware architecture is deliberately small: a device captures, a node infers, and one protocol connects them over either of two transports. Nothing about the LLM, the dashboards, or the memory loop lives on the device.

## Two transport modes, one protocol

A device reaches the node one of two ways.

- **Mode B (direct WiFi)** - the device opens a WebSocket straight to the node. This is the home case, where known WiFi is available and the node is reachable.
- **Mode A (BLE relay)** - the device speaks BLE to a paired phone, and the phone relays the traffic to the node over a WebSocket. This is the mobile, away-from-home case.

The logical protocol is **identical** in both modes. In Mode A the phone is a transparent tunnel: it pipes RHP control frames (JSON) and audio frames (binary) between the BLE GATT characteristics and the node WebSocket, one to one. The node cannot tell the two modes apart except by the `relay` flag in the device's `hello`. See [Protocol](/docs/hardware/protocol) for the frame and message detail.

Because the ESP32-S3 has both radios, the device picks its mode at runtime:

<Steps>
  <Step>
    If provisioned WiFi creds are present and the node WebSocket is reachable, use Mode B (direct).
  </Step>
  <Step>
    Otherwise, if a paired phone is in BLE range and connected, use Mode A (relay).
  </Step>
  <Step>
    Otherwise buffer to flash (for ambient capture) or show an offline state (for chat), and retry.
  </Step>
</Steps>

## Where inference happens

All inference is on the node, in-process. When a device streams audio, the node decodes Opus, transcribes with whisper.cpp, runs the chat turn on the same `/api/chat/stream` path the rest of Ryu uses, synthesizes TTS with OuteTTS, and (for ambient capture) feeds the meetings chunk pipeline that drives transcription, summary, and recall. The hardware did not add a new inference stack; it bridges into the one Core already runs.

This is why the device stays cheap and battery-friendly: an ESP32-S3 maxes out at 8 MB of PSRAM, while a 4-bit 8B model needs several gigabytes. The device physically cannot run a useful model, so it never tries.

## The relay is the load-bearing piece

The relay exists because a wearable cannot run WiFi continuously. WiFi transmit dominates the power budget, so a direct-WiFi wearable lasts only a few hours. Every shipping always-on wearable instead puts a low-power radio on the device and relays through a phone, which has power and a real network. Ryu follows that same model.

The relay does three jobs:

- Tunnels RHP control and Opus audio between BLE and the node WebSocket.
- Holds the `device_token` and adds the `Authorization: Bearer <device_token>` header on the upgrade.
- Reconnects with backoff and replays `hello` when the node drops, so a flaky link recovers on its own.

The relay is the mobile app today (`apps/native`), which is also the pairing UI. The desk hub, being always plugged in, is always Mode B and needs no relay.

## One chip for every device

Ryu copies Omi's architecture, not its chip. Omi-class wearables use a Nordic nRF52840 for its ultra-low-power BLE, and that is genuinely better for raw battery life. Ryu still chose **ESP32-S3 for all three devices** for two reasons:

- **A unified codebase.** One toolchain (ESP-IDF), one set of shared firmware components, and one protocol implementation across the watch, necklace, and desk. A second chip family would fork the firmware.
- **WiFi fallback.** The ESP32-S3 carries both WiFi and BLE, so a device can run Mode B directly when WiFi is available and fall back to the BLE relay when it is not. A BLE-only chip cannot do Mode B at all.

The tradeoff is honest: the BLE-relay battery story is shorter than a dedicated nRF52840 design would deliver, which is why on-device VAD is mandatory rather than optional. [Devices](/docs/hardware/devices) covers the battery reality per device.
