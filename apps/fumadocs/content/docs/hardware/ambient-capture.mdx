title: "Ambient capture"
description: "How the wearables capture speech 24/7 with on-device VAD, feed it into the meetings pipeline for transcription and recall, and what that costs in battery and ASR."
---

Ambient capture is the always-on listening mode. The necklace, the watch (when relayed), and the desk hub continuously listen, and when someone speaks the device streams that speech to the node, where it joins the same meetings pipeline that powers transcription, summary, and recall. There is no separate ambient backend; the device feeds the loop Ryu already runs.

## How it works

A device in `mode: ambient` does not stream a constant audio feed. That would drain the battery and run up the ASR bill. Instead it runs on-device VAD (voice-activity detection) and only transmits when speech is present.

<Steps>
  <Step>
    On `hello`, an ambient-capable device gets an `ambient_session_id` from the node - a long-running meeting that the device's speech accumulates into.
  </Step>
  <Step>
    The device's VAD watches the mic. When speech starts, it streams Opus, including a roughly 0.8 s pre-roll ring buffer so the first words are not clipped. When speech stops, it stops streaming. Silence is never sent.
  </Step>
  <Step>
    The node buffers about 1 s of decoded audio and feeds it to the meetings chunk pipeline, in-process, the same path as `POST /api/meetings/:id/chunk`. It transcribes the chunk, emits a segment, indexes it for recall, and replies with `ambient_ack` (or `ambient_skip` if it turned out to be silence).
  </Step>
  <Step>
    Rolling summarization runs over the accumulating session and flows back into long-term memory and chat context. Desk devices receive `display` updates carrying the rolling summary for the e-ink panel.
  </Step>
</Steps>

This is the [meeting notes](/docs/core/meeting-notes) pipeline doing the work. An ambient session is a meeting that never finalizes on a fixed boundary; it rolls. Everything downstream of the chunk endpoint (transcribe, segment, summarize, recall) is unchanged.

## On-device VAD is mandatory

VAD is not an optimization here, it is a requirement, for two reasons.

**Battery.** A wearable that transmits continuously dies in hours. Gating to detected speech is what gets a relayed wearable into the 10-to-14-hour range.

**ASR cost.** Transcription is the recurring bill, and it scales with audio minutes. The difference is large:

| Approach | Rough monthly ASR cost per device |
|---|---|
| Cloud Whisper, full 24 h/day | ~$130 to $260 |
| Cloud Whisper, VAD-trimmed to a few speech hours/day | ~$25 to $50 |
| Self-hosted whisper.cpp on the node | ~$0 per minute (just compute) |

Self-hosted whisper.cpp on the node is the obvious default. You already run it for meetings, it has no per-minute charge, and it keeps audio on hardware you control. That is a strong reason to keep ASR on the node rather than reaching for a cloud transcription API.

For storage, 24/7 Opus is on the order of 7 MB/hour, so roughly 5 GB/device/month before VAD, and much less after. Storage is cheap; the cost lever is VAD.

<Callout type="warn">
  Always-on capture records bystanders, which carries real consent and retention obligations. Design for it from the start: make ambient capture opt-in, keep on-device VAD so silence is never recorded, set retention limits, and give the device a hard mute the user can trust. Extend the existing meeting-consent UX rather than inventing a separate path.
</Callout>
